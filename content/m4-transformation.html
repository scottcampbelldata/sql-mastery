-- module 4 / five sessions /

Transformation

Real data is messy: wrong types, missing values, ugly strings, and above all dates. Senior analyst interviews are heavy on date math because reporting is heavy on date math. This module is CASE, the NULL toolkit, casting, strings, and serious time work - now with date arithmetic (intervals, AGE, durations), string surgery (SUBSTRING, POSITION, REGEXP_REPLACE), conditional-aggregation pivots that turn rows into columns, and TO_CHAR / TO_DATE formatting - practiced on chinook's clean invoices and tracks and on Stack Overflow's quarter-million posts.

4.1 CASE: SQL's if/else

-- chinook: bucket every invoice by its total
SELECT invoice_id, total,
       CASE
         WHEN total < 5  THEN 'small'
         WHEN total < 10 THEN 'medium'
         ELSE 'large'
       END AS size_tier
FROM invoice;

Three things to be able to say about CASE:

  • Branches are evaluated top-down and stop at the first match - so WHEN total < 10 after WHEN total < 5 effectively means "5 to under 10."
  • No ELSE means unmatched rows get NULL, silently.
  • CASE is an expression, so it works anywhere an expression works: inside SUM(), in ORDER BY, in GROUP BY, in a join condition.
-- CASE inside ORDER BY: custom sort (US invoices first, then by total)
SELECT invoice_id, billing_country, total
FROM invoice
ORDER BY CASE WHEN billing_country = 'USA' THEN 0 ELSE 1 END, total DESC;

4.2 The NULL toolkit: COALESCE and NULLIF

-- stackoverflow: score-per-view per question, guarded against divide-by-zero
SELECT
  id
, COALESCE(title, '(untitled)')      -- first non-NULL argument; any number of args
, NULLIF(score, 0)                 -- returns NULL if the two args are equal
, score::numeric / NULLIF(viewcount, 0) -- THE division-by-zero guard: x/0 becomes x/NULL = NULL
FROM posts WHERE posttypeid = 1 LIMIT 5;
Pair to memorize COALESCE turns NULL into a value; NULLIF turns a value into NULL. Together they handle nearly every messy-value situation, and x / NULLIF(y, 0) should be pure reflex.

4.3 Casting

-- chinook: casting rules, demonstrated one row from the track table
SELECT '123'::int,                    -- Postgres shorthand
       CAST('2024-01-15' AS date),   -- portable standard form; know both
       10::numeric / 3,               -- 3.3333 (numeric division)
       10 / 3,                        -- 3 (integer division!)
       ROUND(2.71828::numeric, 2),    -- ROUND(x, n) needs numeric, not float
       milliseconds / 60000.0          -- integer/float: keeps the fractional minutes
FROM track LIMIT 1;

The one that bites in practice: ROUND(AVG(float_col), 2) errors in Postgres because AVG of a float returns double precision. Cast first: ROUND(AVG(col)::numeric, 2). You have already seen this in earlier solutions - now you know why. A chinook favorite: a track's milliseconds is an integer, so milliseconds / 60000 truncates to whole minutes; milliseconds / 60000.0 keeps the fraction.

4.4 Strings

-- The working set for interviews, on chinook customers:
SELECT
  first_name || ' ' || last_name              AS full_name       -- concat (|| returns NULL if any part is NULL)
, CONCAT(first_name, ' ', last_name)          AS safe_concat     -- treats NULL as ''
, LOWER(email), UPPER(country)
, TRIM('  padded  ')                          -- also LTRIM / RTRIM
, LENGTH(company)
, SUBSTRING(email FROM 1 FOR 5)
, POSITION('@' IN email)                      -- 1-based index, 0 if absent
, SPLIT_PART(email, '@', 2)                   AS domain          -- the Postgres gem
, REPLACE(phone, '-', '')
FROM customer LIMIT 5;
-- SPLIT_PART earns its keep on Stack Overflow's pipe-delimited tags:
-- |postgresql|trigger|plpgsql| -> strip the outer pipes, take the first tag
SELECT id, tags,
       SPLIT_PART(TRIM(BOTH '|' FROM tags), '|', 1) AS first_tag
FROM posts
WHERE posttypeid = 1 AND tags LIKE '%|postgresql|%'
LIMIT 5;
-- STRING_AGG: collapse a group into one delimited string (interviews love it)
-- chinook: each artist with a comma-separated list of album titles
SELECT ar.name,
       STRING_AGG(al.title, ', ' ORDER BY al.title) AS albums
FROM artist ar
JOIN album al ON al.artist_id = ar.artist_id
GROUP BY ar.name
LIMIT 10;
-- The inverse of STRING_AGG: explode one delimited string into many rows.
-- Stack Overflow tags are pipe-wrapped (|postgresql|trigger|plpgsql|). Strip the outer
-- pipes, split into an array, then UNNEST it to one tag per row:
SELECT id,
       UNNEST(STRING_TO_ARRAY(TRIM(BOTH '|' FROM tags), '|')) AS tag
FROM posts
WHERE tags IS NOT NULL
LIMIT 20;
-- Once each tag is its own row, an ordinary GROUP BY answers "most popular tags":
SELECT tag, COUNT(*) AS uses
FROM (
    SELECT UNNEST(STRING_TO_ARRAY(TRIM(BOTH '|' FROM tags), '|')) AS tag
    FROM posts
    WHERE tags IS NOT NULL
) exploded
GROUP BY tag
ORDER BY uses DESC
LIMIT 15;
Explode to rows, then aggregate The pattern is universal: whenever a column packs several values into one string (tags, CSV lists, multi-select answers), STRING_TO_ARRAY + UNNEST turns it into a normal one-value-per-row table you can filter, join, and GROUP BY. Postgres also has regexp_split_to_table(TRIM(BOTH '|' FROM tags), '\|') which returns the rows directly. In BigQuery the same move is UNNEST(SPLIT(tags, '|')); in Snowflake it is LATERAL FLATTEN. Naming the explode-then-aggregate shape - and that it is the inverse of STRING_AGG - is a senior-level string answer.

4.5 Dates and times: the senior analyst's bread and butter

-- The core toolkit, on stackoverflow posts:
SELECT
  creationdate
, creationdate::date                              AS post_day       -- timestamp -> date
, date_trunc('month', creationdate)              AS post_month     -- floor to month (keeps type)
, EXTRACT(DOW  FROM creationdate)               AS day_of_week    -- 0=Sunday .. 6=Saturday
, EXTRACT(HOUR FROM creationdate)               AS hr
, EXTRACT(YEAR FROM creationdate)               AS yr
, TO_CHAR(creationdate, 'YYYY-MM')               AS month_label    -- formatting for display
, creationdate + INTERVAL '90 days'              AS plus_90
FROM posts LIMIT 5;
date_trunc vs EXTRACT vs TO_CHAR date_trunc groups periods on a timeline (Jan 2013 and Jan 2014 stay distinct) - use for time series. EXTRACT pulls one component (all Januaries collapse together) - use for seasonality. TO_CHAR makes display strings - use last, never for math or grouping if you can avoid it. Choosing the right one immediately signals experience.
-- generate_series: manufacture a complete calendar so gaps show as zeros
-- (Without this, months with no questions are simply missing from GROUP BY output.)
SELECT mo::date AS month_start, COUNT(p.id) AS questions
FROM generate_series(DATE '2013-01-01', DATE '2013-12-01', INTERVAL '1 month') mo
LEFT JOIN posts p
  ON date_trunc('month', p.creationdate) = mo
  AND p.posttypeid = 1
GROUP BY mo
ORDER BY mo;

4.6 Splitting delimited values into rows

SPLIT_PART pulls out one piece. The senior move on a multi-valued column - Stack Overflow's pipe-delimited tags like |c#|.net|linq| - is to explode it into one row per value so you can group and count. Postgres does this with STRING_TO_ARRAY + UNNEST (or regexp_split_to_table):

-- one row per (post, tag): strip the outer pipes, split on '|', unnest to rows
SELECT p.id, LOWER(tag) AS tag
FROM posts p,
     UNNEST(STRING_TO_ARRAY(TRIM(BOTH '|' FROM p.tags), '|')) AS tag
WHERE p.tags IS NOT NULL AND p.tags <> '';

The comma in the FROM clause is an implicit LATERAL join: UNNEST runs once per posts row and fans it out into as many rows as it has tags. Once each tag is its own row, the usual aggregation follows - the most-used tags are a GROUP BY away:

-- the 20 most common tags across all questions
SELECT LOWER(tag) AS tag, COUNT(*) AS posts
FROM posts p,
     UNNEST(STRING_TO_ARRAY(TRIM(BOTH '|' FROM p.tags), '|')) AS tag
WHERE p.posttypeid = 1 AND p.tags IS NOT NULL
GROUP BY 1
ORDER BY posts DESC
LIMIT 20;
Explode, then aggregate The pattern - turn the multi-valued column into rows, then GROUP BY - is the whole game for tags, CSV fields, and array columns. With each tag on its own row, "top tags," "tag co-occurrence" (drill P4.13), and "tags per question" become ordinary aggregations. Section 4.7 shows the same explode in BigQuery and Snowflake.

4.7 Portability: the same idea across warehouses

4.8 Date arithmetic: intervals, AGE, and durations

-- chinook: how long has each employee been on staff, as of today?
SELECT first_name || ' ' || last_name          AS emp,
       hire_date::date                          AS hired,
       AGE(CURRENT_DATE, hire_date)              AS tenure,       -- calendar interval: years, months, days
       EXTRACT(YEAR FROM AGE(CURRENT_DATE, hire_date))::int AS full_years,   -- just the whole years
       (hire_date + INTERVAL '6 months')::date  AS review_due     -- shift a date by an interval
FROM employee
ORDER BY hire_date;

Three moves cover almost every date-math question. Add or subtract an INTERVAL to shift a date (d + INTERVAL '90 days', d - INTERVAL '1 year'). AGE(a, b) gives the calendar-aware gap between two timestamps as a years/months/days interval - it respects leap years and month lengths, which raw subtraction does not. EXTRACT then pulls a single number out of that interval.

-- Subtracting two timestamps gives a raw interval; EXTRACT(EPOCH ...) turns any
-- interval into a plain number of seconds - the reliable way to measure a duration.
SELECT track_id,
       milliseconds,
       EXTRACT(EPOCH FROM (INTERVAL '1 millisecond' * milliseconds)) AS seconds
FROM track LIMIT 3;

Why it matters: reporting lives on "how long ago," "how long between," and "shift this deadline." AGE answers whole-unit questions honestly; EXTRACT(EPOCH FROM ...) converts any interval to seconds so you can divide it into hours, days, or years without hand-waving about 365.25.

4.9 String surgery: slicing, cleaning, and regex

-- chinook: pull pieces out of messy customer contact fields
SELECT phone,
       REGEXP_REPLACE(phone, '[^0-9]', '', 'g')          AS digits_only,  -- strip everything but digits
       LEFT(REGEXP_REPLACE(phone, '[^0-9]', '', 'g'), 3) AS lead,        -- first 3 chars
       RIGHT(postal_code, 3)                          AS last3,       -- last 3 chars
       SUBSTRING(email FROM POSITION('@' IN email) + 1) AS domain  -- slice from a found position
FROM customer
WHERE phone IS NOT NULL LIMIT 5;

The toolkit divides cleanly. LEFT / RIGHT grab a fixed number of characters from an end. SUBSTRING(s FROM start FOR len) takes an arbitrary slice, and pairing its start with POSITION(needle IN s) lets you cut relative to a character you found rather than a hard-coded offset. REGEXP_REPLACE(s, pattern, replacement, 'g') is the sledgehammer: the 'g' flag replaces every match, and '[^0-9]' ("anything that is not a digit") is the canonical way to normalize a phone number or ID down to bare digits.

Why it matters: real text columns are dirty - inconsistent punctuation, stray whitespace, values glued together. SPLIT_PART handles clean delimiters; POSITION + SUBSTRING handle "cut at the first X"; and REGEXP_REPLACE handles "keep only the characters that match a rule." Reaching for the narrowest tool that does the job reads as fluency.

4.10 Conditional aggregation: pivoting rows into columns

-- chinook: one row per genre, revenue spread across year columns (rows -> columns)
SELECT g.name AS genre,
       SUM(il.unit_price * il.quantity) FILTER (WHERE EXTRACT(YEAR FROM i.invoice_date) = 2022) AS y2022,
       SUM(il.unit_price * il.quantity) FILTER (WHERE EXTRACT(YEAR FROM i.invoice_date) = 2023) AS y2023,
       SUM(il.unit_price * il.quantity)                                             AS all_years
FROM invoice_line il
JOIN invoice i ON i.invoice_id = il.invoice_id
JOIN track t   ON t.track_id = il.track_id
JOIN genre g   ON g.genre_id = t.genre_id
GROUP BY g.name
ORDER BY all_years DESC
LIMIT 6;

This is the "pivot" every spreadsheet user asks for: the values that were down the rows (one year per row) become across the columns (one year per column). The mechanism is a per-column aggregate that only sees some of the rows. Two equivalent forms:

  • FILTER (shown above) - SUM(x) FILTER (WHERE cond). Postgres-native, the cleanest to read.
  • SUM(CASE) - SUM(CASE WHEN cond THEN x ELSE 0 END). The portable classic that works on every engine; know it because interviews on other databases expect it.

Why it matters: "give me metric X broken out by category across time" is the single most common reporting request, and conditional aggregation is how you serve it in one grouped pass instead of several queries stitched together. An un-FILTERed SUM in the same SELECT (the all_years column) hands you a total alongside the breakdown for free.

4.11 Casting and formatting: TO_CHAR and TO_DATE

-- chinook: format a date for display, parse a string into a date, coerce a type
SELECT TO_CHAR(invoice_date, 'FMDay DD Mon YYYY') AS pretty,      -- date -> display string
       TO_DATE('2023-07-04', 'YYYY-MM-DD')         AS parsed,      -- string -> date
       invoice_date::date                        AS just_date,   -- timestamp -> date (drop the time)
       '42'::int + 8                             AS math         -- text -> int so arithmetic works
FROM invoice LIMIT 3;

TO_CHAR turns a date or number into a formatted string for humans (the FM prefix trims the padding Postgres would otherwise add to Day). TO_DATE does the reverse, parsing a string into a real date using a format mask - reach for it when a source feeds you dates as text. And plain ::type coercions clear up the everyday traps: cast a text column to int before you do math on it, and cast a timestamp to date when the time-of-day is noise.

Format for display, cast for math The discipline: keep values in their real type (date, numeric) for every calculation, grouping, and comparison, and only convert to a string with TO_CHAR at the very end, for output. Grouping or sorting by a TO_CHAR string works by luck at best - '2013-2' sorts before '2013-10' as text - so it is a display step, never a computation step. This is the same lesson as 4.5's "date_trunc groups, TO_CHAR labels," now stated as a rule.

Practice - Session A (problems 1-3)

P4.1chinookwarmup

Classify every track as 'short' (under 3 min), 'standard' (3 to under 6 min), or 'long' (6+ min), and count tracks per class.

solution
SELECT CASE
         WHEN milliseconds < 180000 THEN 'short'
         WHEN milliseconds < 360000 THEN 'standard'
         ELSE 'long'
       END AS length_class,
       COUNT(*) AS tracks
FROM track
GROUP BY 1
ORDER BY tracks DESC;

Top-down evaluation means the second WHEN only sees rows that failed the first - no need to write BETWEEN 180000 AND 359999.

P4.2chinookwarmup

From invoice: invoice_id, billing_state with NULL replaced by 'unspecified', and each invoice's share of its own country's total revenue, as a percent rounded to 1 decimal, NULL-safe against a zero country total. Compute the country total inline. Sort by share descending, NULLs last.

hint
A window SUM over billing_country gives you the country total on every row: SUM(total) OVER (PARTITION BY billing_country). Wrap that denominator in NULLIF so a zero can never crash the division.
solution
SELECT invoice_id,
       COALESCE(billing_state, 'unspecified') AS state,
       ROUND(total / NULLIF(SUM(total) OVER (PARTITION BY billing_country), 0) * 100, 1) AS share_pct
FROM invoice
ORDER BY share_pct DESC NULLS LAST;

The NULLIF guard turns a zero country total into NULL instead of crashing the query - and NULLS LAST keeps any NULL shares out of the way without hiding them. COALESCE and NULLIF are the two halves of the NULL toolkit doing exactly what each is for.

P4.3chinookcore

Track metadata report: per track, its length in whole seconds, and a status of 'no composer' if composer is NULL, 'short' if under 3 minutes, else 'full length'. Then aggregate: count per status. (composer is genuinely NULL for many tracks - that is the point.)

hint
Handle the NULL branch first in the CASE. A comparison like milliseconds < 180000 is never NULL, but you want the "no composer" label to win regardless of length, so test it up top.
solution
-- per track
SELECT track_id,
       milliseconds / 1000 AS seconds,
       CASE
         WHEN composer IS NULL          THEN 'no composer'
         WHEN milliseconds < 180000     THEN 'short'
         ELSE 'full length'
       END AS status
FROM track;

-- aggregate
SELECT CASE
         WHEN composer IS NULL          THEN 'no composer'
         WHEN milliseconds < 180000     THEN 'short'
         ELSE 'full length'
       END AS status,
       COUNT(*) AS tracks
FROM track
GROUP BY 1;

Putting the NULL test first is the habit to build: elsewhere a comparison against a NULL column yields NULL (not false), so a row can silently fall through to the ELSE branch and land in the wrong bucket - a wrong answer that looks plausible.

Practice - Session B (problems 4-6)

P4.4chinookcore

Customer email domains: extract the domain from each customer email, count customers per domain, most common first.

solution
SELECT LOWER(SPLIT_PART(email, '@', 2)) AS domain,
       COUNT(*) AS customers
FROM customer
WHERE email IS NOT NULL
GROUP BY 1
ORDER BY customers DESC;

LOWER prevents Gmail.com and gmail.com from splitting into two groups. Portable alternative to SPLIT_PART: SUBSTRING(email FROM POSITION('@' IN email) + 1).

P4.5stackoverflowcore

Monthly question time series for 2013: questions created per calendar month, with a display label like '2013-03'. Chronological. Which function anchors the grouping and which makes the label?

solution
SELECT date_trunc('month', creationdate) AS month_start,
       TO_CHAR(date_trunc('month', creationdate), 'YYYY-MM') AS label,
       COUNT(*) AS questions
FROM posts
WHERE posttypeid = 1
  AND creationdate >= '2013-01-01'
  AND creationdate <  '2014-01-01'
GROUP BY 1, 2
ORDER BY month_start;

date_trunc groups; TO_CHAR labels. Grouping by the TO_CHAR string alone works but sorts as text and throws away the timestamp - keep the truncated timestamp as the real axis. The half-open range (>= '2013-01-01' AND < '2014-01-01') keeps the filter sargable, a habit Module 8 rewards.

P4.6stackoverflowcore

Answer-quality distribution: for answers only, bucket each answer by score - 'negative' (below 0), 'zero', 'low' (1-4), or 'high' (5+) - and count answers per bucket. (Real Q&A data is heavily skewed toward low scores; quantify it.)

solution
SELECT CASE
         WHEN score < 0 THEN 'a. negative'
         WHEN score = 0 THEN 'b. zero'
         WHEN score < 5 THEN 'c. low (1-4)'
         ELSE 'd. high (5+)'
       END AS bucket,
       COUNT(*) AS answers
FROM posts
WHERE posttypeid = 2
GROUP BY 1
ORDER BY bucket;

Ordering by a prefixed label ('a.', 'b.', ...) forces the buckets into logical order instead of alphabetical, a small trick that makes bucketed reports readable. Volunteering "most answers cluster at low scores, so the mean is misleading here" is the analyst instinct interviewers listen for.

Practice - Session C (problems 7-9)

P4.7stackoverflowinterview

Day-of-week posting pattern: for questions only, count posts per day of week with readable day names, ordered Monday first. (Hint: you need both the number and the name - one to sort, one to display.)

solution
SELECT TO_CHAR(creationdate, 'Day')      AS day_name,
       COUNT(*)                           AS questions
FROM posts
WHERE posttypeid = 1
GROUP BY day_name, EXTRACT(ISODOW FROM creationdate)
ORDER BY EXTRACT(ISODOW FROM creationdate);

EXTRACT in GROUP BY needs the same expression per group, and since day_name determines ISODOW this grouping is legal and correct. ISODOW is Monday=1 through Sunday=7 (plain DOW is Sunday=0). Sorting a day-name column correctly is a surprisingly common practical interview task - and here it reveals the weekday/weekend drop in question volume.

P4.8stackoverflowinterview

Complete monthly calendar: questions per month for all of 2013, including any month with zero questions. Use generate_series. Confirm your output has exactly 12 rows.

solution
SELECT mo::date AS month_start,
       COUNT(p.id) AS questions
FROM generate_series(DATE '2013-01-01', DATE '2013-12-01', INTERVAL '1 month') mo
LEFT JOIN posts p
  ON date_trunc('month', p.creationdate) = mo
  AND p.posttypeid = 1
GROUP BY mo
ORDER BY mo;

The calendar is the left table so empty months survive, and COUNT(p.id) - not COUNT(*) - makes those months show 0. Note the answer-vs-question filter lives in the ON clause, not a WHERE: put it in WHERE and you turn the LEFT JOIN back into an inner join, and the zero-months vanish. This exact "don't let missing periods disappear" requirement shows up in most reporting take-homes.

P4.9chinookinterview

Per support rep: a single comma-separated, alphabetized list of the countries their customers live in, plus the customer count. Use STRING_AGG with DISTINCT.

solution
SELECT e.first_name || ' ' || e.last_name AS rep,
       COUNT(c.customer_id) AS customers,
       STRING_AGG(DISTINCT c.country, ', ' ORDER BY c.country) AS countries
FROM employee e
JOIN customer c ON c.support_rep_id = e.employee_id
GROUP BY rep
ORDER BY customers DESC;

With DISTINCT inside STRING_AGG, the ORDER BY must use the same expression as the DISTINCT - a Postgres rule that trips people the first time.

Practice - Session D (problems 10-13)

P4.10chinookinterview

Sales pace by month: from invoice, the average, min, and max invoice total per order month across the whole date range. Label each month like '2023-04'. Round the average to 2 decimals, and order chronologically.

solution
SELECT date_trunc('month', invoice_date)::date AS order_month,
       TO_CHAR(invoice_date, 'YYYY-MM') AS label,
       ROUND(AVG(total)::numeric, 2) AS avg_total,
       MIN(total) AS min_total,
       MAX(total) AS max_total
FROM invoice
GROUP BY 1, 2
ORDER BY order_month;

The ::numeric cast on AVG matters: AVG over a floating column returns double precision, and ROUND(x, 2) wants numeric - the same casting trap from 4.3, now in the wild. date_trunc gives the real time axis; TO_CHAR just decorates it.

P4.11stackoverflowhard

Account age vs reputation: bucket users by account age in whole years (as of today) and show avg reputation per bucket. Then answer: why is AGE() vs plain subtraction a meaningful choice here?

solution
SELECT EXTRACT(YEAR FROM AGE(CURRENT_DATE, creationdate))::int AS account_years,
       COUNT(*) AS users,
       ROUND(AVG(reputation)::numeric, 0) AS avg_reputation
FROM users
GROUP BY 1
ORDER BY account_years;

AGE() gives calendar-aware years/months/days (handles leap years and month lengths); subtracting timestamps gives a raw interval in days that you would have to divide by 365.25 and defend. For "whole years old," AGE is the honest tool. Expect the follow-up "would avg or median be better here?" - reputation is heavily skewed, so median (Module 7) tells a truer story.

P4.12stackoverflowhard

The weekend effect: compare weekday vs weekend questions in one row - post counts, average score, and average answer count - using conditional aggregation with a day-of-week CASE or FILTER.

solution
SELECT
  COUNT(*) FILTER (WHERE EXTRACT(ISODOW FROM creationdate) < 6)  AS weekday_questions,
  COUNT(*) FILTER (WHERE EXTRACT(ISODOW FROM creationdate) >= 6) AS weekend_questions,
  ROUND(AVG(score) FILTER (WHERE EXTRACT(ISODOW FROM creationdate) < 6)::numeric, 2)  AS weekday_avg_score,
  ROUND(AVG(score) FILTER (WHERE EXTRACT(ISODOW FROM creationdate) >= 6)::numeric, 2) AS weekend_avg_score,
  ROUND(AVG(answercount) FILTER (WHERE EXTRACT(ISODOW FROM creationdate) < 6)::numeric, 2)  AS weekday_avg_answers,
  ROUND(AVG(answercount) FILTER (WHERE EXTRACT(ISODOW FROM creationdate) >= 6)::numeric, 2) AS weekend_avg_answers
FROM posts
WHERE posttypeid = 1;

The repetition is intentional practice - and it is also the setup for the cleaner refactor: in Module 5 you will compute is_weekend once in a CTE and aggregate from that. Feeling the pain first is what makes CTEs stick. ISODOW >= 6 catches Saturday (6) and Sunday (7).

P4.13stackoverflowhard

Tag co-occurrence: across all questions, find the 15 most common pairs of tags that appear together on the same post. Explode the pipe-delimited tags into rows (4.6), then self-join a post to itself on tag pairs. Return tag_a, tag_b, and how many questions carry both.

hint
Explode to (post_id, tag) in a CTE. Self-join on a.post_id = b.post_id with a.tag < b.tag so each pair is counted once and no tag is paired with itself.
solution
WITH post_tags AS (
    SELECT p.id AS post_id, LOWER(tag) AS tag
    FROM posts p,
         UNNEST(STRING_TO_ARRAY(TRIM(BOTH '|' FROM p.tags), '|')) AS tag
    WHERE p.posttypeid = 1 AND p.tags IS NOT NULL
)
SELECT a.tag AS tag_a, b.tag AS tag_b, COUNT(*) AS questions
FROM post_tags a
JOIN post_tags b ON a.post_id = b.post_id AND a.tag < b.tag
GROUP BY a.tag, b.tag
ORDER BY questions DESC
LIMIT 15;

The a.tag < b.tag predicate does double duty: it drops the self-pair (a tag with itself) and collapses (x, y) and (y, x) into one canonical pair, so each combination is counted exactly once. "Explode, then self-join on the shared key" is the standard way to turn a multi-valued column into a co-occurrence matrix - market-basket analysis in a dozen lines.

Practice - Session E (problems 14-18)

P4.14chinookwarmup

Data density per track: from track, show bytes, milliseconds, the integer division bytes / milliseconds, and the same ratio to 1 decimal place. The two ratios should differ - explain why in one line.

solution
SELECT bytes,
       milliseconds,
       bytes / milliseconds                    AS int_div,
       ROUND(bytes::numeric / milliseconds, 1) AS bytes_per_ms
FROM track LIMIT 5;

Both operands are integers, so bytes / milliseconds is integer division and truncates the fraction. Casting either side to numeric switches Postgres to real division; do it before the ROUND, not after. This is the 4.3 casting trap in one line.

P4.15chinookcore

Tenure cohorts: group employees by their hire year and show, per year, the head count and the average tenure in whole-plus-fractional years as of today, rounded to 1 decimal. Order by hire year.

hint
Get exact fractional years from an interval with EXTRACT(EPOCH FROM AGE(CURRENT_DATE, hire_date)) / 86400 / 365.25. Group on EXTRACT(YEAR FROM hire_date).
solution
SELECT EXTRACT(YEAR FROM hire_date)::int AS hire_year,
       COUNT(*) AS employees,
       ROUND(AVG(EXTRACT(EPOCH FROM AGE(CURRENT_DATE, hire_date)) / 86400 / 365.25)::numeric, 1) AS avg_years
FROM employee
GROUP BY 1
ORDER BY hire_year;

AGE gives a calendar interval; EXTRACT(EPOCH ...) turns it into seconds so you can divide down to fractional years honestly. The ::numeric before ROUND is the same cast from 4.3 - AVG of a float is double precision, and ROUND wants numeric.

P4.16chinookcore

Quarterly invoice pivot: one row per billing country, with four columns counting invoices in Q1, Q2, Q3, and Q4 (across all years combined). Show the eight busiest countries by total invoices.

hint
This is a rows-to-columns pivot (4.10). Use four COUNT(*) FILTER (WHERE EXTRACT(QUARTER FROM invoice_date) = n) expressions, one per quarter, and GROUP BY the country.
solution
SELECT billing_country,
       COUNT(*) FILTER (WHERE EXTRACT(QUARTER FROM invoice_date) = 1) AS q1,
       COUNT(*) FILTER (WHERE EXTRACT(QUARTER FROM invoice_date) = 2) AS q2,
       COUNT(*) FILTER (WHERE EXTRACT(QUARTER FROM invoice_date) = 3) AS q3,
       COUNT(*) FILTER (WHERE EXTRACT(QUARTER FROM invoice_date) = 4) AS q4
FROM invoice
GROUP BY billing_country
ORDER BY COUNT(*) DESC
LIMIT 8;

Each FILTER restricts its own COUNT to one quarter, so the four columns partition every country's invoices. The portable version swaps each FILTER for COUNT(CASE WHEN EXTRACT(QUARTER FROM invoice_date) = n THEN 1 END) - same result, works on any engine.

P4.17stackoverflowinterview

Time-of-day profile: for questions asked in January 2013, count them into four parts of the day in a single row - morning (hours 6-11), afternoon (12-17), evening (18-23), and night (0-5). Filter the month with a half-open date range.

hint
Four COUNT(*) FILTER (...) columns, each keyed on EXTRACT(HOUR FROM creationdate) with a BETWEEN range. Keep the month filter sargable: creationdate >= '2013-01-01' AND creationdate < '2013-02-01'.
solution
SELECT
  COUNT(*) FILTER (WHERE EXTRACT(HOUR FROM creationdate) BETWEEN 6 AND 11)  AS morning,
  COUNT(*) FILTER (WHERE EXTRACT(HOUR FROM creationdate) BETWEEN 12 AND 17) AS afternoon,
  COUNT(*) FILTER (WHERE EXTRACT(HOUR FROM creationdate) BETWEEN 18 AND 23) AS evening,
  COUNT(*) FILTER (WHERE EXTRACT(HOUR FROM creationdate) BETWEEN 0 AND 5)   AS night
FROM posts
WHERE posttypeid = 1
  AND creationdate >= '2013-01-01'
  AND creationdate <  '2013-02-01';

EXTRACT(HOUR ...) buckets the timestamp; the four FILTERs pivot those buckets into one comparison row. The half-open range beats BETWEEN '2013-01-01' AND '2013-01-31' because it cleanly includes every moment of the month without fretting over the last day's time-of-day, and it stays index-friendly (Module 8).

P4.18stackoverflowinterview

Primary-tag leaderboard for Q1 2013: for questions in the first quarter of 2013, take each post's first tag as its primary tag, label the quarter like '2013-1', and return the eight most common primary tags with their counts.

hint
The tags column is pipe-wrapped like |sql-server|tsql|. Strip the outer pipes with TRIM(BOTH '|' FROM tags), then SPLIT_PART(..., '|', 1) gives the first tag. Label the quarter with TO_CHAR(creationdate, 'YYYY-Q').
solution
SELECT TO_CHAR(creationdate, 'YYYY-Q') AS quarter_label,
       SPLIT_PART(TRIM(BOTH '|' FROM tags), '|', 1) AS primary_tag,
       COUNT(*) AS questions
FROM posts
WHERE posttypeid = 1
  AND tags IS NOT NULL
  AND creationdate >= '2013-01-01'
  AND creationdate <  '2013-04-01'
GROUP BY 1, 2
ORDER BY questions DESC
LIMIT 8;

SPLIT_PART pulls out just the first piece (4.4) - no need to explode the whole list here since only the primary tag matters. TRIM(BOTH '|' ...) first, or the leading pipe would make SPLIT_PART return an empty string as part 1. TO_CHAR with 'YYYY-Q' is a display label only, matching the "format last" rule from 4.11.
