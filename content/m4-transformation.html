-- module 4 / four sessions /

Transformation

Real data is messy: wrong types, missing values, ugly strings, and above all dates. Senior analyst interviews are heavy on date math because reporting is heavy on date math. This module is CASE, the NULL toolkit, casting, strings, and serious time work.

4.1 CASE: SQL's if/else

-- northwind: price tiers
SELECT product_name, unit_price,
       CASE
         WHEN unit_price < 20 THEN 'budget'
         WHEN unit_price < 50 THEN 'mid'
         ELSE 'premium'
       END AS tier
FROM products;

Three things to be able to say about CASE:

  • Branches are evaluated top-down and stop at the first match - so WHEN unit_price < 50 after WHEN unit_price < 20 effectively means "20 to under 50."
  • No ELSE means unmatched rows get NULL, silently.
  • CASE is an expression, so it works anywhere an expression works: inside SUM(), in ORDER BY, in GROUP BY, in a join condition.
-- CASE inside ORDER BY: custom sort (in-stock first, then by price)
SELECT product_name, units_in_stock, unit_price
FROM products
ORDER BY CASE WHEN units_in_stock > 0 THEN 0 ELSE 1 END, unit_price;

4.2 The NULL toolkit: COALESCE and NULLIF

SELECT
  COALESCE(color, 'unknown')          -- first non-NULL argument; any number of args
, NULLIF(fare_amount, 0)              -- returns NULL if the two args are equal
, tip_amount / NULLIF(fare_amount, 0) -- THE division-by-zero guard: x/0 becomes x/NULL = NULL
FROM trips LIMIT 5;
Pair to memorize COALESCE turns NULL into a value; NULLIF turns a value into NULL. Together they handle nearly every messy-value situation, and x / NULLIF(y, 0) should be pure reflex.

4.3 Casting

SELECT '123'::int,                    -- Postgres shorthand
       CAST('2024-01-15' AS date),   -- portable standard form; know both
       10::numeric / 3,               -- 3.3333 (numeric division)
       10 / 3,                        -- 3 (integer division!)
       ROUND(2.71828::numeric, 2);    -- ROUND(x, n) needs numeric, not float

The one that bites in practice: ROUND(AVG(float_col), 2) errors in Postgres because AVG of a float returns double precision. Cast first: ROUND(AVG(col)::numeric, 2). You have already seen this in earlier solutions - now you know why.

4.4 Strings

-- The working set for interviews:
SELECT
  first_name || ' ' || last_name              AS full_name       -- concat (|| returns NULL if any part is NULL)
, CONCAT(first_name, ' ', last_name)          AS safe_concat     -- treats NULL as ''
, LOWER(email), UPPER(country)
, TRIM('  padded  ')                          -- also LTRIM / RTRIM
, LENGTH(company)
, LEFT(phone, 3), RIGHT(phone, 4)
, SUBSTRING(email FROM 1 FOR 5)
, POSITION('@' IN email)                      -- 1-based index, 0 if absent
, SPLIT_PART(email, '@', 2)                   AS domain          -- the Postgres gem
, REPLACE(phone, '-', '')
FROM customer LIMIT 5;
-- STRING_AGG: collapse a group into one delimited string (interviews love it)
-- chinook: each artist with a comma-separated list of album titles
SELECT ar.name,
       STRING_AGG(al.title, '; ' ORDER BY al.title) AS albums
FROM artist ar
JOIN album al ON al.artist_id = ar.artist_id
GROUP BY ar.name
LIMIT 10;

4.5 Dates and times: the senior analyst's bread and butter

-- The core toolkit, on nyctaxi:
SELECT
  pickup_datetime
, pickup_datetime::date                              AS pickup_day     -- timestamp -> date
, date_trunc('month', pickup_datetime)              AS pickup_month   -- floor to month (keeps type)
, EXTRACT(DOW  FROM pickup_datetime)               AS day_of_week    -- 0=Sunday .. 6=Saturday
, EXTRACT(HOUR FROM pickup_datetime)               AS hr
, dropoff_datetime - pickup_datetime                 AS duration       -- interval
, EXTRACT(EPOCH FROM dropoff_datetime - pickup_datetime) / 60 AS minutes
, TO_CHAR(pickup_datetime, 'YYYY-MM')               AS month_label    -- formatting for display
, pickup_datetime + INTERVAL '90 minutes'           AS plus_90
FROM trips LIMIT 5;
date_trunc vs EXTRACT vs TO_CHAR date_trunc groups periods on a timeline (Jan 2016 and Jan 2017 stay distinct) - use for time series. EXTRACT pulls one component (all Januaries collapse together) - use for seasonality. TO_CHAR makes display strings - use last, never for math or grouping if you can avoid it. Choosing the right one immediately signals experience.
-- generate_series: manufacture a complete calendar so gaps show as zeros
-- (Without this, days with no orders are simply missing from GROUP BY output.)
SELECT d::date AS day, COUNT(o.order_id) AS orders
FROM generate_series(DATE '1997-01-01', DATE '1997-01-31', INTERVAL '1 day') d
LEFT JOIN orders o ON o.order_date = d::date
GROUP BY d
ORDER BY d;

Practice - Session A (problems 1-3)

P4.1chinookwarmup

Classify every track as 'short' (under 3 min), 'standard' (3 to under 6 min), or 'long' (6+ min), and count tracks per class.

solution
SELECT CASE
         WHEN milliseconds < 180000 THEN 'short'
         WHEN milliseconds < 360000 THEN 'standard'
         ELSE 'long'
       END AS length_class,
       COUNT(*) AS tracks
FROM track
GROUP BY 1
ORDER BY tracks DESC;

Top-down evaluation means the second WHEN only sees rows that failed the first - no need to write BETWEEN 180000 AND 359999.

P4.2adventureworkswarmup

From production.product: name, color with NULL replaced by 'unclassified', and margin percent = (listprice - standardcost) / listprice * 100, rounded to 1 decimal, NULL-safe against listprice = 0. Products with listprice > 0 only... actually, keep all products and let the guard do its job. Sort by margin, NULLs last.

solution
SELECT name,
       COALESCE(color, 'unclassified') AS color,
       ROUND((listprice - standardcost) / NULLIF(listprice, 0) * 100, 1) AS margin_pct
FROM production.product
ORDER BY margin_pct DESC NULLS LAST;

Zero-price products get a NULL margin instead of crashing the query - and NULLS LAST keeps them out of the way without hiding them.

P4.3northwindcore

Shipping performance report: per order, days from order_date to shipped_date, and a status of 'on time' if shipped on or before required_date, 'late' if after, 'unshipped' if shipped_date is NULL. Then aggregate: count per status.

hint
Date minus date gives an integer number of days in Postgres. Handle the NULL branch first in the CASE.
solution
-- per order
SELECT order_id,
       shipped_date - order_date AS days_to_ship,
       CASE
         WHEN shipped_date IS NULL        THEN 'unshipped'
         WHEN shipped_date <= required_date THEN 'on time'
         ELSE 'late'
       END AS status
FROM orders;

-- aggregate
SELECT CASE
         WHEN shipped_date IS NULL        THEN 'unshipped'
         WHEN shipped_date <= required_date THEN 'on time'
         ELSE 'late'
       END AS status,
       COUNT(*) AS orders
FROM orders
GROUP BY 1;

Putting the NULL test first matters: shipped_date <= required_date is NULL (not false) for unshipped orders, and without the first branch they would fall through to 'late' via ELSE - a wrong answer that looks plausible.

Practice - Session B (problems 4-6)

P4.4chinookcore

Customer email domains: extract the domain from each customer email, count customers per domain, most common first.

solution
SELECT LOWER(SPLIT_PART(email, '@', 2)) AS domain,
       COUNT(*) AS customers
FROM customer
WHERE email IS NOT NULL
GROUP BY 1
ORDER BY customers DESC;

LOWER prevents Gmail.com and gmail.com from splitting into two groups. Portable alternative to SPLIT_PART: SUBSTRING(email FROM POSITION('@' IN email) + 1).

P4.5nyctaxicore

Monthly revenue time series: total_amount summed per calendar month, with a display label like '2016-03'. Chronological. Which function anchors the grouping and which makes the label?

solution
SELECT date_trunc('month', pickup_datetime) AS month_start,
       TO_CHAR(date_trunc('month', pickup_datetime), 'YYYY-MM') AS label,
       ROUND(SUM(total_amount)::numeric, 2) AS revenue
FROM trips
GROUP BY 1, 2
ORDER BY month_start;

date_trunc groups; TO_CHAR labels. Grouping by the TO_CHAR string alone works but sorts as text and throws away the timestamp - keep the truncated timestamp as the real axis.

P4.6nyctaxicore

Trip duration quality check: compute duration in minutes and count trips in buckets - negative or zero, 0-1 min, over 4 hours, and 'normal'. (Real dispatch data always has garbage; quantify it.)

solution
SELECT CASE
         WHEN dropoff_datetime <= pickup_datetime THEN 'a. nonpositive'
         WHEN dropoff_datetime <  pickup_datetime + INTERVAL '1 minute' THEN 'b. under 1 min'
         WHEN dropoff_datetime >  pickup_datetime + INTERVAL '4 hours'  THEN 'd. over 4 hours'
         ELSE 'c. normal'
       END AS bucket,
       COUNT(*) AS trips
FROM trips
GROUP BY 1
ORDER BY bucket;

Comparing timestamps directly against pickup + INTERVAL avoids computing EPOCH per row and stays readable. Volunteering "I would report these buckets to whoever owns the pipeline" is the analyst mindset interviewers want.

Practice - Session C (problems 7-9)

P4.7stackoverflowinterview

Day-of-week posting pattern: for questions only, count posts per day of week with readable day names, ordered Monday first. (Hint: you need both the number and the name - one to sort, one to display.)

solution
SELECT TO_CHAR(creation_date, 'Day')      AS day_name,
       COUNT(*)                           AS questions
FROM posts
WHERE post_type_id = 1
GROUP BY day_name, EXTRACT(ISODOW FROM creation_date)
ORDER BY EXTRACT(ISODOW FROM creation_date);

Wait - EXTRACT in GROUP BY needs the same expression per group, and since day_name determines ISODOW this grouping is legal and correct. ISODOW is Monday=1 through Sunday=7 (plain DOW is Sunday=0). Sorting a day-name column correctly is a surprisingly common practical interview task.

P4.8northwindinterview

Complete daily calendar: orders per day for June 1997, including days with zero orders. Use generate_series. Confirm your output has exactly 30 rows.

solution
SELECT d::date AS day,
       COUNT(o.order_id) AS orders
FROM generate_series(DATE '1997-06-01', DATE '1997-06-30', INTERVAL '1 day') d
LEFT JOIN orders o ON o.order_date = d::date
GROUP BY d
ORDER BY d;

The calendar is the left table so empty days survive, and COUNT(o.order_id) - not COUNT(*) - makes those days show 0. This exact "don't let missing days disappear" requirement shows up in most reporting take-homes.

P4.9chinookinterview

Per support rep: a single comma-separated, alphabetized list of the countries their customers live in, plus the customer count. Use STRING_AGG with DISTINCT.

solution
SELECT e.first_name || ' ' || e.last_name AS rep,
       COUNT(c.customer_id) AS customers,
       STRING_AGG(DISTINCT c.country, ', ' ORDER BY c.country) AS countries
FROM employee e
JOIN customer c ON c.support_rep_id = e.employee_id
GROUP BY rep
ORDER BY customers DESC;

With DISTINCT inside STRING_AGG, the ORDER BY must use the same expression as the DISTINCT - a Postgres rule that trips people the first time.

Practice - Session D (problems 10-12)

P4.10adventureworksinterview

Fulfillment speed by month: from sales.salesorderheader, average days between orderdate and shipdate per order month. Also show the slowest single order per month is not needed - just min and max days to bracket the average. Round to 2 decimals.

solution
SELECT date_trunc('month', orderdate)::date AS order_month,
       ROUND(AVG(shipdate::date - orderdate::date)::numeric, 2) AS avg_days,
       MIN(shipdate::date - orderdate::date) AS min_days,
       MAX(shipdate::date - orderdate::date) AS max_days
FROM sales.salesorderheader
WHERE shipdate IS NOT NULL
GROUP BY 1
ORDER BY order_month;

The ::date casts matter if the columns are timestamps: timestamp subtraction yields an interval (which AVG can handle, but the output is uglier); date subtraction yields clean integers.

P4.11stackoverflowhard

Account age vs reputation: bucket users by account age in whole years (as of today) and show avg reputation per bucket. Then answer: why is AGE() vs plain subtraction a meaningful choice here?

solution
SELECT EXTRACT(YEAR FROM AGE(CURRENT_DATE, creation_date))::int AS account_years,
       COUNT(*) AS users,
       ROUND(AVG(reputation)::numeric, 0) AS avg_reputation
FROM users
GROUP BY 1
ORDER BY account_years;

AGE() gives calendar-aware years/months/days (handles leap years and month lengths); subtracting timestamps gives a raw interval in days that you would have to divide by 365.25 and defend. For "whole years old," AGE is the honest tool. Expect the follow-up "would avg or median be better here?" - reputation is heavily skewed, so median (Module 7) tells a truer story.

P4.12nyctaxihard

The weekend premium: compare weekday vs weekend trips in one row - trip counts, average fare, and average tip percentage (fare > 0), using conditional aggregation with a day-of-week CASE or FILTER.

solution
SELECT
  COUNT(*) FILTER (WHERE EXTRACT(ISODOW FROM pickup_datetime) < 6)  AS weekday_trips,
  COUNT(*) FILTER (WHERE EXTRACT(ISODOW FROM pickup_datetime) >= 6) AS weekend_trips,
  ROUND(AVG(fare_amount) FILTER (WHERE EXTRACT(ISODOW FROM pickup_datetime) < 6)::numeric, 2)  AS weekday_avg_fare,
  ROUND(AVG(fare_amount) FILTER (WHERE EXTRACT(ISODOW FROM pickup_datetime) >= 6)::numeric, 2) AS weekend_avg_fare,
  ROUND(AVG(tip_amount / NULLIF(fare_amount, 0)) FILTER (WHERE EXTRACT(ISODOW FROM pickup_datetime) < 6)::numeric * 100, 1)  AS weekday_tip_pct,
  ROUND(AVG(tip_amount / NULLIF(fare_amount, 0)) FILTER (WHERE EXTRACT(ISODOW FROM pickup_datetime) >= 6)::numeric * 100, 1) AS weekend_tip_pct
FROM trips
WHERE fare_amount > 0;

The repetition is intentional practice - and it is also the setup for the cleaner refactor: in Module 5 you will compute is_weekend once in a CTE and aggregate from that. Feeling the pain first is what makes CTEs stick.
