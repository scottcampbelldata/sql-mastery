-- module 1 / four sessions /

Fundamentals

Retrieval, filtering, sorting, and the three-valued logic of NULL, plus the everyday tools that go with them: COALESCE and NULLIF for substituting NULLs, DISTINCT vs GROUP BY and DISTINCT ON for de-duplication, LIKE and regex pattern matching, and stable LIMIT/OFFSET paging. Nothing here is hard, but senior candidates get dinged on NULL handling and operator precedence more often than on window functions. Everything runs against one table at a time - mostly chinook's clean track, invoice, and customer - so the only thing you have to think about is the clause you are practicing. Get these airtight.

1.1 SELECT, aliases, and expressions

-- chinook: columns, an expression, and aliases
SELECT
    name                              AS track_name,
    milliseconds / 60000.0            AS minutes,   -- 60000.0 forces decimal division
    unit_price,
    unit_price * 1.10                 AS price_after_10pct_increase
FROM track
LIMIT 10;
Integer division trap In Postgres, milliseconds / 60000 with two integers returns an integer (truncated). Divide by 60000.0 or cast one side: milliseconds::numeric / 60000. Interviewers plant this trap constantly in "average" questions.

1.2 WHERE: comparison, IN, BETWEEN, LIKE

-- chinook: several filter styles in one query
SELECT name, unit_price, milliseconds
FROM track
WHERE unit_price BETWEEN 0.99 AND 1.99        -- inclusive on both ends
  AND genre_id IN (1, 2, 3)                  -- Rock, Jazz, Metal
  AND name ILIKE '%love%'                   -- ILIKE = case-insensitive (Postgres)
  AND composer IS NOT NULL;
-- Operator precedence: AND binds tighter than OR. These two differ:
WHERE country = 'USA' OR country = 'Canada' AND city = 'Vancouver'
-- means: USA (any city), OR (Canada AND Vancouver)

WHERE (country = 'USA' OR country = 'Canada') AND city = 'Vancouver'
-- means: Vancouver rows in either country (i.e., just Canada in practice)
HabitAny time you mix AND with OR, parenthesize. Zero exceptions. It reads as careful, not as insecure.

1.3 NULL: the part beginners underestimate

NULL means "unknown," and comparisons with unknown are neither true nor false - they are NULL, and WHERE only keeps rows where the condition is true. In chinook, customer.state is NULL for most non-US customers, which makes it a perfect NULL playground.

-- chinook: customers with no state recorded
SELECT customer_id, first_name, last_name, country, state
FROM customer
WHERE state IS NULL;      -- correct

-- WHERE state = NULL       returns ZERO rows, always. = NULL is never true.
-- WHERE state != 'CA'      also silently DROPS every NULL-state row.
-- To keep NULLs when using != you must ask for them explicitly:
WHERE state IS DISTINCT FROM 'CA'
-- IS DISTINCT FROM treats NULL as a comparable value: a NULL state is "distinct from"
-- 'CA', so it is kept. A great phrase to use in interviews.

1.4 ORDER BY, LIMIT, DISTINCT

-- stackoverflow: top reputation, newest account first as tiebreaker
SELECT displayname, reputation, creationdate
FROM users
ORDER BY reputation DESC, creationdate DESC
LIMIT 20;
-- Postgres sorts NULLs LAST on ASC and FIRST on DESC by default. Control it:
-- chinook invoices, biggest first, but never let a NULL state float to the top:
SELECT invoice_id, billing_country, billing_state, total
FROM invoice
ORDER BY billing_state DESC NULLS LAST;
-- DISTINCT applies to the whole selected row, not the first column:
SELECT DISTINCT billing_country, billing_city FROM invoice;  -- unique (country, city) pairs
Interview soundbite "LIMIT without ORDER BY returns an arbitrary set of rows - the database gives you whatever is cheapest. If the question says top or first, I need an ORDER BY." Saying this unprompted is an easy credibility win.

1.5 COALESCE and NULLIF: turning NULLs into answers

Once you can detect NULLs, you need to substitute them. COALESCE(a, b, c) returns the first non-NULL argument, so it is the standard "fallback" tool. NULLIF(a, b) is the mirror image: it returns NULL when a = b, which is how you demote a placeholder value (like an empty string or a zero) back into a real NULL.

-- chinook: fall back from state to country, and from company to a label
SELECT customer_id,
       first_name,
       last_name,
       COALESCE(state, country)                       AS region,        -- state if we have one, else country
       COALESCE(NULLIF(TRIM(company), ''), 'individual') AS account_type
FROM customer
ORDER BY region, last_name
LIMIT 10;

The NULLIF(TRIM(company), '') trick is the reason NULLIF exists: a blank string is not NULL, so a plain COALESCE(company, 'individual') would happily return an empty string. NULLIF converts the blank into a real NULL first, and only then does COALESCE reach the fallback.

Division trap NULLIF(denominator, 0) is the classic guard against divide-by-zero: x / NULLIF(d, 0) yields NULL instead of erroring when d is 0. You will reuse this in every ratio you ever compute.

1.6 DISTINCT vs GROUP BY, and DISTINCT ON

Two ways to de-duplicate look interchangeable and mostly are. SELECT DISTINCT col and SELECT col ... GROUP BY col return the same set of rows; GROUP BY is what you reach for once you also need an aggregate per group (Module 2), while DISTINCT reads better when you just want the unique values.

-- chinook: these two return the identical 24 countries
SELECT DISTINCT billing_country FROM invoice;
SELECT billing_country FROM invoice GROUP BY billing_country;

Postgres also has DISTINCT ON, which keeps the first row per group as defined by ORDER BY. It is the shortest way to answer "one row per X, the biggest/newest one" without a window function.

-- chinook: the single largest invoice per country
SELECT DISTINCT ON (billing_country)
       billing_country, invoice_id, total
FROM invoice
ORDER BY billing_country, total DESC;   -- leading key MUST match the DISTINCT ON column
Why it mattersThe ORDER BY has to lead with the DISTINCT ON column, then sort by the tiebreaker that decides which row survives (here total DESC keeps the biggest). Interviewers love this because it exposes whether you understand that DISTINCT ON picks a row, not just a value.

1.7 Pattern matching: LIKE, ILIKE, and regex

You have already used ILIKE '%love%'. Two wildcards drive LIKE: % matches any run of characters (including none) and _ matches exactly one character. ILIKE is the case-insensitive form; anchor a pattern by leaving off the leading %.

-- chinook: names that START with "The " (anchored), and 5-char postal codes starting 1
SELECT name FROM track WHERE name ILIKE 'the %';      -- no leading %, so it anchors to the start
SELECT DISTINCT billing_postal_code
FROM invoice
WHERE billing_postal_code LIKE '1____';             -- '1' then exactly four more chars

When wildcards are not expressive enough, use a POSIX regular expression with ~ (case-sensitive) or ~* (case-insensitive). Regex gives you anchors (^, $), character classes ([0-9]), and alternation (a|b).

-- chinook: track names that begin with a digit
SELECT name FROM track WHERE name ~ '^[0-9]';         -- ^ = start, [0-9] = any digit
Why it mattersA pattern that starts with a wildcard ('%love%', ~ 'bach') cannot use a plain B-tree index, so it scans. Anchored patterns ('the%', ~ '^The') can. On the large stackoverflow tables, keep another filter alongside a leading-wildcard search so the query stays fast; Module 8 goes deep on this.

1.8 Paging with LIMIT and OFFSET

LIMIT n OFFSET m skips the first m rows and returns the next n - the mechanism behind "page 2, page 3" in an application. It is only meaningful with a fully deterministic ORDER BY; without a unique tiebreaker, rows can shuffle between pages and you will drop or repeat records.

-- chinook: page 2 of the longest tracks (10 per page, so skip the first 10)
SELECT track_id, name, milliseconds
FROM track
ORDER BY milliseconds DESC, track_id      -- track_id breaks ties so paging is stable
LIMIT 10 OFFSET 10;                  -- rows 11 through 20
Interview soundbite "Every paging query needs a tiebreaker in its ORDER BY - usually the primary key - or the pages are not stable. And large OFFSETs get slow because the database still walks and discards every skipped row; keyset pagination fixes that." The second sentence signals real production experience.

Practice - Session A (problems 1-4)

P1.1chinookwarmup

List the 15 longest tracks. Show the track name, its length in minutes (rounded to 1 decimal), and its unit price. Longest first.

hint
Length lives in milliseconds. Watch integer division; use ROUND(x, 1).
solution
SELECT name,
       ROUND(milliseconds / 60000.0, 1) AS minutes,
       unit_price
FROM track
ORDER BY milliseconds DESC
LIMIT 15;

Note we ORDER BY the raw column, not the rounded alias - rounding can create ties that reorder rows. Either is accepted, but knowing the difference is the point.

P1.2chinookwarmup

Find all tracks priced between 0.99 and 1.99 (inclusive) whose composer is recorded (not NULL), in genres 1, 2, or 7. Show name, composer, and unit_price - shortest tracks first, breaking length ties by name A-Z.

solution
SELECT name, composer, unit_price
FROM track
WHERE unit_price BETWEEN 0.99 AND 1.99
  AND composer IS NOT NULL
  AND genre_id IN (1, 2, 7)
ORDER BY milliseconds ASC, name ASC;

Four filters, three of them the workhorse trio (BETWEEN, IN, IS NOT NULL). The composer IS NOT NULL guard is the one beginners forget - without it you would silently include tracks with an unknown composer.

P1.3chinookcore

Show every customer with a country but no state recorded - the "state unknown" set that trips up naive filters. Return customer_id, first_name, last_name, country, ordered by country then last name.

hint
= NULL will silently return nothing. What is the correct predicate?
solution
SELECT customer_id, first_name, last_name, country
FROM customer
WHERE state IS NULL
ORDER BY country, last_name;

If asked "how would you verify this?", answer: compare COUNT(*) with COUNT(state) on the whole table - the difference should equal this query's row count, because COUNT(column) skips NULLs.

P1.4stackoverflowcore

Find users whose displayname contains "sql" in any casing and whose reputation is at least 1000. Show name, reputation, and location, highest reputation first, and push NULL locations to the end without excluding them.

solution
SELECT displayname, reputation, location
FROM users
WHERE displayname ILIKE '%sql%'
  AND reputation >= 1000
ORDER BY reputation DESC, location NULLS LAST;

Two Postgres-flavored details in one query: ILIKE and NULLS LAST. In a dialect without ILIKE you would write LOWER(displayname) LIKE '%sql%'. Remember the schema trap: it is displayname, one word, no underscore.

Practice - Session B (problems 5-8)

P1.5stackoverflowcore

Pull 25 suspicious posts: a huge view count (over 100000) but a non-positive score - the kind of read-heavy-yet-unloved content a data-quality sweep flags. Show id, title, viewcount, score, most-viewed first. (Hunting for these mismatches is a real senior analyst task.)

solution
SELECT id, title, viewcount, score
FROM posts
WHERE viewcount > 100000
  AND score <= 0
ORDER BY viewcount DESC
LIMIT 25;

On a 243k-row table this may be slow without an index - fine for now; Module 8 revisits exactly this shape of query with EXPLAIN ANALYZE. Note viewcount and score are both single-word lowercase columns.

P1.6chinookcore

List invoices billed to Germany or France where the total exceeded 10, OR any invoice (any country) with a total above 20. Parenthesize correctly. Return invoice_id, billing_country, total, biggest total first.

hint
Read it as two groups joined by OR: (country condition AND total > 10) OR (total > 20).
solution
SELECT invoice_id, billing_country, total
FROM invoice
WHERE (billing_country IN ('Germany', 'France') AND total > 10)
   OR total > 20
ORDER BY total DESC;

Without the parentheses, AND would bind first and quietly change the logic. This exact structure appears in interview screeners because it is quick to grade.

P1.7chinookcore

Produce the unique list of (billing_country, billing_city) pairs that appear on invoices, sorted by country then city. Then, without running it, predict: will SELECT DISTINCT billing_country return more, fewer, or the same number of rows?

solution
SELECT DISTINCT billing_country, billing_city
FROM invoice
ORDER BY billing_country, billing_city;

Fewer. DISTINCT on the pair keeps one row per unique combination; collapsing to country alone can only merge rows, never split them.

P1.8chinookcore

From track, list tracks with no composer recorded and a unit price above 0. Show name, album_id, unit_price, priciest first. State out loud (or in a comment) why composer != 'Bach' would not find these rows.

solution
SELECT name, album_id, unit_price
FROM track
WHERE composer IS NULL
  AND unit_price > 0
ORDER BY unit_price DESC;

composer != 'Bach' evaluates to NULL (not true) when composer is NULL, so those rows are dropped by WHERE. NULL rows only match IS NULL / IS NOT NULL / IS DISTINCT FROM.

Practice - Session C (problems 9-12)

P1.9stackoverflowcore

Return the 10 highest-scoring questions (not answers) created in 2018. Show title, score, viewcount, creationdate.

hint
posttypeid = 1 means question. For "created in 2018", a date range beats extracting the year (Module 8 explains why).
solution
SELECT title, score, viewcount, creationdate
FROM posts
WHERE posttypeid = 1
  AND creationdate >= DATE '2018-01-01'
  AND creationdate <  DATE '2019-01-01'
ORDER BY score DESC
LIMIT 10;

The half-open range [Jan 1, next Jan 1) is the professional idiom: it handles timestamps correctly (Dec 31 23:59:59 is included) and lets an index on creationdate do its job, unlike EXTRACT(YEAR ...) = 2018. Watch the column names: posttypeid, viewcount, creationdate - all lowercase, no underscores.

P1.10chinookinterview

From invoice_line, compute line revenue (unit_price * quantity) and return the 20 most valuable lines with invoice_id, track_id, quantity, unit_price, and the computed revenue rounded to 2 decimals. Then answer: why can you not write WHERE revenue > 1 using the alias?

solution
SELECT invoice_id, track_id, quantity, unit_price,
       ROUND((unit_price * quantity)::numeric, 2) AS revenue
FROM invoice_line
ORDER BY unit_price * quantity DESC
LIMIT 20;

WHERE is evaluated before SELECT in the logical order of execution, so the alias does not exist yet at WHERE time. ORDER BY runs after SELECT, which is why the alias would work there. This ordering question is a top-5 interview screener - Module 2 covers the full sequence. (Note chinook has no discount column, so line revenue is simply unit_price * quantity.)

P1.11chinookinterview

Find tracks where the composer is unknown (NULL) or the composer field contains the word "Bach" in any casing. Show track name and composer. How many rows does each branch of the OR contribute? (Answer with two extra COUNT queries.)

solution
SELECT name, composer
FROM track
WHERE composer IS NULL
   OR composer ILIKE '%bach%';

-- branch counts:
SELECT COUNT(*) FROM track WHERE composer IS NULL;
SELECT COUNT(*) FROM track WHERE composer ILIKE '%bach%';

The two branches cannot overlap (a NULL composer cannot also match ILIKE), so the branch counts sum to the total - a sanity check worth narrating. Here that is 977 + 8 = 985 rows.

P1.12stackoverflowinterview

Pull the 50 most recent questions (posttypeid 1) that scored exactly 0. Then explain: if score were nullable, what rows might this query miss, and how would you include them?

solution
SELECT id, title, creationdate, score
FROM posts
WHERE posttypeid = 1
  AND score = 0
ORDER BY creationdate DESC
LIMIT 50;

If score could be NULL, score = 0 misses NULL rows (an unknown score is not the same as a zero score - and saying that distinction out loud is the senior move). To treat unknown as zero: COALESCE(score, 0) = 0, while noting it may hide data-quality issues.

Practice - Session D (problems 13-17)

P1.13chinookcore

For every customer, show first_name, last_name, and a single region column that is the state when present and the country otherwise. Order by region, then last name. There should be no NULLs in the region column.

hint
COALESCE returns its first non-NULL argument. Feed it state first, then country.
solution
SELECT first_name, last_name,
       COALESCE(state, country) AS region
FROM customer
ORDER BY region, last_name;

29 of the 59 customers have no state, so without COALESCE a plain state column would be NULL for roughly half the rows. The fallback to country keeps every row labeled with something a human can read.

P1.14stackoverflowinterview

For users with reputation above 50000, compute an upvote ratio: upvotes / (upvotes + downvotes), rounded to 3 decimals. Some users may have zero total votes - your query must not error on those. Show displayname, upvotes, downvotes, and the ratio, highest reputation first, limit 20.

hint
Guard the denominator with NULLIF(upvotes + downvotes, 0) so a zero total yields NULL instead of a divide-by-zero error. Cast to numeric to avoid integer division.
solution
SELECT displayname, upvotes, downvotes,
       ROUND(upvotes::numeric / NULLIF(upvotes + downvotes, 0), 3) AS upvote_ratio
FROM users
WHERE reputation > 50000
ORDER BY reputation DESC
LIMIT 20;

Two guards in one expression: the ::numeric cast prevents integer truncation, and NULLIF(..., 0) prevents the divide-by-zero. A user with no votes gets a NULL ratio, which is honest - you cannot compute a ratio from nothing.

P1.15chinookinterview

Return exactly one row per genre: the shortest track in that genre. Show genre_id, the track name, and its length in milliseconds. Ignore tracks with no genre. Order by genre_id.

hint
This is the "one row per group, pick the smallest" shape. DISTINCT ON (genre_id) with an ORDER BY that leads on genre_id then sorts by milliseconds ASC keeps the shortest.
solution
SELECT DISTINCT ON (genre_id)
       genre_id, name, milliseconds
FROM track
WHERE genre_id IS NOT NULL
ORDER BY genre_id, milliseconds ASC;

The leading ORDER BY key must be genre_id to match the DISTINCT ON, and the second key (milliseconds ASC) decides which row within each genre survives. Swap it to DESC and you would get the longest track instead - that switch is the whole trick.

P1.16stackoverflowcore

Find the 10 highest-scoring questions (posttypeid 1) whose title mentions Postgres or PostgreSQL in any casing. Show id, title, and score, highest score first. Use a regular expression so a single pattern catches both spellings.

hint
Use the case-insensitive regex operator ~*. The pattern 'postgres(ql)?' makes the "ql" suffix optional. Keep the posttypeid = 1 filter so the query stays fast on this large table.
solution
SELECT id, title, score
FROM posts
WHERE posttypeid = 1
  AND title ~* 'postgres(ql)?'
ORDER BY score DESC
LIMIT 10;

The (ql)? group matches "postgres" and "postgresql" with one pattern, and ~* makes it case-insensitive so "Postgres" and "postgres" both hit. Doing this with LIKE would take two patterns joined by OR; regex is the cleaner tool when the shape of the match varies.

P1.17chinookcore

Imagine a track browser that shows 20 tracks per page, sorted by name A-Z. Return page 3 (tracks 41 through 60). Show track_id and name. Make sure the paging is stable even when two tracks share a name.

hint
Page 3 of 20-per-page means OFFSET 40. Add track_id as a tiebreaker in the ORDER BY so duplicate names cannot reshuffle between pages.
solution
SELECT track_id, name
FROM track
ORDER BY name, track_id
LIMIT 20 OFFSET 40;

The offset for page p at page size s is (p - 1) * s, so page 3 skips 40 rows. The track_id tiebreaker matters here because names like "Aces High" repeat - without it, the same title could land on two different pages across requests.
