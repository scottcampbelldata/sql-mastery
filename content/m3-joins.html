Joins

Everything downstream - CTEs, windows, interview patterns - is joins plus something. This module covers all the join types, the two traps that fail otherwise-strong candidates (fan-out and the LEFT-JOIN-killing WHERE clause), and the anti-join, which is the single most-asked join question in interviews.

3.1 INNER, LEFT, RIGHT, FULL in one mental model

A join matches rows by a condition. The only question the four types answer differently is: what happens to rows with no match?

TypeUnmatched left rowsUnmatched right rowsTypical use
INNER JOINdroppeddroppedonly complete pairs
LEFT JOINkept (right columns NULL)dropped"all customers, with orders if any"
RIGHT JOINdroppedkeptrare; rewrite as LEFT for readability
FULL JOINkeptkeptreconciling two lists
-- northwind: every order with its customer name
SELECT o.order_id, o.order_date, c.company_name
FROM orders o
JOIN customers c ON c.customer_id = o.customer_id;
-- Alias every table, prefix every column. Non-negotiable style in interviews.

3.2 The trap that kills LEFT JOINs

-- Goal: ALL customers, and their 1997 order count (zero if none).

-- WRONG: the WHERE filter on the right table silently converts LEFT to INNER,
-- because unmatched rows have o.order_date = NULL, and NULL fails the predicate.
SELECT c.customer_id, COUNT(o.order_id) AS orders_1997
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.customer_id
WHERE o.order_date >= DATE '1997-01-01'
  AND o.order_date <  DATE '1998-01-01'
GROUP BY c.customer_id;

-- RIGHT: put the right-table filter in the ON clause. Unmatched customers survive.
SELECT c.customer_id, COUNT(o.order_id) AS orders_1997
FROM customers c
LEFT JOIN orders o
       ON o.customer_id = c.customer_id
      AND o.order_date >= DATE '1997-01-01'
      AND o.order_date <  DATE '1998-01-01'
GROUP BY c.customer_id;
Rule With OUTER joins: conditions on the preserved (left) table go in WHERE, conditions on the nullable (right) table go in ON. With INNER joins it makes no difference to the result. Being able to state this rule is worth real points.

Also note COUNT(o.order_id), not COUNT(*): for a customer with no orders, COUNT(*) counts the one all-NULL joined row and reports 1; COUNT(o.order_id) correctly reports 0.

3.3 The anti-join: rows in A with no match in B

"Customers who never ordered" - the most common interview join question. Three ways:

-- 1. LEFT JOIN ... IS NULL
SELECT c.customer_id, c.company_name
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.customer_id
WHERE o.order_id IS NULL;

-- 2. NOT EXISTS (the one to prefer)
SELECT c.customer_id, c.company_name
FROM customers c
WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id
);

-- 3. NOT IN -- know it, and know why to avoid it:
-- if the subquery returns even one NULL, NOT IN returns ZERO rows total.
SELECT customer_id, company_name
FROM customers
WHERE customer_id NOT IN (SELECT customer_id FROM orders WHERE customer_id IS NOT NULL);
Interview soundbite "I default to NOT EXISTS: it is NULL-safe, and the planner treats it as a proper anti-join. NOT IN has a footgun where a single NULL in the subquery empties the result."

3.4 Fan-out: the silent revenue inflator

Joining a one-to-many relationship duplicates the "one" side's values across every matched row. Aggregate after joining without noticing, and your totals inflate.

-- northwind: orders (1) to order_details (many).
-- WRONG if you meant to sum freight once per order:
SELECT SUM(o.freight)
FROM orders o
JOIN order_details d ON d.order_id = o.order_id;
-- freight is repeated once per line item. A 3-line order counts its freight 3 times.

-- RIGHT: aggregate the many side first, then join at matching grain:
SELECT SUM(o.freight) AS total_freight,
       SUM(d.order_revenue) AS total_revenue
FROM orders o
JOIN (
    SELECT order_id,
           SUM(unit_price * quantity * (1 - discount)) AS order_revenue
    FROM order_details
    GROUP BY order_id
) d ON d.order_id = o.order_id;
The senior habit Before writing any join, say the grain of each table: "orders is one row per order; order_details is one row per order-product. Joining them puts me at line grain, so order-level columns will repeat." Interviewers listen for exactly this.

3.5 Self-joins and multi-hop joins

-- northwind: employee and manager on one row (self-join)
SELECT e.first_name || ' ' || e.last_name AS employee,
       m.first_name || ' ' || m.last_name AS manager
FROM employees e
LEFT JOIN employees m ON m.employee_id = e.reports_to;
-- LEFT so the top of the hierarchy (reports_to IS NULL) still appears.
-- chinook: four-table chain, genre revenue
SELECT g.name AS genre,
       ROUND(SUM(il.unit_price * il.quantity)::numeric, 2) AS revenue
FROM invoice_line il
JOIN track t ON t.track_id = il.track_id
JOIN genre g ON g.genre_id = t.genre_id
GROUP BY g.name
ORDER BY revenue DESC;

Practice - Session A (problems 1-3)

P3.1chinookwarmup

List every album title with its artist name, alphabetized by artist then album.

solution
SELECT ar.name AS artist, al.title AS album
FROM album al
JOIN artist ar ON ar.artist_id = al.artist_id
ORDER BY ar.name, al.title;
P3.2northwindwarmup

Every product with its category name and supplier company name. Include products even if the supplier link is missing.

hint
Which of the two joins needs to be LEFT to satisfy "even if missing"?
solution
SELECT p.product_name, c.category_name, s.company_name AS supplier
FROM products p
JOIN categories c ON c.category_id = p.category_id
LEFT JOIN suppliers s ON s.supplier_id = p.supplier_id
ORDER BY p.product_name;

Mixed join types in one query are normal - choose per relationship, not per query.

P3.3adventureworkscore

Walk the product hierarchy: every product's name, its subcategory name, and its category name. Keep products with no subcategory (show NULLs). Three tables, two hops.

solution
SELECT p.name AS product,
       ps.name AS subcategory,
       pc.name AS category
FROM production.product p
LEFT JOIN production.productsubcategory ps
       ON ps.productsubcategoryid = p.productsubcategoryid
LEFT JOIN production.productcategory pc
       ON pc.productcategoryid = ps.productcategoryid
ORDER BY category NULLS LAST, subcategory, product;

Both hops must be LEFT: an INNER on the second hop would drop the very rows the first LEFT preserved (their ps columns are NULL and cannot match).

Practice - Session B (problems 4-6)

P3.4chinookcore

Anti-join, all three ways: find every track that has never appeared on any invoice line. Write LEFT JOIN / IS NULL, NOT EXISTS, and NOT IN versions and confirm they return the same count.

solution
-- 1
SELECT COUNT(*) FROM track t
LEFT JOIN invoice_line il ON il.track_id = t.track_id
WHERE il.invoice_line_id IS NULL;

-- 2
SELECT COUNT(*) FROM track t
WHERE NOT EXISTS (SELECT 1 FROM invoice_line il WHERE il.track_id = t.track_id);

-- 3
SELECT COUNT(*) FROM track
WHERE track_id NOT IN (SELECT track_id FROM invoice_line);

All three match here because invoice_line.track_id has no NULLs. In an interview, write NOT EXISTS and mention the NOT IN NULL hazard unprompted.

P3.5stackoverflowcore

Self-join on posts: for the 20 highest-scoring questions, show the question title, question score, and the score of its accepted answer. Questions without an accepted answer must still appear.

hint
Join posts to posts on q.accepted_answer_id = a.id. Which side is preserved?
solution
SELECT q.title,
       q.score AS question_score,
       a.score AS accepted_answer_score
FROM posts q
LEFT JOIN posts a ON a.id = q.accepted_answer_id
WHERE q.post_type_id = 1
ORDER BY q.score DESC
LIMIT 20;

The WHERE filter is on q, the preserved side, so it does not break the LEFT JOIN - contrast with the 3.2 trap where the filter was on the nullable side.

P3.6northwindcore

The 3.2 trap, live: all customers with their count of orders shipped to Germany (ship_country = 'Germany'), including customers with zero. Verify your row count equals the customers table row count.

solution
SELECT c.customer_id, c.company_name,
       COUNT(o.order_id) AS german_orders
FROM customers c
LEFT JOIN orders o
       ON o.customer_id = c.customer_id
      AND o.ship_country = 'Germany'
GROUP BY c.customer_id, c.company_name
ORDER BY german_orders DESC;

The country filter lives in ON. Put it in WHERE and customers with no German orders vanish; the row count check catches it instantly. Making that check without being told is senior behavior.

Practice - Session C (problems 7-9)

P3.7chinookinterview

Five-table chain: total revenue per artist (invoice_line -> track -> album -> artist). Top 10 artists. Then answer: could an artist be missing from your result, and does that matter?

solution
SELECT ar.name AS artist,
       ROUND(SUM(il.unit_price * il.quantity)::numeric, 2) AS revenue
FROM invoice_line il
JOIN track t  ON t.track_id  = il.track_id
JOIN album al ON al.album_id = t.album_id
JOIN artist ar ON ar.artist_id = al.artist_id
GROUP BY ar.name
ORDER BY revenue DESC
LIMIT 10;

Yes: artists with zero sales never appear (INNER joins drop them). For a top-10 that is fine; for "revenue per artist including zero" you would start FROM artist and LEFT JOIN down the chain. Choosing the join direction from the question's wording is the skill being tested.

P3.8northwindinterview

Fan-out drill. Compute, in one query per order: order_id, freight, and total line revenue. Then produce the grand totals of both across all orders, being careful that freight is not inflated. (Two queries total: per-order, then grand total.)

hint
Per-order is safe if you GROUP BY order_id and take MAX(freight) or aggregate details in a subquery first. For grand totals, sum the pre-aggregated per-order rows.
solution
-- per order (subquery pre-aggregation, the pattern to internalize)
SELECT o.order_id, o.freight, d.line_revenue
FROM orders o
JOIN (
    SELECT order_id,
           SUM(unit_price * quantity * (1 - discount)) AS line_revenue
    FROM order_details
    GROUP BY order_id
) d ON d.order_id = o.order_id;

-- grand totals: wrap the same shape
SELECT SUM(o.freight) AS total_freight,
       SUM(d.line_revenue) AS total_revenue
FROM orders o
JOIN (
    SELECT order_id,
           SUM(unit_price * quantity * (1 - discount)) AS line_revenue
    FROM order_details
    GROUP BY order_id
) d ON d.order_id = o.order_id;

Sanity check: SELECT SUM(freight) FROM orders alone should match total_freight. If it does not, fan-out got you. This aggregate-then-join move becomes one CTE in Module 5.

P3.9northwindinterview

Finish P2.9 properly: average orders per customer including customers with zero orders. Compare the number with P2.9's answer.

solution
SELECT ROUND(AVG(order_count), 2) AS avg_orders_per_customer
FROM (
    SELECT c.customer_id, COUNT(o.order_id) AS order_count
    FROM customers c
    LEFT JOIN orders o ON o.customer_id = c.customer_id
    GROUP BY c.customer_id
) per_customer;

The average drops slightly because zero-order customers now pull it down. Presenting both numbers and asking "which does the business mean?" is the senior answer to an ambiguous prompt.

Practice - Session D (problems 10-12)

P3.10stackoverflowinterview

Users who have earned badges but have never posted anything. Return display_name, reputation, and their badge count. (Combines a join, an anti-join, and aggregation.)

solution
SELECT u.display_name, u.reputation, COUNT(b.id) AS badge_count
FROM users u
JOIN badges b ON b.user_id = u.id
WHERE NOT EXISTS (
    SELECT 1 FROM posts p WHERE p.owner_user_id = u.id
)
GROUP BY u.id, u.display_name, u.reputation
ORDER BY badge_count DESC;

Note GROUP BY includes u.id: two users can share a display_name, and grouping by the key preserves them as separate rows. Grouping by name alone would silently merge them - a subtle correctness point interviewers love.

P3.11chinookhard

Employee-customer-revenue chain: each support rep (employee) with the number of customers they support and those customers' total invoice revenue. Include reps with zero customers. Watch the double fan-out: employee -> customers -> invoices.

hint
Aggregate invoices to one row per customer first, then LEFT JOIN employee -> customer -> that aggregate.
solution
SELECT e.employee_id,
       e.first_name || ' ' || e.last_name AS rep,
       COUNT(DISTINCT c.customer_id) AS customers,
       COALESCE(SUM(ci.customer_revenue), 0) AS revenue
FROM employee e
LEFT JOIN customer c ON c.support_rep_id = e.employee_id
LEFT JOIN (
    SELECT customer_id, SUM(total) AS customer_revenue
    FROM invoice
    GROUP BY customer_id
) ci ON ci.customer_id = c.customer_id
GROUP BY e.employee_id, rep
ORDER BY revenue DESC;

Because invoices were pre-collapsed to one row per customer, the join cannot inflate revenue, and COUNT(DISTINCT c.customer_id) guards the customer count. COALESCE turns the no-customer reps' NULL sum into 0.

P3.12northwindhard

FULL JOIN reconciliation: build two lists - countries that appear as customer countries, and countries that appear as supplier countries - and produce one report showing every country, whether it has customers, suppliers, or both, with counts. (This "reconcile two lists" shape is a real senior take-home staple.)

solution
SELECT COALESCE(cu.country, su.country) AS country,
       COALESCE(cu.customer_count, 0)   AS customers,
       COALESCE(su.supplier_count, 0)   AS suppliers,
       CASE
         WHEN cu.country IS NULL THEN 'suppliers only'
         WHEN su.country IS NULL THEN 'customers only'
         ELSE 'both'
       END AS presence
FROM (
    SELECT country, COUNT(*) AS customer_count
    FROM customers GROUP BY country
) cu
FULL JOIN (
    SELECT country, COUNT(*) AS supplier_count
    FROM suppliers GROUP BY country
) su ON su.country = cu.country
ORDER BY country;

The FULL JOIN essentials in one query: COALESCE on the join key (either side can be NULL) and IS NULL tests to classify which side was missing.
