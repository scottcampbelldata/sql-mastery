Interview Patterns

Senior interviews recycle about a dozen named patterns in different costumes. Learn to hear the costume and name the pattern - "that's top-N per group," "that's a self-join on consecutive periods" - and the rest is typing. Each pattern below has the trigger phrases, the shape, and a worked example on your data.

Pattern 1: Deduplication

Trigger phrases: "remove duplicates," "keep the latest record per," "the table has multiple rows per user."

-- Shape: ROW_NUMBER over the duplicate key, ordered by the keep-rule, filter rn = 1.
-- northwind flavor: one row per customer = their most recent order
WITH ranked AS (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY customer_id
                              ORDER BY order_date DESC, order_id DESC) AS rn
    FROM orders
)
SELECT * FROM ranked WHERE rn = 1;

Always add a deterministic tiebreaker (order_id here). If asked to delete duplicates rather than select around them, the same ranking drives a DELETE with a USING or IN clause - describe it rather than run it in an interview.

Pattern 2: Nth highest

Trigger phrases: "second highest salary," "third most expensive," any ordinal.

-- The answer is DENSE_RANK (values), with ROW_NUMBER (rows) as the contrast:
WITH ranked AS (
    SELECT name, unit_price,
           DENSE_RANK() OVER (ORDER BY unit_price DESC) AS dr
    FROM track
)
SELECT * FROM ranked WHERE dr = 2;
-- Also know: OFFSET 1 LIMIT 1 after ORDER BY DESC (fails on ties, breaks with NULLs);
-- and MAX(x) WHERE x < (SELECT MAX(x)) (only reaches 2nd). Explain why windows win.

Pattern 3: Year-over-year / period-over-period

Trigger phrases: "compared to the same month last year," "growth rate," "MoM/YoY."

-- MoM = LAG(1). YoY on monthly data = LAG(12) - but ONLY if every month exists!
-- Missing months silently shift LAG(12) to the wrong row. The robust form is a self-join:
WITH monthly AS (
    SELECT date_trunc('month', orderdate)::date AS month,
           SUM(totaldue) AS revenue
    FROM sales.salesorderheader
    GROUP BY 1
)
SELECT cur.month,
       cur.revenue,
       prev.revenue AS revenue_last_year,
       ROUND(100.0 * (cur.revenue - prev.revenue) / NULLIF(prev.revenue, 0), 1) AS yoy_pct
FROM monthly cur
LEFT JOIN monthly prev ON prev.month = cur.month - INTERVAL '1 year'
ORDER BY cur.month;

Saying "LAG(12) assumes a gap-free series; the self-join on month minus one year is robust to gaps" is a top-tier senior answer.

Pattern 4: Cohort retention

Trigger phrases: "retention," "of users who signed up in January, how many were active in February," "cohort."

-- Shape: (1) each user's cohort = first activity month, (2) all activity months,
-- (3) join and count distinct users per (cohort, months_since).
-- chinook flavor: purchase retention by first-purchase month
WITH cohorts AS (
    SELECT customer_id,
           date_trunc('month', MIN(invoice_date))::date AS cohort_month
    FROM invoice
    GROUP BY customer_id
),
activity AS (
    SELECT DISTINCT customer_id,
           date_trunc('month', invoice_date)::date AS active_month
    FROM invoice
)
SELECT c.cohort_month,
       (EXTRACT(YEAR FROM a.active_month)  - EXTRACT(YEAR FROM c.cohort_month)) * 12
     + (EXTRACT(MONTH FROM a.active_month) - EXTRACT(MONTH FROM c.cohort_month)) AS months_since,
       COUNT(DISTINCT a.customer_id) AS active_customers
FROM cohorts c
JOIN activity a ON a.customer_id = c.customer_id
GROUP BY 1, 2
ORDER BY 1, 2;

Pattern 5: Consecutive events / gaps and islands

Trigger phrases: "3 days in a row," "consecutive," "streak," "continuous ranges."

-- The magic trick: for consecutive sequences, (value - ROW_NUMBER) is constant per island.
-- northwind flavor: streaks of consecutive order DATES (days on which any order arrived)
WITH days AS (
    SELECT DISTINCT order_date FROM orders
),
grouped AS (
    SELECT order_date,
           order_date - (ROW_NUMBER() OVER (ORDER BY order_date))::int AS island_key
    FROM days
)
SELECT MIN(order_date) AS streak_start,
       MAX(order_date) AS streak_end,
       COUNT(*)        AS streak_days
FROM grouped
GROUP BY island_key
ORDER BY streak_days DESC;

Why it works: along a run of consecutive dates, both the date and the row number increase by 1 per row, so their difference is constant - and it changes the moment a day is skipped. Walk the interviewer through that sentence; it is the whole trick.

Pattern 6: Pivoting with conditional aggregation

Trigger phrases: "one column per year," "reshape long to wide," "crosstab."

-- northwind: category revenue by year, years as columns
SELECT c.category_name,
  ROUND(SUM(d.unit_price * d.quantity * (1-d.discount)) FILTER (WHERE o.order_date >= '1996-01-01' AND o.order_date < '1997-01-01')::numeric, 0) AS y1996,
  ROUND(SUM(d.unit_price * d.quantity * (1-d.discount)) FILTER (WHERE o.order_date >= '1997-01-01' AND o.order_date < '1998-01-01')::numeric, 0) AS y1997,
  ROUND(SUM(d.unit_price * d.quantity * (1-d.discount)) FILTER (WHERE o.order_date >= '1998-01-01' AND o.order_date < '1999-01-01')::numeric, 0) AS y1998
FROM order_details d
JOIN orders o     ON o.order_id = d.order_id
JOIN products p   ON p.product_id = d.product_id
JOIN categories c ON c.category_id = p.category_id
GROUP BY c.category_name
ORDER BY c.category_name;

SQL cannot make columns dynamically - the column list is fixed at write time. If asked "what if the years are not known in advance," the answer is: generate the SQL programmatically or pivot in the BI layer. Knowing the boundary is the senior part.

Pattern 7: Sessionization

Trigger phrases: "group events into sessions," "a new session starts after 30 minutes of inactivity."

-- Shape: LAG to measure the gap, CASE to flag session starts, running SUM of flags = session id.
-- stackoverflow flavor: a user's posting sessions (new session after 1 hour idle)
WITH events AS (
    SELECT owner_user_id, creation_date,
           CASE WHEN creation_date - LAG(creation_date) OVER w > INTERVAL '1 hour'
                  OR LAG(creation_date) OVER w IS NULL
                THEN 1 ELSE 0 END AS new_session
    FROM posts
    WHERE owner_user_id = 22656
    WINDOW w AS (PARTITION BY owner_user_id ORDER BY creation_date)
)
SELECT *,
       SUM(new_session) OVER (PARTITION BY owner_user_id
                              ORDER BY creation_date) AS session_id
FROM events;

Flag-then-running-sum is the same engine as gaps and islands; recognizing that they are one family is what makes both easy to reconstruct under pressure.

Practice - twelve problems, mixed costume

The prompts below deliberately do not name their pattern. Your first job on each: say which pattern it is before writing a line.

P7.1stackoverflowinterview

Some users appear to have duplicate badge awards (same user, same badge name, possibly different dates). Produce one row per (user, badge name) keeping the earliest award date, for badges named 'Nice Answer'.

hint
Pattern 1. The duplicate key is (user_id, name); the keep-rule is earliest date.
solution
WITH ranked AS (
    SELECT user_id, name, date,
           ROW_NUMBER() OVER (PARTITION BY user_id, name
                              ORDER BY date, id) AS rn
    FROM badges
    WHERE name = 'Nice Answer'
)
SELECT user_id, name, date FROM ranked WHERE rn = 1;

Alternative for "earliest date only": plain GROUP BY user_id, name with MIN(date). ROW_NUMBER wins when you must keep the whole row, not just the date - say that distinction out loud.

P7.2chinookinterview

What is the third-highest invoice total, and which invoices hit it? All ties must appear.

solution
WITH ranked AS (
    SELECT invoice_id, customer_id, total,
           DENSE_RANK() OVER (ORDER BY total DESC) AS dr
    FROM invoice
)
SELECT * FROM ranked WHERE dr = 3;

"All ties must appear" + "third-highest value" = DENSE_RANK, stated before typing. That is the whole grade.

P7.3nyctaxiinterview

Weekly trip volume with percent change from the prior week, and flag weeks that dropped more than 10%.

solution
WITH weekly AS (
    SELECT date_trunc('week', pickup_datetime)::date AS week,
           COUNT(*) AS trips
    FROM trips
    GROUP BY 1
),
compared AS (
    SELECT week, trips,
           ROUND(100.0 * (trips - LAG(trips) OVER (ORDER BY week))
                 / NULLIF(LAG(trips) OVER (ORDER BY week), 0), 1) AS wow_pct
    FROM weekly
)
SELECT *,
       CASE WHEN wow_pct < -10 THEN 'ALERT' END AS flag
FROM compared
ORDER BY week;

Note the second CTE exists so the CASE can reference wow_pct by name - you cannot filter or branch on a window in the same SELECT that defines it. Partial first and last weeks will look like fake drops; mention trimming them.

P7.4northwindinterview

Of the customers who placed their first order in 1996, what percentage also placed at least one order in 1997? And in 1998?

hint
Pattern 4, simplified to one cohort. CTE for first-order year = 1996 customers; conditional EXISTS or FILTER for later-year activity.
solution
WITH cohort_1996 AS (
    SELECT customer_id
    FROM orders
    GROUP BY customer_id
    HAVING EXTRACT(YEAR FROM MIN(order_date)) = 1996
)
SELECT COUNT(*) AS cohort_size,
       ROUND(100.0 * COUNT(*) FILTER (WHERE EXISTS (
           SELECT 1 FROM orders o
           WHERE o.customer_id = c.customer_id
             AND o.order_date >= '1997-01-01' AND o.order_date < '1998-01-01'))
           / COUNT(*), 1) AS pct_active_1997,
       ROUND(100.0 * COUNT(*) FILTER (WHERE EXISTS (
           SELECT 1 FROM orders o
           WHERE o.customer_id = c.customer_id
             AND o.order_date >= '1998-01-01' AND o.order_date < '1999-01-01'))
           / COUNT(*), 1) AS pct_active_1998
FROM cohort_1996 c;

Retention questions always hide a definition question: "active" = placed an order? logged in? Ask. Here the data only knows orders, so state the assumption and proceed.

P7.5stackoverflowhard

Find users who posted on at least 5 consecutive calendar days at least once. Return the user, their longest streak, and its start and end dates. (Pick a scoped set of users first if the table is huge, e.g. reputation > 50000.)

hint
Pattern 5 with a partition: DISTINCT (user, day), then day - ROW_NUMBER() OVER (PARTITION BY user ORDER BY day) as the island key.
solution
WITH user_days AS (
    SELECT DISTINCT p.owner_user_id, p.creation_date::date AS day
    FROM posts p
    JOIN users u ON u.id = p.owner_user_id
    WHERE u.reputation > 50000
),
islands AS (
    SELECT owner_user_id, day,
           day - (ROW_NUMBER() OVER (PARTITION BY owner_user_id
                                     ORDER BY day))::int AS island_key
    FROM user_days
),
streaks AS (
    SELECT owner_user_id,
           MIN(day) AS streak_start,
           MAX(day) AS streak_end,
           COUNT(*) AS streak_days
    FROM islands
    GROUP BY owner_user_id, island_key
),
best AS (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY owner_user_id
                              ORDER BY streak_days DESC, streak_start) AS rn
    FROM streaks
    WHERE streak_days >= 5
)
SELECT owner_user_id, streak_days, streak_start, streak_end
FROM best
WHERE rn = 1
ORDER BY streak_days DESC;

Four CTEs, each a named step, and the final one is Pattern 1 (top-1 per user) riding on top of Pattern 5. Compound patterns like this are exactly what a hard senior round looks like.

P7.6chinookinterview

Build a wide report: one row per billing country, with total revenue in columns for 2021, 2022, and 2023 (adjust to the years in your data - find them first).

solution
-- find the years first (Module 0 reflex):
SELECT MIN(invoice_date), MAX(invoice_date) FROM invoice;

SELECT billing_country,
  SUM(total) FILTER (WHERE EXTRACT(YEAR FROM invoice_date) = 2021) AS y2021,
  SUM(total) FILTER (WHERE EXTRACT(YEAR FROM invoice_date) = 2022) AS y2022,
  SUM(total) FILTER (WHERE EXTRACT(YEAR FROM invoice_date) = 2023) AS y2023
FROM invoice
GROUP BY billing_country
ORDER BY billing_country;

Countries with no sales in a year show NULL, not 0 - wrap in COALESCE if the report needs zeros. Small detail, real report bug.

P7.7nyctaxihard

Fare distribution summary in one row: min, 25th percentile, median, 75th, 95th, 99th, max, and mean of fare_amount (positive fares only). Then: which single number would you report to a stakeholder as "typical fare," and why?

solution
SELECT
  MIN(fare_amount) AS min,
  PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY fare_amount) AS p25,
  PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY fare_amount) AS median,
  PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY fare_amount) AS p75,
  PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY fare_amount) AS p95,
  PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY fare_amount) AS p99,
  MAX(fare_amount) AS max,
  ROUND(AVG(fare_amount)::numeric, 2) AS mean
FROM trips
WHERE fare_amount > 0;

Median. Fares are right-skewed (airport runs, data errors), so the mean sits above what most riders pay. p95/p99 vs max also quantifies how bad the outliers are - a data-quality statement in the same breath.

P7.8stackoverflowhard

Sessionize one busy user's comments with a 30-minute idle threshold: session id, session start, session end, comments in session. Then their average session length in minutes.

solution
WITH flagged AS (
    SELECT user_id, creation_date,
           CASE WHEN creation_date - LAG(creation_date) OVER w > INTERVAL '30 minutes'
                  OR LAG(creation_date) OVER w IS NULL
                THEN 1 ELSE 0 END AS is_start
    FROM comments
    WHERE user_id = 22656   -- swap in a busy user from your data
    WINDOW w AS (ORDER BY creation_date)
),
sessions AS (
    SELECT *,
           SUM(is_start) OVER (ORDER BY creation_date) AS session_id
    FROM flagged
),
summarized AS (
    SELECT session_id,
           MIN(creation_date) AS session_start,
           MAX(creation_date) AS session_end,
           COUNT(*) AS comments
    FROM sessions
    GROUP BY session_id
)
SELECT *,
       ROUND((EXTRACT(EPOCH FROM session_end - session_start) / 60)::numeric, 1) AS minutes
FROM summarized
ORDER BY session_id;
-- average session length: wrap once more, AVG(minutes).

Flag, running-sum, group - once you can produce these three steps from memory, every sessionization variant (30 min, 1 hour, per device, per user) is a parameter change.

P7.9adventureworkshard

YoY by quarter, robustly: quarterly revenue with same-quarter-last-year comparison using the self-join form (not LAG(4)). Include quarters where last year is missing, showing NULL.

solution
WITH quarterly AS (
    SELECT date_trunc('quarter', orderdate)::date AS qtr,
           SUM(totaldue) AS revenue
    FROM sales.salesorderheader
    GROUP BY 1
)
SELECT cur.qtr,
       ROUND(cur.revenue::numeric, 0)  AS revenue,
       ROUND(prev.revenue::numeric, 0) AS same_qtr_last_year,
       ROUND(100.0 * (cur.revenue - prev.revenue) / NULLIF(prev.revenue, 0), 1) AS yoy_pct
FROM quarterly cur
LEFT JOIN quarterly prev
       ON prev.qtr = (cur.qtr - INTERVAL '1 year')::date
ORDER BY cur.qtr;

LEFT JOIN keeps year-one quarters with NULL comparison instead of dropping them - which is honest reporting. The cast on the join condition keeps both sides date-typed.

P7.10northwindhard

Share-shift analysis: each category's share of total revenue in 1997 vs 1996, and the change in percentage points. Which categories gained share? (Combines pivot, percent-of-total, and a derived comparison.)

solution
WITH cat_year AS (
    SELECT c.category_name,
           EXTRACT(YEAR FROM o.order_date)::int AS yr,
           SUM(d.unit_price * d.quantity * (1 - d.discount)) AS revenue
    FROM order_details d
    JOIN orders o     ON o.order_id = d.order_id
    JOIN products p   ON p.product_id = d.product_id
    JOIN categories c ON c.category_id = p.category_id
    WHERE o.order_date >= '1996-01-01' AND o.order_date < '1998-01-01'
    GROUP BY 1, 2
),
shares AS (
    SELECT category_name, yr,
           100.0 * revenue / SUM(revenue) OVER (PARTITION BY yr) AS share_pct
    FROM cat_year
)
SELECT category_name,
       ROUND(MAX(share_pct) FILTER (WHERE yr = 1996)::numeric, 1) AS share_1996,
       ROUND(MAX(share_pct) FILTER (WHERE yr = 1997)::numeric, 1) AS share_1997,
       ROUND((MAX(share_pct) FILTER (WHERE yr = 1997)
            - MAX(share_pct) FILTER (WHERE yr = 1996))::numeric, 1) AS pp_change
FROM shares
GROUP BY category_name
ORDER BY pp_change DESC;

MAX(x) FILTER (WHERE yr = ...) is the standard trick for pivoting a value you know is unique per (group, year) - MAX of one value is that value. Percentage points, not percent change of a percentage - vocabulary interviewers absolutely notice.

P7.11chinookhard

Customer lifecycle snapshot: one row per customer with first purchase date, last purchase date, total invoices, lifetime value, and a status of 'active' (purchased in the final 6 months of the dataset), 'lapsing' (6-12 months), or 'churned' (older). Anchor "now" to MAX(invoice_date) in the data, not the real today.

solution
WITH anchor AS (
    SELECT MAX(invoice_date) AS data_now FROM invoice
),
per_customer AS (
    SELECT customer_id,
           MIN(invoice_date) AS first_purchase,
           MAX(invoice_date) AS last_purchase,
           COUNT(*)          AS invoices,
           SUM(total)        AS ltv
    FROM invoice
    GROUP BY customer_id
)
SELECT p.customer_id, p.first_purchase::date, p.last_purchase::date,
       p.invoices, ROUND(p.ltv::numeric, 2) AS ltv,
       CASE
         WHEN p.last_purchase >= a.data_now - INTERVAL '6 months'  THEN 'active'
         WHEN p.last_purchase >= a.data_now - INTERVAL '12 months' THEN 'lapsing'
         ELSE 'churned'
       END AS status
FROM per_customer p
CROSS JOIN anchor a
ORDER BY ltv DESC;

Anchoring "now" to the data's own max date makes results reproducible on a static dataset - a detail that reads as real analyst experience, because it is.

P7.12nyctaxihard

Capstone: build a one-query "vendor scorecard" - per vendor_id: trips, revenue, revenue share of total, median fare, average trips per day, and rank by revenue. Present it as if handing a table to a director.

solution
WITH per_vendor AS (
    SELECT vendor_id,
           COUNT(*)          AS trips,
           SUM(total_amount) AS revenue,
           PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY fare_amount) AS median_fare,
           COUNT(DISTINCT pickup_datetime::date) AS active_days
    FROM trips
    WHERE fare_amount > 0
    GROUP BY vendor_id
)
SELECT vendor_id,
       trips,
       ROUND(revenue::numeric, 0) AS revenue,
       ROUND(100.0 * revenue / SUM(revenue) OVER (), 1) AS revenue_share_pct,
       ROUND(median_fare::numeric, 2) AS median_fare,
       ROUND(trips::numeric / active_days, 0) AS avg_trips_per_day,
       RANK() OVER (ORDER BY revenue DESC) AS revenue_rank
FROM per_vendor
ORDER BY revenue_rank;

Everything in one place: aggregation, ordered-set aggregate, window over the grouped result, defensive filter. If you can write this cold and narrate it, you are interview-ready on patterns - the remaining work is speed and performance talk.
