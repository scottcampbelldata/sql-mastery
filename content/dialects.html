SQL Dialects: Postgres vs. BigQuery vs. Snowflake

The logic you have learned here is standard SQL and ports everywhere. What changes between engines is the surface: string concatenation, row limiting, identifier quoting, booleans, case-insensitive matching, date functions, casting, NULL ordering, string aggregation, and write-path features like upsert and RETURNING. This is the reference to skim before a BI interview - your engine here is PostgreSQL, so the Postgres column is what you have been writing; the other columns (BigQuery and Snowflake in the warehouse tables, MySQL, SQL Server, and SQLite in the engine tables) are the ones you will be asked to "also know." When unsure in an interview, say the standard-SQL form and note the dialect difference out loud.

String concatenation in depth

Three ways to glue strings together, and they disagree about NULL. Postgres gives you both || and CONCAT; the difference is what happens when a piece is NULL. The || operator is NULL-propagating - one NULL input makes the whole result NULL - while CONCAT treats NULL as an empty string. This is the single most common source of "why is my label blank" bugs.

-- Same two rows, two behaviors. company is NULL here.
SELECT company,
       company || ' (VIP)'        AS piped,   -- NULL: || propagates
       CONCAT(company, ' (VIP)') AS func     -- ' (VIP)': CONCAT skips NULL
FROM customer
WHERE company IS NULL
LIMIT 2;

The fix when you want || but must survive NULLs is to guard the optional piece so the separator disappears with it:

-- Attach company only when present, no dangling space.
SELECT first_name || COALESCE(' ' || company, '') AS label
FROM customer
ORDER BY customer_id
LIMIT 3;
-- Luis ... / Leonie / Francois (no trailing space when company is NULL)

Row limiting: LIMIT, TOP, and FETCH FIRST

You have been writing LIMIT here, and it is the shortest form. But the ANSI-standard spelling is FETCH FIRST n ROWS ONLY, and SQL Server uses TOP. Postgres accepts both LIMIT and the standard FETCH FIRST, which is handy to know because the standard form also supports WITH TIES - keep every row that ties the last one on the ORDER BY.

-- Standard spelling; equivalent to LIMIT 3 here.
SELECT name, milliseconds
FROM track
ORDER BY milliseconds DESC
FETCH FIRST 3 ROWS ONLY;

-- WITH TIES keeps extra rows that match the 3rd row's length.
-- Needs an ORDER BY to define what "tie" means.
SELECT name, milliseconds
FROM track
ORDER BY milliseconds DESC
FETCH FIRST 3 ROWS WITH TIES;

The one that trips people up SQL Server's OFFSET ... FETCH requires an ORDER BY - it will not run without one. And plain TOP without ORDER BY gives you an arbitrary 10 rows, not a meaningful "top." MySQL's two-argument LIMIT 20, 10 reverses the intuition: the first number is the offset, the second is the count. When you cross engines, always pair a row limit with an explicit ORDER BY.

Identifiers, booleans, and case-insensitive matching

Two surface details bite you the moment you move a query between engines: how you quote a weird column name, and whether the engine even has a real boolean type. Postgres quotes identifiers with double quotes and has a first-class boolean type; a comparison like score > 0 is a value you can select, not just a filter.

-- Double-quote an identifier to allow spaces or force casing.
-- Double quotes = identifier, single quotes = string literal.
SELECT t.name AS "Track Name"
FROM track t
ORDER BY t.track_id
LIMIT 2;

-- A boolean expression is a selectable value in Postgres.
SELECT id, (score > 0) AS is_positive
FROM posts
WHERE posttypeid = 1
ORDER BY id
LIMIT 3;

For case-insensitive matching, Postgres gives you ILIKE. Other engines fall back to lowering both sides, and some are case-insensitive by default depending on the column's collation.

-- ILIKE is Postgres-only shorthand for case-insensitive LIKE.
SELECT name
FROM track
WHERE name ILIKE '%love%'
ORDER BY name
LIMIT 3;

String aggregation across engines

Rolling many rows into one delimited string is the same idea everywhere but has a different name in each engine. Postgres calls it STRING_AGG, and it takes an optional ORDER BY inside the call so the concatenated order is deterministic.

-- All track names in one genre, one row, ordered.
SELECT g.name AS genre,
       STRING_AGG(t.name, '; ' ORDER BY t.name) AS tracks
FROM genre g
JOIN track t ON t.genre_id = g.genre_id
WHERE g.name = 'Opera'
GROUP BY g.name;

-- Distinct badge names for one Stack Overflow user, alphabetized.
SELECT u.displayname,
       STRING_AGG(DISTINCT b.name, ', ' ORDER BY b.name) AS badges
FROM users u
JOIN badges b ON b.userid = u.id
WHERE u.id = 162483
GROUP BY u.displayname;

And SQL Server's STRING_AGG puts the ordering in a WITHIN GROUP clause, not inside the parentheses like Postgres. When an interviewer asks you to "list the values per group," name the function for their stack: STRING_AGG, GROUP_CONCAT, or LISTAGG.

Upsert, RETURNING, and generated keys

Write-path features diverge more than read-path ones, so these are worth recognizing even though our read-only sandbox will not run the write examples. The pattern names are what interviewers listen for. An upsert - insert, but update instead if the row already exists - is ON CONFLICT in Postgres, ON DUPLICATE KEY UPDATE in MySQL, and the ANSI MERGE statement in SQL Server and Oracle.

-- PostgreSQL upsert (illustrative; our sandbox is read-only).
INSERT INTO genre (genre_id, name)
VALUES (1, 'Rock')
ON CONFLICT (genre_id)
DO UPDATE SET name = EXCLUDED.name;

Postgres also has RETURNING, which hands back the rows an INSERT, UPDATE, or DELETE just wrote - so you can get a generated key without a second round trip.

-- RETURNING gives back the row the write produced.
INSERT INTO genre (name)
VALUES ('Synthwave')
RETURNING genre_id, name;

Feature support at a glance

Finally, a support matrix for the analytical features you lean on. The good news for interviews: CTEs and window functions are now supported across every major engine, including recent MySQL and SQLite. The details that still vary are NULL ordering control, QUALIFY, and FILTER.

-- NULLS LAST is native in Postgres: push missing values to the end.
SELECT first_name, company
FROM customer
ORDER BY company ASC NULLS LAST, first_name
LIMIT 4;

-- FILTER is a clean Postgres way to do conditional aggregates.
SELECT COUNT(*) FILTER (WHERE score > 0) AS positive,
       COUNT(*) AS total
FROM posts
WHERE posttypeid = 1
  AND creationdate >= '2020-01-01';

The one that trips people up MySQL and SQL Server have no NULLS LAST, so the portable trick is to sort on a computed flag first: ORDER BY (company IS NULL), company pushes NULLs to the end because FALSE (0) sorts before TRUE (1). Likewise, when FILTER is unavailable, the universal fallback is COUNT(CASE WHEN score > 0 THEN 1 END). Knowing the fallback for each convenience is what lets you write clean SQL on any engine.

How to use this

Do not memorize it cold. Learn the shape of each operation in Postgres here - split-then-explode, truncate-then-group, guard-with-NULLIF - and keep this page for the day you switch dialects. Interview move: write the standard form, and when a function name might differ, say "in BigQuery that's SPLIT and UNNEST; in Snowflake it's FLATTEN" - that one sentence turns a syntax gap into a demonstration of range.
