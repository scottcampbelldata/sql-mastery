-- module 8 / three sessions /

Performance

At senior level nobody expects you to be a DBA. They expect you to (a) read an EXPLAIN plan without panic, (b) know what an index does and when it helps, (c) not write queries that defeat indexes, and (d) talk about all three calmly. Your Stack Overflow and NYC Taxi databases are big enough to make every one of these lessons visible on a stopwatch.

8.1 EXPLAIN and EXPLAIN ANALYZE

-- EXPLAIN shows the plan; EXPLAIN ANALYZE runs the query and shows reality:
EXPLAIN ANALYZE
SELECT * FROM trips WHERE fare_amount > 200 AND trip_distance < 1;

How to read the output, in the order that matters:

  • The node types. Seq Scan = read the whole table. Index Scan = walk the index to the matching rows. Index Only Scan = the index alone answered it. Bitmap Heap Scan = index found many scattered rows, fetched in batches.
  • Join strategies. Nested Loop (good for few rows with an index on the inner side), Hash Join (build a hash of one side; the default for big unsorted inputs), Merge Join (both sides sorted).
  • Estimated vs actual rows. rows=1000 estimated but actual rows=2000000 means stale statistics led the planner astray - ANALYZE tablename; refreshes them. Spotting a bad estimate is a genuinely impressive interview move.
  • Where the milliseconds went. The deepest expensive node is usually the story.
Interview script "My first move on a slow query is EXPLAIN ANALYZE. I look for sequential scans on large tables that a filter should have narrowed, then check whether an index exists and whether my predicate is written so the index can be used, then check row estimates against actuals." That paragraph, delivered calmly, passes most performance screens on its own.

8.2 Indexes in one page

-- A B-tree index is a sorted lookup structure on one or more columns:
CREATE INDEX idx_trips_pickup ON trips (pickup_datetime);
CREATE INDEX idx_posts_owner_type ON posts (owner_user_id, post_type_id);
  • Helps: equality and range predicates on the leading column(s), JOIN keys, ORDER BY matching the index order.
  • Column order matters: an index on (a, b) serves WHERE a = ... and WHERE a = ... AND b = ..., but not WHERE b = ... alone. Rule of thumb: equality columns first, then the range column.
  • Costs: every index slows INSERT/UPDATE/DELETE and takes disk. Analysts reading from a warehouse rarely pay this, but say it to show you know writes exist.
  • Won't help: low-selectivity predicates (WHERE payment_type = 1 matching 60% of rows - the planner rightly prefers a full scan), tiny tables, or predicates that wrap the column in a function.

8.3 Sargability: don't handcuff the index

A predicate is sargable ("search-argument-able") when the column stands alone on one side of the comparison. Wrap the column in a function and the index on it becomes useless.

-- NOT sargable: the function hides the column from the index
WHERE EXTRACT(YEAR FROM pickup_datetime) = 2016
WHERE pickup_datetime::date = '2016-01-15'
WHERE LOWER(display_name) = 'jon skeet'

-- Sargable rewrites: move the work to the constant side
WHERE pickup_datetime >= '2016-01-01' AND pickup_datetime < '2017-01-01'
WHERE pickup_datetime >= '2016-01-15' AND pickup_datetime < '2016-01-16'
-- (for the LOWER case: create an expression index ON users (LOWER(display_name)))

This is why every date filter in this curriculum used half-open ranges. It was performance training in disguise.

8.4 The talking points that round out senior answers

  • Filter early, join late: reduce rows in a CTE before joining, especially before joining two big tables.
  • SELECT only what you need: SELECT * on wide tables drags dead columns through every node and can prevent index-only scans.
  • COUNT(DISTINCT x) is expensive on big data (requires dedup); approximate counts exist in warehouses (HLL) - name-dropping that shows range.
  • CTEs and the optimizer: since PostgreSQL 12, plain CTEs are inlined (no artificial fence) unless you write WITH x AS MATERIALIZED (...) to force materialization - useful when a CTE is referenced many times. Knowing this kills the outdated "CTEs are slow" myth interviewers sometimes probe with.
  • EXISTS vs IN vs JOIN: usually the same plan; choose for correctness and readability (Modules 3 and 5).
  • LIMIT is not free if there is an ORDER BY on an unindexed column - the whole set must be sorted before the top rows are known.

Drills (all timed against your own hardware)

P8.1nyctaxicore

Baseline read: run EXPLAIN ANALYZE on the P1.5 query (fare > 200, distance < 1). Identify the scan type, total runtime, and estimated vs actual rows. Write the three numbers down.

solution
EXPLAIN ANALYZE
SELECT pickup_datetime, trip_distance, fare_amount, tip_amount
FROM trips
WHERE fare_amount > 200 AND trip_distance < 1
ORDER BY fare_amount DESC
LIMIT 25;

Expect a Seq Scan (possibly parallel) - no index covers these columns - followed by a Sort feeding the Limit (probably reported as top-N heapsort). This is your "before" picture for P8.2.

P8.2nyctaxiinterview

Create an index on fare_amount, rerun P8.1's EXPLAIN ANALYZE, and compare. Then answer honestly: did it help much, and why or why not?

solution
CREATE INDEX idx_trips_fare ON trips (fare_amount);
ANALYZE trips;
-- rerun the EXPLAIN ANALYZE from P8.1

fare_amount > 200 is highly selective (few trips), so the planner should switch to an Index Scan or Bitmap Heap Scan and the runtime should drop hard. The distance predicate is then applied as a filter on the fetched rows. If your planner still chooses Seq Scan, check that ANALYZE ran and consider how selective > 200 actually is in your data - reasoning about that is the drill. Clean up after: DROP INDEX idx_trips_fare; if you want a pristine baseline later.

P8.3nyctaxiinterview

Sargability, on a stopwatch: with an index on pickup_datetime in place, time EXTRACT(YEAR FROM pickup_datetime) = 2016 AND EXTRACT(MONTH ...) = 3 against the half-open range version for March 2016. Compare plans and runtimes.

solution
CREATE INDEX IF NOT EXISTS idx_trips_pickup ON trips (pickup_datetime);

EXPLAIN ANALYZE
SELECT COUNT(*) FROM trips
WHERE EXTRACT(YEAR FROM pickup_datetime) = 2016
  AND EXTRACT(MONTH FROM pickup_datetime) = 3;

EXPLAIN ANALYZE
SELECT COUNT(*) FROM trips
WHERE pickup_datetime >= '2016-03-01'
  AND pickup_datetime <  '2016-04-01';

Version 1: Seq Scan, every row evaluated. Version 2: index range scan. Same answer, different orders of magnitude. This before/after is the single best performance story you can tell in an interview, because you have personally measured it.

P8.4stackoverflowinterview

Join strategy safari: EXPLAIN (no ANALYZE needed) the P3.10 query (badge earners who never posted). Which join strategies did the planner pick for the JOIN and for the NOT EXISTS? Why do they make sense?

solution

Typical plan: a Hash Join between users and badges (both large, unsorted - hash the smaller side), and the NOT EXISTS appears as a Hash Anti Join against posts. The word "anti join" in the plan is the payoff: the planner recognizes NOT EXISTS as a first-class operation, which is exactly the argument for preferring it over NOT IN. Screenshot-level familiarity with these plan node names is what "can read a plan" means at interview time.

P8.5stackoverflowinterview

Composite index design: you frequently run "answers by user X, newest first" - WHERE owner_user_id = ? AND post_type_id = 2 ORDER BY creation_date DESC. Design the best single index, create it, and prove with EXPLAIN that the sort disappears.

hint
Equality columns first (both of them), then the ORDER BY column. Three columns total.
solution
CREATE INDEX idx_posts_owner_type_date
    ON posts (owner_user_id, post_type_id, creation_date DESC);

EXPLAIN ANALYZE
SELECT id, title, score, creation_date
FROM posts
WHERE owner_user_id = 22656
  AND post_type_id = 2
ORDER BY creation_date DESC
LIMIT 20;

The plan should show an Index Scan on the new index with no separate Sort node - the index already delivers rows in the right order, so LIMIT 20 stops after 20 rows. "Index serves the filter AND the sort" is the sentence to say.

P8.6nyctaxihard

Aggregation cost: time COUNT(*), COUNT(DISTINCT passenger_count), and COUNT(DISTINCT pickup_datetime) on the full trips table. Rank them by cost and explain the ranking.

solution
EXPLAIN ANALYZE SELECT COUNT(*) FROM trips;
EXPLAIN ANALYZE SELECT COUNT(DISTINCT passenger_count) FROM trips;
EXPLAIN ANALYZE SELECT COUNT(DISTINCT pickup_datetime) FROM trips;

COUNT(*) is a scan and a counter (parallelizes well). COUNT(DISTINCT low-cardinality) adds cheap dedup over a handful of values. COUNT(DISTINCT high-cardinality) must track millions of distinct timestamps - sort or hash, memory pressure, possibly spilling to disk - and is the slow one. The general lesson: DISTINCT's cost scales with cardinality, which is why warehouses offer approximate distinct counts.

P8.7stackoverflowhard

Refactor for speed: "top 10 users by total answer score in 2018" can be written join-first or filter-first. Write both, EXPLAIN ANALYZE both, and report which wins and by how much.

solution
-- A: join everything, then filter and aggregate
EXPLAIN ANALYZE
SELECT u.display_name, SUM(p.score) AS total_score
FROM users u
JOIN posts p ON p.owner_user_id = u.id
WHERE p.post_type_id = 2
  AND p.creation_date >= '2018-01-01' AND p.creation_date < '2019-01-01'
GROUP BY u.id, u.display_name
ORDER BY total_score DESC LIMIT 10;

-- B: aggregate posts down to 10 user ids first, join last
EXPLAIN ANALYZE
WITH top_scorers AS (
    SELECT owner_user_id, SUM(score) AS total_score
    FROM posts
    WHERE post_type_id = 2
      AND creation_date >= '2018-01-01' AND creation_date < '2019-01-01'
    GROUP BY owner_user_id
    ORDER BY total_score DESC LIMIT 10
)
SELECT u.display_name, t.total_score
FROM top_scorers t
JOIN users u ON u.id = t.owner_user_id
ORDER BY t.total_score DESC;

B joins 10 rows to users instead of millions; on most hardware it wins clearly. The principle - shrink the data before the join - is the highest-leverage rewrite an analyst controls, and now you have your own measured numbers to cite when you say so.

P8.8anyinterview

Verbal drill (no SQL): a stakeholder says the daily report "suddenly got slow this week." Give your diagnostic plan out loud in under 90 seconds. Record yourself. Then compare against the model answer.

model answer

"First I would establish what changed: is the query slower, or is there more data, or is the server busier? I would run EXPLAIN ANALYZE and compare against the plan from when it was fast if we have it. Common culprits in rough order: the table grew past a threshold and the planner changed strategy on stale statistics - ANALYZE fixes that; an index was dropped or a query change made a predicate non-sargable; a join started fanning out because upstream data introduced duplicates - I would check row counts at each step; or concurrent load. I would fix the cheapest probable cause first, measure, and only then consider new indexes, since those carry write costs and need a DBA conversation." That covers method, causes, and judgment - the three things being graded.
