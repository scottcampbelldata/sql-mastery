-- module 8 / three sessions /

Performance

At senior level nobody expects you to be a DBA. They expect you to (a) read an EXPLAIN plan without panic, (b) know what an index does and when it helps, (c) not write queries that defeat indexes, and (d) talk about all three calmly. Your Stack Overflow database - a quarter-million posts, nearly a million votes - is big enough to make every one of these lessons visible on a stopwatch.

8.1 EXPLAIN and EXPLAIN ANALYZE

-- EXPLAIN shows the plan; EXPLAIN ANALYZE runs the query and shows reality:
EXPLAIN ANALYZE
SELECT * FROM posts WHERE score > 200 AND answercount = 0;

How to read the output, in the order that matters:

  • The node types. Seq Scan = read the whole table. Index Scan = walk the index to the matching rows. Index Only Scan = the index alone answered it. Bitmap Heap Scan = index found many scattered rows, fetched in batches.
  • Join strategies. Nested Loop (good for few rows with an index on the inner side), Hash Join (build a hash of one side; the default for big unsorted inputs), Merge Join (both sides sorted).
  • Estimated vs actual rows. rows=1000 estimated but actual rows=200000 means stale statistics led the planner astray - ANALYZE posts; refreshes them. Spotting a bad estimate is a genuinely impressive interview move.
  • Where the milliseconds went. The deepest expensive node is usually the story.
Interview script "My first move on a slow query is EXPLAIN ANALYZE. I look for sequential scans on large tables that a filter should have narrowed, then check whether an index exists and whether my predicate is written so the index can be used, then check row estimates against actuals." That paragraph, delivered calmly, passes most performance screens on its own.

8.2 Indexes in one page

-- A B-tree index is a sorted lookup structure on one or more columns:
CREATE INDEX idx_posts_creationdate ON posts (creationdate);
CREATE INDEX idx_posts_owner_type ON posts (owneruserid, posttypeid);
  • Helps: equality and range predicates on the leading column(s), JOIN keys, ORDER BY matching the index order.
  • Column order matters: an index on (a, b) serves WHERE a = ... and WHERE a = ... AND b = ..., but not WHERE b = ... alone. Rule of thumb: equality columns first, then the range column.
  • Costs: every index slows INSERT/UPDATE/DELETE and takes disk. Analysts reading from a warehouse rarely pay this, but say it to show you know writes exist.
  • Won't help: low-selectivity predicates (WHERE posttypeid = 2 matching well over half the rows - the planner rightly prefers a full scan), tiny tables, or predicates that wrap the column in a function.

8.3 Sargability: don't handcuff the index

A predicate is sargable ("search-argument-able") when the column stands alone on one side of the comparison. Wrap the column in a function and the index on it becomes useless.

-- NOT sargable: the function hides the column from the index
WHERE EXTRACT(YEAR FROM creationdate) = 2013
WHERE creationdate::date = '2013-01-15'
WHERE LOWER(displayname) = 'aaron bertrand'

-- Sargable rewrites: move the work to the constant side
WHERE creationdate >= '2013-01-01' AND creationdate < '2014-01-01'
WHERE creationdate >= '2013-01-15' AND creationdate < '2013-01-16'
-- (for the LOWER case: create an expression index ON users (LOWER(displayname)))

This is why every date filter in this curriculum used half-open ranges. It was performance training in disguise.

8.4 The talking points that round out senior answers

  • Filter early, join late: reduce rows in a CTE before joining, especially before joining two big tables like posts and votes.
  • SELECT only what you need: SELECT * on wide tables like posts (it carries a large body text column) drags dead columns through every node and can prevent index-only scans.
  • COUNT(DISTINCT x) is expensive on big data (requires dedup); approximate counts exist in warehouses (HLL) - name-dropping that shows range.
  • CTEs and the optimizer: since PostgreSQL 12, plain CTEs are inlined (no artificial fence) unless you write WITH x AS MATERIALIZED (...) to force materialization - useful when a CTE is referenced many times. Knowing this kills the outdated "CTEs are slow" myth interviewers sometimes probe with.
  • EXISTS vs IN vs JOIN: usually the same plan; choose for correctness and readability (Modules 3 and 5).
  • LIMIT is not free if there is an ORDER BY on an unindexed column - the whole set must be sorted before the top rows are known.

8.5 Partitioning: pruning whole tables, not rows

An index helps the planner find a few rows quickly inside a table. Partitioning helps it skip entire slices of the table it can prove no matching row lives in. They solve different problems, and the senior move is knowing which one a slow query actually needs.

Declarative partitioning splits one logical table into physical child tables by a key. Three shapes:

  • RANGE - by an ordered key, almost always a date. The natural fit for append-only time-series like posts.
  • LIST - by a discrete set, e.g. one partition per region, or per posttypeid.
  • HASH - spread rows evenly across N partitions when there is no natural range, to cap partition size and parallelize.
-- Range-partition posts by year (illustrative DDL - see the note below):
CREATE TABLE posts (/* ...same columns... */) PARTITION BY RANGE (creationdate);

CREATE TABLE posts_2015 PARTITION OF posts
    FOR VALUES FROM ('2015-01-01') TO ('2016-01-01');
CREATE TABLE posts_2016 PARTITION OF posts
    FOR VALUES FROM ('2016-01-01') TO ('2017-01-01');
-- ...one child per year. Each partition is indexed on its own (or an index on the
-- parent cascades to all of them).

When it helps vs. an index. A B-tree on creationdate still reads one big table and walks the index to your rows. A range-partitioned table lets the planner discard every year but the one you asked for before it scans anything - this is partition pruning:

EXPLAIN
SELECT COUNT(*) FROM posts
WHERE creationdate >= '2016-01-01' AND creationdate < '2017-01-01';
-- On the partitioned table the plan is an Append over posts_2016 ALONE;
-- posts_2015, posts_2017, ... never appear in it. Pruned partitions are never read.

Read the plan for an Append node listing only the partitions that survived pruning. The rest are gone at plan time - or shown as (never executed) when the cutoff is only known at run time (run-time pruning, e.g. the value comes from a parameter or a join).

So the division of labor:

  • Reach for a partition when a huge, append-only table is almost always queried by a range key (time), and when data lifecycle matters: archiving old data becomes DETACH PARTITION / DROP TABLE posts_2015 - instant - instead of a day-long DELETE that bloats the table.
  • Reach for an index when the predicate is selective within the data you keep - a specific user, a score threshold - which is most analyst queries.
  • They stack: partition to prune by time, then index each partition for the selective predicates inside it. Partitioning is not a general speed-up - on a small table, or a query that must touch every partition, it only adds planning overhead.
Interview script "Partitioning is for pruning and archival, not point lookups. If a table is enormous, append-only, and almost always filtered by a time range, I range-partition on that date so the planner skips whole partitions and dropping old data is instant - then I still index inside each partition for the selective predicates. If a query is just selective on some column, an index is the right tool and partitioning would only add overhead." Naming pruning and the drop-old-partition lifecycle win is what separates a real answer from a buzzword.
On this course's databases Your stackoverflow and chinook databases are read-only, and the DDL above rebuilds a table, so this section is read-and-reason rather than a live drill - the runner would reject CREATE TABLE. Picture it on posts: 243k rows spanning 2008-2024, partitioned by year, with the 2016 date-range query you have already run pruning straight to one child. That mental picture is exactly what the interview question tests.

Drills (all timed against your own hardware)

P8.1stackoverflowcore

Baseline read: run EXPLAIN ANALYZE on "the 25 highest-scoring posts with a score over 200." Identify the scan type, total runtime, and estimated vs actual rows. Write the three numbers down.

solution
EXPLAIN ANALYZE
SELECT id, title, score, creationdate
FROM posts
WHERE score > 200
ORDER BY score DESC
LIMIT 25;

Expect a Seq Scan (possibly parallel) - no index covers score out of the box - followed by a Sort feeding the Limit (probably reported as top-N heapsort). Only ~58 posts clear 200, so the filter is highly selective even though the scan reads everything. This is your "before" picture for P8.2.

P8.2stackoverflowinterview

Create an index on score, rerun P8.1's EXPLAIN ANALYZE, and compare. Then answer honestly: did it help, and why?

solution
CREATE INDEX idx_posts_score ON posts (score);
ANALYZE posts;
-- rerun the EXPLAIN ANALYZE from P8.1

score > 200 is highly selective (about 58 posts out of 243k), so the planner should switch to an Index Scan or Bitmap Heap Scan and the runtime should drop hard. If your planner still chooses Seq Scan, check that ANALYZE ran and consider how selective > 200 actually is in your data - reasoning about that is the drill. Clean up after: DROP INDEX idx_posts_score; if you want a pristine baseline later.

P8.3stackoverflowinterview

Sargability, on a stopwatch: with an index on creationdate in place, time EXTRACT(YEAR FROM creationdate) = 2013 against the half-open range version for 2013. Compare plans and runtimes.

solution
CREATE INDEX IF NOT EXISTS idx_posts_creationdate ON posts (creationdate);

EXPLAIN ANALYZE
SELECT COUNT(*) FROM posts
WHERE EXTRACT(YEAR FROM creationdate) = 2013;

EXPLAIN ANALYZE
SELECT COUNT(*) FROM posts
WHERE creationdate >= '2013-01-01'
  AND creationdate <  '2014-01-01';

Version 1: Seq Scan, every row's date passed through EXTRACT. Version 2: index range scan straight to the ~18k posts from 2013. Same answer, different orders of magnitude. This before/after is the single best performance story you can tell in an interview, because you have personally measured it.

P8.4stackoverflowinterview

Join strategy safari: EXPLAIN (no ANALYZE needed) a query for "users who earned a badge but never posted" - join users to badges, with a NOT EXISTS against posts. Which join strategies did the planner pick, and why do they make sense?

solution
EXPLAIN
SELECT DISTINCT u.id, u.displayname
FROM users u
JOIN badges b ON b.userid = u.id
WHERE NOT EXISTS (
    SELECT 1 FROM posts p WHERE p.owneruserid = u.id
);

Typical plan: a Hash Join between users and badges (both large, unsorted - hash the smaller side), and the NOT EXISTS appears as a Hash Anti Join against posts. The word "anti join" in the plan is the payoff: the planner recognizes NOT EXISTS as a first-class operation, which is exactly the argument for preferring it over NOT IN. About 184k of the 248k users never posted, so this set is large. Screenshot-level familiarity with these plan node names is what "can read a plan" means at interview time.

P8.5stackoverflowinterview

Composite index design: you frequently run "answers by user X, newest first" - WHERE owneruserid = ? AND posttypeid = 2 ORDER BY creationdate DESC. Design the best single index, create it, and prove with EXPLAIN that the sort disappears.

hint
Equality columns first (both of them), then the ORDER BY column. Three columns total.
solution
CREATE INDEX idx_posts_owner_type_date
    ON posts (owneruserid, posttypeid, creationdate DESC);

EXPLAIN ANALYZE
SELECT id, title, score, creationdate
FROM posts
WHERE owneruserid = 1876
  AND posttypeid = 2
ORDER BY creationdate DESC
LIMIT 20;

User 1876 is the most prolific answerer here (5,557 answers), so this is a real workload. The plan should show an Index Scan on the new index with no separate Sort node - the index already delivers rows in the right order, so LIMIT 20 stops after 20 rows. "Index serves the filter AND the sort" is the sentence to say.

P8.6stackoverflowhard

Aggregation cost: time COUNT(*), COUNT(DISTINCT posttypeid), and COUNT(DISTINCT owneruserid) on the full posts table. Rank them by cost and explain the ranking.

solution
EXPLAIN ANALYZE SELECT COUNT(*) FROM posts;
EXPLAIN ANALYZE SELECT COUNT(DISTINCT posttypeid) FROM posts;
EXPLAIN ANALYZE SELECT COUNT(DISTINCT owneruserid) FROM posts;

COUNT(*) is a scan and a counter (parallelizes well). COUNT(DISTINCT low-cardinality) adds cheap dedup over a handful of values - posttypeid has under ten. COUNT(DISTINCT high-cardinality) must track ~64k distinct user ids - sort or hash, memory pressure, possibly spilling to disk - and is the slow one. The general lesson: DISTINCT's cost scales with cardinality, which is why warehouses offer approximate distinct counts.

P8.7stackoverflowhard

Refactor for speed: "top 10 users by total answer score in 2013" can be written join-first or filter-first. Write both, EXPLAIN ANALYZE both, and report which wins and by how much.

solution
-- A: join everything, then filter and aggregate
EXPLAIN ANALYZE
SELECT u.displayname, SUM(p.score) AS total_score
FROM users u
JOIN posts p ON p.owneruserid = u.id
WHERE p.posttypeid = 2
  AND p.creationdate >= '2013-01-01' AND p.creationdate < '2014-01-01'
GROUP BY u.id, u.displayname
ORDER BY total_score DESC LIMIT 10;

-- B: aggregate posts down to 10 user ids first, join last
EXPLAIN ANALYZE
WITH top_scorers AS (
    SELECT owneruserid, SUM(score) AS total_score
    FROM posts
    WHERE posttypeid = 2
      AND creationdate >= '2013-01-01' AND creationdate < '2014-01-01'
    GROUP BY owneruserid
    ORDER BY total_score DESC LIMIT 10
)
SELECT u.displayname, t.total_score
FROM top_scorers t
JOIN users u ON u.id = t.owneruserid
ORDER BY t.total_score DESC;

B joins 10 rows to users instead of the whole 2013 answer set; on most hardware it wins clearly. The principle - shrink the data before the join - is the highest-leverage rewrite an analyst controls, and now you have your own measured numbers to cite when you say so.

P8.8anyinterview

Verbal drill (no SQL): a stakeholder says the daily report "suddenly got slow this week." Give your diagnostic plan out loud in under 90 seconds. Record yourself. Then compare against the model answer.

model answer

"First I would establish what changed: is the query slower, or is there more data, or is the server busier? I would run EXPLAIN ANALYZE and compare against the plan from when it was fast if we have it. Common culprits in rough order: the table grew past a threshold and the planner changed strategy on stale statistics - ANALYZE fixes that; an index was dropped or a query change made a predicate non-sargable; a join started fanning out because upstream data introduced duplicates - I would check row counts at each step; or concurrent load. I would fix the cheapest probable cause first, measure, and only then consider new indexes, since those carry write costs and need a DBA conversation." That covers method, causes, and judgment - the three things being graded.
