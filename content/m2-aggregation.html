Aggregation

GROUP BY collapses rows into groups; aggregate functions summarize each group. This module also contains the single most-asked conceptual interview question in existence: the logical order of query execution.

2.1 The logical order of execution (memorize this)

-- Written order:          Logical execution order:
-- SELECT                  1. FROM / JOIN     (build the working set)
-- FROM / JOIN             2. WHERE           (filter individual rows)
-- WHERE                   3. GROUP BY        (collapse into groups)
-- GROUP BY                4. HAVING          (filter whole groups)
-- HAVING                  5. SELECT          (compute output columns)
-- ORDER BY                6. DISTINCT        (dedup the output)
-- LIMIT                   7. ORDER BY        (sort the output)
--                         8. LIMIT / OFFSET  (trim the output)

Every "why can't I..." question is answered by this list. Why can't WHERE see a SELECT alias? WHERE runs first. Why can't WHERE filter on SUM()? Groups do not exist yet at WHERE time - that is HAVING's job. Why can ORDER BY use an alias? It runs after SELECT.

2.2 The aggregate functions and their NULL behavior

-- northwind: one row summarizing all orders
SELECT
    COUNT(*)                 AS all_rows,          -- counts rows, NULLs included
    COUNT(shipped_date)      AS shipped_rows,      -- skips NULLs
    COUNT(DISTINCT customer_id) AS customers,      -- unique non-NULL values
    SUM(freight)             AS total_freight,     -- ignores NULLs
    AVG(freight)             AS avg_freight,       -- ignores NULLs in the denominator too!
    MIN(order_date)          AS first_order,
    MAX(order_date)          AS last_order
FROM orders;
The AVG trap AVG(col) divides by the count of non-NULL values. If you want NULLs to count as zero, you must say so: AVG(COALESCE(col, 0)). "Which average does the business want?" is a genuine senior clarifying question.

2.3 GROUP BY and HAVING

-- chinook: revenue by billing country, big markets only
SELECT billing_country,
       COUNT(*)          AS invoice_count,
       SUM(total)        AS revenue,
       ROUND(AVG(total), 2) AS avg_invoice
FROM invoice
GROUP BY billing_country
HAVING SUM(total) > 100
ORDER BY revenue DESC;

Rule: every column in SELECT must either be inside an aggregate or listed in GROUP BY. Postgres enforces this; interviewers test it.

-- WHERE and HAVING together: filter rows first, then filter groups
-- northwind: among 1997 orders only, which employees handled more than 50?
SELECT employee_id, COUNT(*) AS orders_1997
FROM orders
WHERE order_date >= DATE '1997-01-01'
  AND order_date <  DATE '1998-01-01'
GROUP BY employee_id
HAVING COUNT(*) > 50
ORDER BY orders_1997 DESC;

2.4 Conditional aggregation with FILTER

Counting different things in one pass is a constant real-world need. Postgres has the cleanest syntax anywhere:

-- stackoverflow: questions vs answers per user, one scan of posts
SELECT owner_user_id,
       COUNT(*) FILTER (WHERE post_type_id = 1) AS questions,
       COUNT(*) FILTER (WHERE post_type_id = 2) AS answers
FROM posts
WHERE owner_user_id IS NOT NULL
GROUP BY owner_user_id
ORDER BY answers DESC
LIMIT 20;

-- Portable equivalent (know both - many interviews assume this form):
SUM(CASE WHEN post_type_id = 1 THEN 1 ELSE 0 END) AS questions

Practice - Session A (problems 1-3)

P2.1chinookwarmup

How many customers, how many distinct countries, and how many customers have no assigned support rep? One query, one row.

solution
SELECT COUNT(*)                        AS customers,
       COUNT(DISTINCT country)         AS countries,
       COUNT(*) - COUNT(support_rep_id) AS no_rep
FROM customer;

The COUNT(*) - COUNT(col) idiom for NULL counting comes up in nearly every data-quality discussion.

P2.2northwindwarmup

Average, minimum, and maximum freight per ship_country, only for countries with at least 10 orders, highest average first. Round the average to 2 decimals.

solution
SELECT ship_country,
       COUNT(*)              AS orders,
       ROUND(AVG(freight), 2) AS avg_freight,
       MIN(freight)          AS min_freight,
       MAX(freight)          AS max_freight
FROM orders
GROUP BY ship_country
HAVING COUNT(*) >= 10
ORDER BY avg_freight DESC;
P2.3nyctaxicore

Per payment_type: trip count, average fare, average tip, and average tip percentage of fare (fare > 0 only, to avoid division problems). Round money to 2 decimals.

hint
The average of per-trip percentages and the ratio of averages are different numbers. Compute the per-trip percentage inside AVG. Filter fare_amount > 0 in WHERE.
solution
SELECT payment_type,
       COUNT(*)                                  AS trips,
       ROUND(AVG(fare_amount)::numeric, 2)      AS avg_fare,
       ROUND(AVG(tip_amount)::numeric, 2)       AS avg_tip,
       ROUND(AVG(tip_amount / fare_amount)::numeric * 100, 1) AS avg_tip_pct
FROM trips
WHERE fare_amount > 0
GROUP BY payment_type
ORDER BY trips DESC;

Interview follow-up you should expect: "why did you filter fare > 0?" Answer: to prevent division by zero and to exclude voided/negative trips that would distort the ratio - and I would confirm with the business whether they should be excluded from the counts too.

Practice - Session B (problems 4-6)

P2.4northwindcore

Using order_details only: total revenue per product (revenue = unit_price * quantity * (1 - discount)). Show product_id, number of order lines, total quantity, and total revenue rounded to 2 decimals. Products with revenue over 10,000 only, richest first.

solution
SELECT product_id,
       COUNT(*)      AS line_count,
       SUM(quantity) AS total_qty,
       ROUND(SUM(unit_price * quantity * (1 - discount))::numeric, 2) AS revenue
FROM order_details
GROUP BY product_id
HAVING SUM(unit_price * quantity * (1 - discount)) > 10000
ORDER BY revenue DESC;

Note HAVING repeats the expression rather than using the alias - HAVING runs before SELECT, so the alias does not exist yet. (Postgres actually permits GROUP BY aliases as a convenience, but not HAVING aliases.)

P2.5stackoverflowcore

Which badge names have been awarded more than 1,000 times, and to how many distinct users? Most-awarded first.

solution
SELECT name,
       COUNT(*)               AS times_awarded,
       COUNT(DISTINCT user_id) AS distinct_users
FROM badges
GROUP BY name
HAVING COUNT(*) > 1000
ORDER BY times_awarded DESC;

If times_awarded is much larger than distinct_users for a badge, it is a repeatable badge - the kind of observation a senior volunteers while the query runs.

P2.6chinookcore

Per genre_id in track: track count, average length in minutes (1 decimal), and total catalog value (SUM of unit_price). Only genres with 50+ tracks. Why might genre_id be NULL for some tracks, and what happens to those rows here?

solution
SELECT genre_id,
       COUNT(*)                                AS tracks,
       ROUND(AVG(milliseconds) / 60000.0, 1)  AS avg_minutes,
       SUM(unit_price)                         AS catalog_value
FROM track
GROUP BY genre_id
HAVING COUNT(*) >= 50
ORDER BY tracks DESC;

GROUP BY treats all NULLs as one group, so unclassified tracks appear as a single genre_id = NULL row (if 50+ of them exist). That NULLs group together in GROUP BY - but never equal each other in joins - is a classic conceptual question.

Practice - Session C (problems 7-9)

P2.7stackoverflowinterview

One row per year of posts.creation_date: total posts, questions, answers, and the answer-to-question ratio (2 decimals). Use FILTER. Chronological order.

hint
Group by EXTRACT(YEAR FROM creation_date) or date_trunc('year', creation_date). Guard the ratio against zero questions with NULLIF.
solution
SELECT EXTRACT(YEAR FROM creation_date)::int AS yr,
       COUNT(*) AS posts,
       COUNT(*) FILTER (WHERE post_type_id = 1) AS questions,
       COUNT(*) FILTER (WHERE post_type_id = 2) AS answers,
       ROUND(
         COUNT(*) FILTER (WHERE post_type_id = 2)::numeric
         / NULLIF(COUNT(*) FILTER (WHERE post_type_id = 1), 0), 2) AS ans_per_q
FROM posts
GROUP BY 1
ORDER BY yr;

Three interview staples in one: FILTER, NULLIF-guarded division, and the ::numeric cast to avoid integer division. GROUP BY 1 (positional) is fine to use, just be able to name the tradeoff: shorter but fragile if the SELECT list is reordered.

P2.8adventureworksinterview

From sales.salesorderheader: per territoryid, count orders and split total revenue (totaldue) into online (onlineorderflag true) vs offline using conditional aggregation. Sort by total revenue descending.

solution
SELECT territoryid,
       COUNT(*) AS orders,
       SUM(totaldue)                                        AS revenue,
       SUM(totaldue) FILTER (WHERE onlineorderflag)         AS online_rev,
       SUM(totaldue) FILTER (WHERE NOT onlineorderflag)     AS offline_rev
FROM sales.salesorderheader
GROUP BY territoryid
ORDER BY revenue DESC;

If your port stores the flag as 0/1 instead of boolean, the FILTER becomes WHERE onlineorderflag = 1. Checking the type with \d before writing is exactly the Module 0 reflex.

P2.9northwindinterview

The classic two-level aggregation: what is the average number of orders per customer? (Not average order value - average order count.) Solve it with a subquery in FROM; you will redo it with a CTE in Module 5.

hint
Step 1: orders per customer (GROUP BY customer_id). Step 2: AVG over those counts.
solution
SELECT ROUND(AVG(order_count), 2) AS avg_orders_per_customer
FROM (
    SELECT customer_id, COUNT(*) AS order_count
    FROM orders
    GROUP BY customer_id
) per_customer;

Clarifying question a senior asks first: "customers with zero orders - do they count in the average?" This version excludes them (they have no rows in orders). Including them requires starting from the customers table with a LEFT JOIN - Module 3, P3.9.

Practice - Session D (problems 10-12)

P2.10nyctaxiinterview

Hourly demand curve: for one week of data (pick any 7-day range), count trips per hour of day (0-23) and show average fare per hour. Which three hours have the highest average fare?

solution
SELECT EXTRACT(HOUR FROM pickup_datetime)::int AS hr,
       COUNT(*) AS trips,
       ROUND(AVG(fare_amount)::numeric, 2) AS avg_fare
FROM trips
WHERE pickup_datetime >= '2016-01-04'
  AND pickup_datetime <  '2016-01-11'
GROUP BY hr
ORDER BY avg_fare DESC
LIMIT 3;

Adjust the date range to whatever your dataset covers (find it with SELECT MIN(pickup_datetime), MAX(pickup_datetime) FROM trips - which is itself a Module 0 move). Rerun without LIMIT and ORDER BY hr to see the full curve.

P2.11chinookinterview

Debug drill. A teammate wrote this and got an error. Fix it two different ways and explain the difference in results, if any:

SELECT billing_country, billing_city, SUM(total) AS revenue
FROM invoice
WHERE SUM(total) > 50
GROUP BY billing_country;
solution

Two bugs: aggregates are illegal in WHERE (groups do not exist yet), and billing_city is selected without being grouped or aggregated.

-- Fix 1: group by both columns (city-level revenue)
SELECT billing_country, billing_city, SUM(total) AS revenue
FROM invoice
GROUP BY billing_country, billing_city
HAVING SUM(total) > 50;

-- Fix 2: drop the city (country-level revenue)
SELECT billing_country, SUM(total) AS revenue
FROM invoice
GROUP BY billing_country
HAVING SUM(total) > 50;

They answer different questions: Fix 1's grain is (country, city), Fix 2's grain is country. "What grain do you want?" is the clarifying question that resolves it. Debug-the-query rounds like this are common in senior screens.

P2.12stackoverflowhard

Distribution question: bucket users by reputation - under 100, 100-999, 1000-9999, 10000+ - and show the user count and percentage of all users in each bucket (1 decimal). One query.

hint
CASE builds the bucket (Module 4 previews it). For the percentage inside a grouped query, divide COUNT(*) by SUM(COUNT(*)) OVER () - or keep it simple with a subquery for the total.
solution
SELECT CASE
         WHEN reputation < 100   THEN 'a. under 100'
         WHEN reputation < 1000  THEN 'b. 100-999'
         WHEN reputation < 10000 THEN 'c. 1000-9999'
         ELSE 'd. 10000+'
       END AS bucket,
       COUNT(*) AS users,
       ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER (), 1) AS pct
FROM users
GROUP BY 1
ORDER BY bucket;

The letter prefixes make the buckets sort correctly as text - a small practical trick. SUM(COUNT(*)) OVER () is a window over the grouped result; if that feels like magic now, it will be obvious after Module 6. The subquery alternative: ROUND(100.0 * COUNT(*) / (SELECT COUNT(*) FROM users), 1).
