Subqueries and CTEs

Multi-step logic is where senior questions live. CTEs are how you write multi-step logic that another human - the interviewer - can read. From this module on, WITH is your default structure for anything with more than one step. You will also meet the rest of the subquery family - quantified IN / ANY / ALL comparisons, LATERAL joins for top-N-per-group, and recursive CTEs that both walk a hierarchy and generate a gap-free series. The clean chinook tables let you eyeball each result; the big stackoverflow dump is where correlated subqueries and recursion earn their keep.

5.1 The four places a subquery can live

-- 1. Scalar (returns one value) - in SELECT or WHERE (chinook):
SELECT name, unit_price,
       unit_price - (SELECT AVG(unit_price) FROM track) AS vs_avg
FROM track;

-- 2. Column/list (returns one column) - with IN (chinook):
SELECT * FROM invoice
WHERE customer_id IN (SELECT customer_id FROM customer WHERE country = 'Brazil');

-- 3. Table / derived table (returns rows) - in FROM, must be aliased (chinook):
SELECT AVG(invoice_count) FROM (
    SELECT customer_id, COUNT(*) AS invoice_count FROM invoice GROUP BY customer_id
) t;

-- 4. Correlated (references the outer row; runs logically per row) - usually with EXISTS (chinook):
SELECT c.first_name, c.last_name FROM customer c
WHERE EXISTS (SELECT 1 FROM invoice i WHERE i.customer_id = c.customer_id
                                        AND i.total > 20);
IN vs EXISTS, the interview answer Semantically: EXISTS stops at the first match and is NULL-safe; IN builds the full list and NOT IN has the NULL footgun (Module 3). Practically: the Postgres planner usually rewrites both into the same semi-join, so the real argument is correctness and readability, not speed. Saying exactly that - "same plan, different failure modes" - is a strong answer.

5.2 Correlated subqueries: one comparison per row

A correlated subquery mentions a column from the outer query, so it is re-evaluated for each outer row. The canonical shape is "rows that beat the average of their own group." On the big stackoverflow tables it reads beautifully - "posts scoring above their author's own average" - and the correlation key (owneruserid) is indexed, so it stays fast when you scope the outer set.

-- stackoverflow: questions that scored above their author's own average post score.
-- Scoped to the postgresql tag so the correlated scan stays quick.
SELECT p.id, p.owneruserid, p.score, p.title
FROM posts p
WHERE p.posttypeid = 1
  AND p.tags LIKE '%|postgresql|%'
  AND p.score > (SELECT AVG(p2.score) FROM posts p2 WHERE p2.owneruserid = p.owneruserid)
ORDER BY p.score DESC
LIMIT 10;

Read the inner query as "the average score of this author's posts." Because p.owneruserid comes from the outer row, the average changes as the outer row changes - that is the whole idea of a correlated subquery. The same pattern answers "customers above their own city's average", "products above their category's average", and half of the analytics questions you will ever be asked.

Correlated vs derived table A correlated subquery is conceptually per-row; a derived table (or CTE) computes each group's average once and joins it back. Same answer, and the derived-table form is usually the faster and more scalable one - which is exactly the "compute the aggregate once, join it" refactor you saw in Modules 2 and 4. Know both, and say why you picked one.

5.3 CTEs: named steps

-- chinook: revenue per invoice, computed once and named, then joined back.
WITH invoice_revenue AS (
    SELECT invoice_id,
           SUM(unit_price * quantity) AS revenue
    FROM invoice_line
    GROUP BY invoice_id
)
SELECT i.invoice_id, i.total, r.revenue
FROM invoice i
JOIN invoice_revenue r ON r.invoice_id = i.invoice_id;

(Line revenue is unit_price * quantity - chinook has no discount column, so the math stays honest.)

Chaining is the real power: each CTE can read the previous ones, so a hard query becomes a stack of named steps - roughly WITH step1 AS (...), step2 AS (SELECT ... FROM step1), step3 AS (SELECT ... FROM step2 JOIN step1) SELECT * FROM step3. Here is that shape made concrete on chinook - revenue per invoice, rolled up to revenue per customer, then filtered to the above-average customers:

-- Three chained CTEs: invoice revenue -> customer revenue -> above-average customers.
WITH invoice_rev AS (
    SELECT invoice_id, SUM(unit_price * quantity) AS revenue
    FROM invoice_line
    GROUP BY invoice_id
),
customer_rev AS (
    SELECT i.customer_id, SUM(r.revenue) AS revenue
    FROM invoice i
    JOIN invoice_rev r ON r.invoice_id = i.invoice_id
    GROUP BY i.customer_id
)
SELECT c.first_name || ' ' || c.last_name AS customer,
       ROUND(cr.revenue::numeric, 2) AS revenue
FROM customer_rev cr
JOIN customer c ON c.customer_id = cr.customer_id
WHERE cr.revenue > (SELECT AVG(revenue) FROM customer_rev)
ORDER BY cr.revenue DESC;
Interview technique Before writing a hard query, say the CTE plan in English: "First I will get revenue per invoice, then join it to invoices, then aggregate per customer." Then name your CTEs exactly those steps. It shows planning, and if you run out of time, the plan itself earns partial credit.

5.4 Recursive CTEs: walking hierarchies

-- chinook: the whole staff tree under the General Manager, Andrew Adams.
WITH RECURSIVE org AS (
    -- anchor: the starting row(s)
    SELECT employee_id, first_name, last_name, reports_to, 1 AS depth
    FROM employee
    WHERE reports_to IS NULL

    UNION ALL

    -- recursive step: everyone reporting to someone already in org
    SELECT e.employee_id, e.first_name, e.last_name, e.reports_to, org.depth + 1
    FROM employee e
    JOIN org ON e.reports_to = org.employee_id
)
SELECT * FROM org ORDER BY depth, employee_id;

The mental model: the anchor seeds a working set (here, Andrew Adams, whose reports_to is NULL); the recursive step repeatedly joins new rows against what was added last round (Nancy Edwards and Michael Mitchell at depth 2, then their agents and staff at depth 3); it stops when a round adds nothing. Chinook's employee table is a clean three-level tree, which makes it the perfect first recursive CTE. Senior interviews rarely demand you write one cold, but "how would you handle an org chart / category tree / bill of materials?" expects you to say "recursive CTE" and sketch the anchor + step.

5.5 IN, ANY, and ALL: quantified comparison

IN is the friendly face of a whole family. x IN (subquery) is exactly x = ANY (subquery), and once you see that, ANY and ALL open up any operator, not just equality. > ANY means "bigger than at least one of them" (so, bigger than the smallest); > ALL means "bigger than every one of them" (so, bigger than the largest). Rock And Roll in chinook is a tight little genre - 12 tracks, none longer than about 163 seconds - which makes the two read cleanly side by side.

-- chinook: > ANY = longer than the SHORTEST rock-and-roll track.
SELECT name, milliseconds FROM track
WHERE milliseconds > ANY (
    SELECT t.milliseconds FROM track t
    JOIN genre g ON g.genre_id = t.genre_id
    WHERE g.name = 'Rock And Roll'
)
ORDER BY milliseconds
LIMIT 3;

-- chinook: > ALL = longer than the LONGEST rock-and-roll track.
SELECT name, milliseconds FROM track
WHERE milliseconds > ALL (
    SELECT t.milliseconds FROM track t
    JOIN genre g ON g.genre_id = t.genre_id
    WHERE g.name = 'Rock And Roll'
)
ORDER BY milliseconds
LIMIT 3;

The > ANY version keeps everything above 106 seconds (3432 tracks); the > ALL version keeps only what beats 163 seconds (3177 tracks). Why it matters: > ALL (subquery) is the readable way to say "beats the maximum" without a separate MAX() subquery, and it is the honest tool when the interviewer's list has NULLs - NOT IN breaks on a NULL in the list (Module 3), but the planner and NULL rules for <> ALL are the ones you actually reason about.

The one gotcha <> ANY is almost never what people mean - it is true whenever the value differs from any single list member, so it is true for almost everything. "Not equal to any of them" is NOT IN, which is <> ALL. Mixing these up is a classic self-inflicted bug.

5.6 LATERAL: a subquery that sees the current row

A derived table in FROM normally cannot reference the other tables in the same FROM - it is computed on its own. LATERAL lifts that restriction: the subquery may mention columns from tables listed to its left, so it runs once per outer row. That makes it the clean answer to top-N-per-group - "the single best-selling album for each artist" - which a plain GROUP BY cannot express, because you want a whole row per group, not one aggregate.

-- chinook: each genre's two longest tracks, via a per-row LATERAL top-N.
SELECT g.name AS genre, t.name AS track, t.milliseconds
FROM genre g
CROSS JOIN LATERAL (
    SELECT name, milliseconds FROM track
    WHERE genre_id = g.genre_id
    ORDER BY milliseconds DESC
    LIMIT 2
) t
ORDER BY g.name, t.milliseconds DESC;

Read g.genre_id inside the subquery: that is the correlation, and it is only legal because of LATERAL. Why it matters: this is the per-group top-N pattern that comes up constantly - top 3 orders per customer, latest login per user - and interviewers love it because the naive answers (a correlated LIMIT in SELECT, or a self-join with a counting subquery) are clumsy by comparison. Module 6 shows the window-function version with ROW_NUMBER(); knowing both, and when LATERAL wins (a genuinely different subquery per row, or calling a set-returning function), is the senior signal.

CROSS vs LEFT JOIN LATERAL CROSS JOIN LATERAL drops outer rows whose subquery returns nothing - a genre with no tracks would vanish. Use LEFT JOIN LATERAL ... ON true to keep every outer row and get NULLs where the subquery was empty. Same choice you make with any join, just applied to the per-row subquery.

5.7 Recursive CTEs part two: generating series

Recursion is not only for walking existing rows - it can also manufacture rows that do not exist yet. The anchor is the first value; the recursive step produces the next one and stops at a bound. The workhorse use is a gap-free calendar: a report that must show every month, including the ones with zero sales, needs a row source for the empty months, and a recursive series is the portable way to build one.

-- chinook: every month of 2023 with its revenue, zero-filled where there were no sales.
WITH RECURSIVE months AS (
    SELECT DATE '2023-01-01' AS month_start
    UNION ALL
    SELECT (month_start + INTERVAL '1 month')::date
    FROM months
    WHERE month_start < DATE '2023-12-01'
)
SELECT m.month_start,
       COALESCE(ROUND(SUM(i.total)::numeric, 2), 0) AS revenue
FROM months m
LEFT JOIN invoice i ON date_trunc('month', i.invoice_date)::date = m.month_start
GROUP BY m.month_start
ORDER BY m.month_start;

The LEFT JOIN is doing the real work: the series guarantees all 12 rows exist, and the join fills revenue where it can and leaves NULL (then zero, via COALESCE) where it cannot. Why it matters: "make sure empty periods still show up" is a requirement in almost every real dashboard, and the wrong instinct - grouping the invoices alone - silently drops the quiet months. Postgres also ships generate_series, which does this in one call; the recursive form is worth knowing because it is standard SQL and because the same shape (anchor, step, stop) generates any sequence, not just dates.

Practice - Session A (problems 1-3)

P5.1chinookwarmup

Tracks priced above the average track price: name, unit_price, and how far above average (2 decimals). Scalar subquery - and note how many times you had to write it.

solution
SELECT name, unit_price,
       ROUND(unit_price - (SELECT AVG(unit_price) FROM track), 2) AS above_avg
FROM track
WHERE unit_price > (SELECT AVG(unit_price) FROM track);

Twice. The CTE version writes it once - WITH avg_price AS (SELECT AVG(unit_price) AS ap FROM track) then CROSS JOIN it - which is why CTEs win as logic grows.

P5.2chinookcore

Customers who have ever bought a Heavy Metal track. Once with IN, once with EXISTS. Full names only, alphabetized.

solution
-- IN
SELECT first_name || ' ' || last_name AS customer FROM customer
WHERE customer_id IN (
    SELECT i.customer_id
    FROM invoice i
    JOIN invoice_line il ON il.invoice_id = i.invoice_id
    JOIN track t ON t.track_id = il.track_id
    JOIN genre g ON g.genre_id = t.genre_id
    WHERE g.name = 'Heavy Metal'
)
ORDER BY customer;

-- EXISTS
SELECT c.first_name || ' ' || c.last_name AS customer FROM customer c
WHERE EXISTS (
    SELECT 1
    FROM invoice i
    JOIN invoice_line il ON il.invoice_id = i.invoice_id
    JOIN track t ON t.track_id = il.track_id
    JOIN genre g ON g.genre_id = t.genre_id
    WHERE i.customer_id = c.customer_id
      AND g.name = 'Heavy Metal'
)
ORDER BY customer;

A plain JOIN would also work but needs DISTINCT to kill duplicates from customers who bought several metal tracks - the semi-join forms never duplicate, which is exactly why they exist.

P5.3chinookcore

Refactor with a CTE: label each invoice as is_large (total > 10) once in a CTE, then aggregate count, average total, and average line count for large vs small invoices from it. Compare readability with writing the total > 10 test three times.

solution
WITH labeled AS (
    SELECT i.invoice_id,
           i.total,
           i.total > 10 AS is_large,
           COUNT(il.invoice_line_id) AS line_count
    FROM invoice i
    JOIN invoice_line il ON il.invoice_id = i.invoice_id
    GROUP BY i.invoice_id, i.total
)
SELECT is_large,
       COUNT(*) AS invoices,
       ROUND(AVG(total)::numeric, 2) AS avg_total,
       ROUND(AVG(line_count)::numeric, 1) AS avg_lines
FROM labeled
GROUP BY is_large;

Same answer, and the is_large label is written once instead of repeated in every aggregate. "Compute the label once, aggregate on it" is one of the highest-value refactors you can show in a live interview.

Practice - Session B (problems 4-6)

P5.4chinookcore

Above-average customers: customers whose lifetime invoice total exceeds the average lifetime total across all customers. Two CTEs (per-customer totals, then the average), then the final filter. Show name and total.

solution
WITH customer_totals AS (
    SELECT c.customer_id,
           c.first_name || ' ' || c.last_name AS customer,
           SUM(i.total) AS lifetime_total
    FROM customer c
    JOIN invoice i ON i.customer_id = c.customer_id
    GROUP BY c.customer_id, customer
),
avg_total AS (
    SELECT AVG(lifetime_total) AS avg_lt FROM customer_totals
)
SELECT ct.customer, ROUND(ct.lifetime_total::numeric, 2) AS lifetime_total
FROM customer_totals ct
CROSS JOIN avg_total a
WHERE ct.lifetime_total > a.avg_lt
ORDER BY ct.lifetime_total DESC;

CROSS JOIN against a one-row CTE is the clean way to make a scalar available everywhere - no repeated subquery.

P5.5stackoverflowinterview

Questions that got an answer within one hour of posting: count them, and report what percentage of all answered questions that is. Structure it as CTEs.

hint
CTE 1: for each question, the earliest answer time (answers have parentid = question id, and posttypeid = 2). CTE 2 or final: compare to the question's creationdate + INTERVAL '1 hour'.
solution
WITH first_answers AS (
    SELECT a.parentid AS question_id,
           MIN(a.creationdate) AS first_answer_at
    FROM posts a
    WHERE a.posttypeid = 2
    GROUP BY a.parentid
),
judged AS (
    SELECT q.id,
           fa.first_answer_at <= q.creationdate + INTERVAL '1 hour' AS within_hour
    FROM posts q
    JOIN first_answers fa ON fa.question_id = q.id
    WHERE q.posttypeid = 1
)
SELECT COUNT(*) FILTER (WHERE within_hour) AS answered_within_hour,
       COUNT(*) AS answered_total,
       ROUND(100.0 * COUNT(*) FILTER (WHERE within_hour) / COUNT(*), 1) AS pct
FROM judged;

Notice the denominator choice: answered questions, per the prompt. If the interviewer meant all questions, the JOIN becomes a LEFT JOIN and unanswered questions count as not-within-hour. Asking which denominator they want is the senior move.

P5.6chinookinterview

Three-CTE chain: (1) revenue per track from invoice_line, (2) revenue per album by joining up through track, (3) final: each album with its artist name, revenue, and share of its artist's total revenue (1 decimal).

solution
WITH track_rev AS (
    SELECT track_id, SUM(unit_price * quantity) AS revenue
    FROM invoice_line
    GROUP BY track_id
),
album_rev AS (
    SELECT al.album_id,
           al.title AS album,
           al.artist_id,
           SUM(tr.revenue) AS revenue
    FROM track_rev tr
    JOIN track t ON t.track_id = tr.track_id
    JOIN album al ON al.album_id = t.album_id
    GROUP BY al.album_id, al.title, al.artist_id
),
artist_rev AS (
    SELECT artist_id, SUM(revenue) AS artist_revenue
    FROM album_rev
    GROUP BY artist_id
)
SELECT ar.name AS artist,
       a.album,
       ROUND(a.revenue::numeric, 2) AS revenue,
       ROUND(100.0 * a.revenue / c.artist_revenue, 1) AS pct_of_artist
FROM album_rev a
JOIN artist_rev c ON c.artist_id = a.artist_id
JOIN artist ar ON ar.artist_id = a.artist_id
ORDER BY c.artist_revenue DESC, revenue DESC;

Each CTE is independently testable - run SELECT * FROM album_rev alone while debugging. Mentioning that habit in an interview lands well. (Module 6 replaces the artist_rev CTE with one window function.)

Practice - Session C (problems 7-8)

P5.7chinookinterview

Recursive CTE: print the full org chart - every employee with their depth from the top (people with reports_to IS NULL are depth 1) and a breadcrumb path like 'Adams > Edwards > Peacock'.

hint
Anchor: the roots (reports_to IS NULL). Carry the path along: parent.path || ' > ' || child.last_name.
solution
WITH RECURSIVE org AS (
    SELECT employee_id, last_name, reports_to,
           1 AS depth,
           last_name::text AS path
    FROM employee
    WHERE reports_to IS NULL

    UNION ALL

    SELECT e.employee_id, e.last_name, e.reports_to,
           org.depth + 1,
           org.path || ' > ' || e.last_name
    FROM employee e
    JOIN org ON e.reports_to = org.employee_id
)
SELECT depth, path FROM org ORDER BY path;

The ::text cast on the anchor matters: both halves of a UNION must agree on types, and last_name may be a bounded varchar, which would truncate the growing path. Chinook's tree is only three levels deep - Adams at the top, two managers below, their staff below that - so you can read the whole result and confirm the recursion by eye.

P5.8chinookinterview

Basket analysis lite: which pairs of genres appear together on the same invoice? Count co-occurrences, top 10 pairs. (CTE for invoice-genre pairs, then a self-join with an inequality to avoid double counting.)

hint
CTE: DISTINCT (invoice_id, genre name) via invoice_line -> track -> genre. Then join the CTE to itself on the same invoice with a.genre < b.genre.
solution
WITH invoice_genres AS (
    SELECT DISTINCT il.invoice_id, g.name AS genre
    FROM invoice_line il
    JOIN track t ON t.track_id = il.track_id
    JOIN genre g ON g.genre_id = t.genre_id
)
SELECT a.genre AS genre_1,
       b.genre AS genre_2,
       COUNT(*) AS invoices_together
FROM invoice_genres a
JOIN invoice_genres b
  ON b.invoice_id = a.invoice_id
 AND a.genre < b.genre
GROUP BY a.genre, b.genre
ORDER BY invoices_together DESC
LIMIT 10;

a.genre < b.genre does two jobs: no self-pairs and no mirrored duplicates (Rock+Jazz vs Jazz+Rock). This inequality self-join is a legitimate senior interview question in its own right.

Practice - Session D (problems 9-10)

P5.9stackoverflowhard

Power-user overlap: what fraction of the top 100 users by reputation are also in the top 100 by badge count? CTEs for each top-100 list, then count the intersection.

solution
WITH top_rep AS (
    SELECT id FROM users ORDER BY reputation DESC LIMIT 100
),
top_badges AS (
    SELECT userid FROM badges
    GROUP BY userid
    ORDER BY COUNT(*) DESC
    LIMIT 100
)
SELECT COUNT(*) AS in_both,
       ROUND(100.0 * COUNT(*) / 100, 1) AS pct_overlap
FROM top_rep r
JOIN top_badges b ON b.userid = r.id;

Worth narrating: LIMIT with ties is arbitrary at the boundary (two users tied at rank 100 - which one made the list?). Module 6's RANK/DENSE_RANK is the principled fix, and mentioning that here is a perfect segue an interviewer will reward. Note the join key is badges.userid = users.id - the user primary key is id, not userid.

P5.10stackoverflowhard

The "second query from the answer" drill: using a CTE chain, find each postgresql-tag author's first ever post and whether it was a question or an answer (posttypeid). Return display name, the first post's date, its type, and its score, highest score first. What if a user made two posts on the same first day?

hint
CTE 1: the set of authors who ever used the postgresql tag. CTE 2: each such author's MIN(creationdate). CTE 3: join back to posts to pull that first post's row(s). Then join to users for the name.
solution
WITH pg_authors AS (
    SELECT DISTINCT owneruserid
    FROM posts
    WHERE tags LIKE '%|postgresql|%' AND owneruserid IS NOT NULL
),
first_post AS (
    SELECT p.owneruserid, MIN(p.creationdate) AS first_date
    FROM posts p
    JOIN pg_authors a ON a.owneruserid = p.owneruserid
    GROUP BY p.owneruserid
),
first_rows AS (
    SELECT p.owneruserid, p.creationdate, p.posttypeid, p.score
    FROM posts p
    JOIN first_post f
      ON f.owneruserid = p.owneruserid
     AND f.first_date  = p.creationdate
)
SELECT u.displayname,
       fr.creationdate::date AS first_post_date,
       fr.posttypeid,
       fr.score
FROM first_rows fr
JOIN users u ON u.id = fr.owneruserid
ORDER BY fr.score DESC
LIMIT 20;

Edge case to name out loud: a user whose two earliest posts share the exact same timestamp appears twice; whether that is right depends on the business question. The cleaner tool for "first row per group" is ROW_NUMBER() - the very first thing Module 6 teaches. This problem exists so you feel the before and after.

Practice - Session E (problems 11-16)

P5.11chinookwarmup

Derived table in FROM: what is the average number of tracks per album? Count tracks per album in a subquery, then average those counts. (One decimal.)

solution
SELECT ROUND(AVG(track_count)::numeric, 1) AS avg_tracks_per_album
FROM (
    SELECT album_id, COUNT(*) AS track_count
    FROM track
    WHERE album_id IS NOT NULL
    GROUP BY album_id
) per_album;

You cannot write AVG(COUNT(*)) directly - that is two levels of aggregation, and the derived table is exactly how you stage one level before the other. The FROM subquery must be aliased (per_album); forgetting the alias is the most common beginner error with derived tables.

P5.12chinookcore

Using > ALL: which Jazz tracks are longer than every Rock And Roll track? Return name and length, shortest qualifying first, top 10.

hint
The inner query is just the set of Rock And Roll track lengths. t.milliseconds > ALL (that set) means "longer than the longest of them" - no MAX() needed.
solution
SELECT t.name, t.milliseconds
FROM track t
JOIN genre g ON g.genre_id = t.genre_id
WHERE g.name = 'Jazz'
  AND t.milliseconds > ALL (
      SELECT t2.milliseconds FROM track t2
      JOIN genre g2 ON g2.genre_id = t2.genre_id
      WHERE g2.name = 'Rock And Roll'
  )
ORDER BY t.milliseconds
LIMIT 10;

Equivalent to t.milliseconds > (SELECT MAX(...) ...), and reads closer to the spoken requirement. If the Rock And Roll set could be empty, note the difference out loud: > ALL (empty set) is true, whereas > (SELECT MAX ...) over no rows returns NULL and the row drops.

P5.13stackoverflowinterview

Correlated subquery: find each python-tag author's own highest-scoring question, i.e. questions whose score is at least as high as every other question that author wrote. Return id, owneruserid, score, title; highest score first, top 10.

hint
Keep a question only if p.score >= ALL the scores of that same author's questions. The correlation is p2.owneruserid = p.owneruserid in the inner query.
solution
SELECT p.id, p.owneruserid, p.score, p.title
FROM posts p
WHERE p.posttypeid = 1
  AND p.tags LIKE '%|python|%'
  AND p.score >= ALL (
      SELECT p2.score FROM posts p2
      WHERE p2.owneruserid = p.owneruserid AND p2.posttypeid = 1
  )
ORDER BY p.score DESC
LIMIT 10;

This fuses the two ideas of the module: a correlated inner query (it reads p.owneruserid from the outer row) feeding a quantified >= ALL. If an author has two questions tied at their personal best, both appear - the same "ties at the top" caveat you will retire with ROW_NUMBER() in Module 6.

P5.14chinookhard

LATERAL top-N: for each artist, their single highest-revenue album (revenue = sum of unit_price * quantity across the album's sold lines). Return artist, album, revenue; top 10 artists by that album's revenue.

hint
Drive from artist, then CROSS JOIN LATERAL a subquery that sums revenue per album for that artist, orders by revenue, and takes LIMIT 1. The subquery references ar.artist_id.
solution
SELECT ar.name AS artist, top.title AS album, top.revenue
FROM artist ar
CROSS JOIN LATERAL (
    SELECT al.title, SUM(il.unit_price * il.quantity) AS revenue
    FROM album al
    JOIN track t ON t.album_id = al.album_id
    JOIN invoice_line il ON il.track_id = t.track_id
    WHERE al.artist_id = ar.artist_id
    GROUP BY al.album_id, al.title
    ORDER BY revenue DESC
    LIMIT 1
) top
ORDER BY top.revenue DESC
LIMIT 10;

CROSS JOIN LATERAL here silently skips artists who never sold a track (their subquery returns no rows) - fine for a "top sellers" list. If you needed every artist listed, LEFT JOIN LATERAL ... ON true keeps them with a NULL album. Module 6 rebuilds this with ROW_NUMBER() OVER (PARTITION BY artist); LATERAL is the pre-window way and still the cleaner tool when the per-group query is genuinely bespoke.

P5.15chinookinterview

Recursive CTE walking up: starting from Laura Callahan, list her whole management chain - herself, her manager, her manager's manager, up to the top. Return each person's first and last name.

hint
Anchor on the one employee (last_name = 'Callahan'). The recursive step joins to the manager: employee m JOIN chain c ON c.reports_to = m.employee_id - the opposite direction from the top-down org walk in 5.4.
solution
WITH RECURSIVE chain AS (
    SELECT employee_id, first_name, last_name, reports_to
    FROM employee
    WHERE last_name = 'Callahan'

    UNION ALL

    SELECT m.employee_id, m.first_name, m.last_name, m.reports_to
    FROM employee m
    JOIN chain c ON c.reports_to = m.employee_id
)
SELECT first_name, last_name FROM chain;

Walking up gives one row per level (Callahan, then Mitchell, then Adams) because each person has exactly one manager; walking down (5.4) fans out. The direction is entirely in the join condition - c.reports_to = m.employee_id climbs, e.reports_to = org.employee_id descends. Same table, same recursion, opposite arrow.

P5.16chinookcore

Gap-free calendar: for every month of 2023, show the invoice count and revenue, including months with none (they must appear as zero, not vanish). Generate the months with a recursive CTE, then LEFT JOIN the invoices.

hint
Anchor at DATE '2023-01-01', add one month per step until December. LEFT JOIN invoices on date_trunc('month', invoice_date)::date = month_start, then COUNT and SUM.
solution
WITH RECURSIVE months AS (
    SELECT DATE '2023-01-01' AS month_start
    UNION ALL
    SELECT (month_start + INTERVAL '1 month')::date
    FROM months
    WHERE month_start < DATE '2023-12-01'
)
SELECT m.month_start,
       COUNT(i.invoice_id) AS invoices,
       COALESCE(ROUND(SUM(i.total)::numeric, 2), 0) AS revenue
FROM months m
LEFT JOIN invoice i ON date_trunc('month', i.invoice_date)::date = m.month_start
GROUP BY m.month_start
ORDER BY m.month_start;

COUNT(i.invoice_id) - not COUNT(*) - correctly returns 0 for an empty month, because counting a specific column ignores the NULL rows the LEFT JOIN produces. That one detail is the difference between a right and a subtly wrong calendar report.
