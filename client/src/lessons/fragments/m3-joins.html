-- module 3 / four sessions /

Joins

Everything downstream - CTEs, windows, interview patterns - is joins plus something. This module covers all the join types, the two traps that fail otherwise-strong candidates (fan-out and the LEFT-JOIN-killing WHERE clause), and the anti-join, which is the single most-asked join question in interviews. Your chinook store has a rich foreign-key graph to walk (artist to album to track to genre; customer to invoice to line; employee to employee), and stackoverflow supplies the self-join and the big anti-join.

3.1 INNER, LEFT, RIGHT, FULL in one mental model

A join matches rows by a condition. The only question the four types answer differently is: what happens to rows with no match?

TypeUnmatched left rowsUnmatched right rowsTypical use
INNER JOINdroppeddroppedonly complete pairs
LEFT JOINkept (right columns NULL)dropped"all customers, with invoices if any"
RIGHT JOINdroppedkeptrare; rewrite as LEFT for readability
FULL JOINkeptkeptreconciling two lists
-- chinook: every invoice with its customer name
SELECT i.invoice_id, i.invoice_date, c.first_name, c.last_name
FROM invoice i
JOIN customer c ON c.customer_id = i.customer_id;
-- Alias every table, prefix every column. Non-negotiable style in interviews.

3.2 The trap that kills LEFT JOINs

-- Goal: ALL customers, and their 2023 invoice count (zero if none).

-- WRONG: the WHERE filter on the right table silently converts LEFT to INNER,
-- because unmatched rows have i.invoice_date = NULL, and NULL fails the predicate.
SELECT c.customer_id, COUNT(i.invoice_id) AS invoices_2023
FROM customer c
LEFT JOIN invoice i ON i.customer_id = c.customer_id
WHERE i.invoice_date >= DATE '2023-01-01'
  AND i.invoice_date <  DATE '2024-01-01'
GROUP BY c.customer_id;

-- RIGHT: put the right-table filter in the ON clause. Unmatched customers survive.
SELECT c.customer_id, COUNT(i.invoice_id) AS invoices_2023
FROM customer c
LEFT JOIN invoice i
       ON i.customer_id = c.customer_id
      AND i.invoice_date >= DATE '2023-01-01'
      AND i.invoice_date <  DATE '2024-01-01'
GROUP BY c.customer_id;
Rule With OUTER joins: conditions on the preserved (left) table go in WHERE, conditions on the nullable (right) table go in ON. With INNER joins it makes no difference to the result. Being able to state this rule is worth real points.

Also note COUNT(i.invoice_id), not COUNT(*): for a customer with no invoices, COUNT(*) counts the one all-NULL joined row and reports 1; COUNT(i.invoice_id) correctly reports 0.

3.3 The anti-join: rows in A with no match in B

"Tracks that never sold" - the anti-join shape is the most common interview join question. Three ways:

-- 1. LEFT JOIN ... IS NULL
SELECT t.track_id, t.name
FROM track t
LEFT JOIN invoice_line il ON il.track_id = t.track_id
WHERE il.invoice_line_id IS NULL;

-- 2. NOT EXISTS (the one to prefer)
SELECT t.track_id, t.name
FROM track t
WHERE NOT EXISTS (
    SELECT 1 FROM invoice_line il WHERE il.track_id = t.track_id
);

-- 3. NOT IN -- know it, and know why to avoid it:
-- if the subquery returns even one NULL, NOT IN returns ZERO rows total.
SELECT track_id, name
FROM track
WHERE track_id NOT IN (SELECT track_id FROM invoice_line WHERE track_id IS NOT NULL);
Interview soundbite "I default to NOT EXISTS: it is NULL-safe, and the planner treats it as a proper anti-join. NOT IN has a footgun where a single NULL in the subquery empties the result."

3.4 Fan-out: the silent revenue inflator

Joining a one-to-many relationship duplicates the "one" side's values across every matched row. Aggregate after joining without noticing, and your totals inflate.

-- chinook: invoice (1) to invoice_line (many).
-- WRONG if you meant to sum each invoice's total once:
SELECT SUM(i.total)
FROM invoice i
JOIN invoice_line il ON il.invoice_id = i.invoice_id;
-- total is repeated once per line item. A 5-line invoice counts its total 5 times.
-- The true SUM(total) is 2328.60; this fanned-out version reports 20848.62.

-- RIGHT: aggregate the many side first, then join at matching grain:
SELECT SUM(i.total) AS total_invoiced,
       SUM(il.line_revenue) AS total_line_revenue
FROM invoice i
JOIN (
    SELECT invoice_id,
           SUM(unit_price * quantity) AS line_revenue
    FROM invoice_line
    GROUP BY invoice_id
) il ON il.invoice_id = i.invoice_id;
The senior habit Before writing any join, say the grain of each table: "invoice is one row per invoice; invoice_line is one row per invoice-track. Joining them puts me at line grain, so invoice-level columns will repeat." Interviewers listen for exactly this.

3.5 Self-joins and multi-hop joins

-- chinook: employee and manager on one row (self-join)
SELECT e.first_name || ' ' || e.last_name AS employee,
       m.first_name || ' ' || m.last_name AS manager
FROM employee e
LEFT JOIN employee m ON m.employee_id = e.reports_to;
-- LEFT so the top of the hierarchy (reports_to IS NULL) still appears.
-- chinook: three-table chain, genre revenue
SELECT g.name AS genre,
       ROUND(SUM(il.unit_price * il.quantity)::numeric, 2) AS revenue
FROM invoice_line il
JOIN track t ON t.track_id = il.track_id
JOIN genre g ON g.genre_id = t.genre_id
GROUP BY g.name
ORDER BY revenue DESC;

Practice - Session A (problems 1-3)

P3.1chinookwarmup

List every album title with its artist name, alphabetized by artist then album.

solution
SELECT ar.name AS artist, al.title AS album
FROM album al
JOIN artist ar ON ar.artist_id = al.artist_id
ORDER BY ar.name, al.title;
P3.2chinookwarmup

Every track with its genre name and its album title. Guard against tracks whose album link could be missing - keep such a track even if no album matches.

hint
Which of the two joins needs to be LEFT to satisfy "keep it even if the album is missing"?
solution
SELECT t.name AS track, g.name AS genre, al.title AS album
FROM track t
JOIN genre g ON g.genre_id = t.genre_id
LEFT JOIN album al ON al.album_id = t.album_id
ORDER BY t.name;

Mixed join types in one query are normal - choose per relationship, not per query. Every track has a genre, so INNER is safe there; the album link is the kind of nullable foreign key you defend with a LEFT so a missing album never silently drops a track.

P3.3chinookcore

Walk the catalog hierarchy: every track's name, its album title, and its artist name. Preserve every track even if the album (and therefore the artist) cannot be resolved - show NULLs rather than dropping the row. Three tables, two hops.

solution
SELECT t.name AS track,
       al.title AS album,
       ar.name AS artist
FROM track t
LEFT JOIN album al
       ON al.album_id = t.album_id
LEFT JOIN artist ar
       ON ar.artist_id = al.artist_id
ORDER BY artist NULLS LAST, album, track;

Both hops must be LEFT: an INNER on the second hop would drop any row the first LEFT preserved with a NULL album, because its al columns are NULL and cannot match an artist. The rule generalizes - once you go LEFT, every downstream hop that depends on the nullable side must stay LEFT too.

Practice - Session B (problems 4-6)

P3.4chinookcore

Anti-join, all three ways: find every track that has never appeared on any invoice line. Write LEFT JOIN / IS NULL, NOT EXISTS, and NOT IN versions and confirm they return the same count (there are 1,519 such tracks).

solution
-- 1
SELECT COUNT(*) FROM track t
LEFT JOIN invoice_line il ON il.track_id = t.track_id
WHERE il.invoice_line_id IS NULL;

-- 2
SELECT COUNT(*) FROM track t
WHERE NOT EXISTS (SELECT 1 FROM invoice_line il WHERE il.track_id = t.track_id);

-- 3
SELECT COUNT(*) FROM track
WHERE track_id NOT IN (SELECT track_id FROM invoice_line);

All three match here because invoice_line.track_id has no NULLs. In an interview, write NOT EXISTS and mention the NOT IN NULL hazard unprompted.

P3.5stackoverflowcore

Self-join on posts: for the 20 highest-scoring questions, show the question title, question score, and the score of its accepted answer. Questions without an accepted answer must still appear.

hint
Join posts to posts on a.id = q.acceptedanswerid. Which side is preserved? And remember: a question is posttypeid = 1.
solution
SELECT q.title,
       q.score AS question_score,
       a.score AS accepted_answer_score
FROM posts q
LEFT JOIN posts a ON a.id = q.acceptedanswerid
WHERE q.posttypeid = 1
ORDER BY q.score DESC
LIMIT 20;

The WHERE filter is on q, the preserved side, so it does not break the LEFT JOIN - contrast with the 3.2 trap where the filter was on the nullable side. Every column here is lowercase, no underscores: acceptedanswerid, posttypeid.

P3.6chinookcore

The 3.2 trap, live: all customers with their count of invoices billed to Brazil (billing_country = 'Brazil'), including customers with zero. Verify your row count equals the customer table row count (59).

solution
SELECT c.customer_id, c.first_name, c.last_name,
       COUNT(i.invoice_id) AS brazil_invoices
FROM customer c
LEFT JOIN invoice i
       ON i.customer_id = c.customer_id
      AND i.billing_country = 'Brazil'
GROUP BY c.customer_id, c.first_name, c.last_name
ORDER BY brazil_invoices DESC;

The country filter lives in ON. Put it in WHERE and customers with no Brazilian invoices vanish; the row-count check catches it instantly. Making that check without being told is senior behavior.

Practice - Session C (problems 7-9)

P3.7chinookinterview

Five-table chain: total revenue per artist (invoice_line -> track -> album -> artist). Top 10 artists. Then answer: could an artist be missing from your result, and does that matter?

solution
SELECT ar.name AS artist,
       ROUND(SUM(il.unit_price * il.quantity)::numeric, 2) AS revenue
FROM invoice_line il
JOIN track t  ON t.track_id  = il.track_id
JOIN album al ON al.album_id = t.album_id
JOIN artist ar ON ar.artist_id = al.artist_id
GROUP BY ar.name
ORDER BY revenue DESC
LIMIT 10;

Yes: artists with zero sales never appear (INNER joins drop them). For a top-10 that is fine; for "revenue per artist including zero" you would start FROM artist and LEFT JOIN down the chain. Choosing the join direction from the question's wording is the skill being tested.

P3.8chinookinterview

Fan-out drill. Produce, in one query per invoice: invoice_id, the invoice's own total, and its total line revenue (SUM(unit_price * quantity)). Then produce the grand totals of both across all invoices, being careful that the invoice total is not inflated. (Two queries total: per-invoice, then grand total.)

hint
Per-invoice is safe if you aggregate the lines in a subquery first, then join at invoice grain. For grand totals, sum the pre-aggregated per-invoice rows - the bare join fans out and multiplies total.
solution
-- per invoice (subquery pre-aggregation, the pattern to internalize)
SELECT i.invoice_id, i.total, il.line_revenue
FROM invoice i
JOIN (
    SELECT invoice_id,
           SUM(unit_price * quantity) AS line_revenue
    FROM invoice_line
    GROUP BY invoice_id
) il ON il.invoice_id = i.invoice_id;

-- grand totals: wrap the same shape
SELECT SUM(i.total) AS total_invoiced,
       SUM(il.line_revenue) AS total_line_revenue
FROM invoice i
JOIN (
    SELECT invoice_id,
           SUM(unit_price * quantity) AS line_revenue
    FROM invoice_line
    GROUP BY invoice_id
) il ON il.invoice_id = i.invoice_id;

Sanity check: SELECT SUM(total) FROM invoice alone should match total_invoiced (2328.60). Join invoice straight to invoice_line without pre-aggregating and it balloons to 20848.62 - that is fan-out. This aggregate-then-join move becomes one CTE in Module 5.

P3.9chinookinterview

Average invoices per customer including customers with zero invoices. Then explain how the number would differ if you had used an INNER join instead.

solution
SELECT ROUND(AVG(invoice_count), 2) AS avg_invoices_per_customer
FROM (
    SELECT c.customer_id, COUNT(i.invoice_id) AS invoice_count
    FROM customer c
    LEFT JOIN invoice i ON i.customer_id = c.customer_id
    GROUP BY c.customer_id
) per_customer;

The LEFT JOIN keeps zero-invoice customers as invoice_count = 0, pulling the average down; an INNER join would silently drop them and inflate the average by dividing over fewer customers. Presenting both numbers and asking "which does the business mean?" is the senior answer to an ambiguous prompt.

Practice - Session D (problems 10-12)

P3.10stackoverflowinterview

Users who have earned badges but have never posted anything. Return displayname, reputation, and their badge count. (Combines a join, an anti-join, and aggregation.) Order by badge count, highest first, and take the top 50.

solution
SELECT u.displayname, u.reputation, COUNT(b.id) AS badge_count
FROM users u
JOIN badges b ON b.userid = u.id
WHERE NOT EXISTS (
    SELECT 1 FROM posts p WHERE p.owneruserid = u.id
)
GROUP BY u.id, u.displayname, u.reputation
ORDER BY badge_count DESC
LIMIT 50;

Note GROUP BY includes u.id: two users can share a displayname, and grouping by the key preserves them as separate rows. Grouping by name alone would silently merge them - a subtle correctness point interviewers love. Every column is lowercase, no underscores: userid on badges, owneruserid on posts, displayname on users. About 92k users hold a badge yet never posted.

P3.11chinookhard

Employee-customer-revenue chain: each support rep (employee) with the number of customers they support and those customers' total invoice revenue. Include reps with zero customers. Watch the double fan-out: employee -> customers -> invoices.

hint
Aggregate invoices to one row per customer first, then LEFT JOIN employee -> customer -> that aggregate.
solution
SELECT e.employee_id,
       e.first_name || ' ' || e.last_name AS rep,
       COUNT(DISTINCT c.customer_id) AS customers,
       COALESCE(SUM(ci.customer_revenue), 0) AS revenue
FROM employee e
LEFT JOIN customer c ON c.support_rep_id = e.employee_id
LEFT JOIN (
    SELECT customer_id, SUM(total) AS customer_revenue
    FROM invoice
    GROUP BY customer_id
) ci ON ci.customer_id = c.customer_id
GROUP BY e.employee_id, rep
ORDER BY revenue DESC;

Because invoices were pre-collapsed to one row per customer, the join cannot inflate revenue, and COUNT(DISTINCT c.customer_id) guards the customer count. COALESCE turns the no-customer reps' NULL sum into 0. Only three employees (the sales agents) support customers here; the rest report zero.

P3.12chinookhard

FULL JOIN reconciliation: build two lists - countries that appear as customer countries, and countries that appear as employee countries - and produce one report showing every country, whether it has customers, employees, or both, with counts. (This "reconcile two lists" shape is a real senior take-home staple.)

solution
SELECT COALESCE(cu.country, em.country) AS country,
       COALESCE(cu.customer_count, 0)   AS customers,
       COALESCE(em.employee_count, 0)   AS employees,
       CASE
         WHEN cu.country IS NULL THEN 'employees only'
         WHEN em.country IS NULL THEN 'customers only'
         ELSE 'both'
       END AS presence
FROM (
    SELECT country, COUNT(*) AS customer_count
    FROM customer GROUP BY country
) cu
FULL JOIN (
    SELECT country, COUNT(*) AS employee_count
    FROM employee GROUP BY country
) em ON em.country = cu.country
ORDER BY country;

The FULL JOIN essentials in one query: COALESCE on the join key (either side can be NULL) and IS NULL tests to classify which side was missing. All eight employees sit in Canada, so most countries land in "customers only" and Canada is "both."
