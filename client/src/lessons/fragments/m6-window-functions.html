-- module 6 / ten sessions /

Window Functions

This is the module that decides senior interviews. A window function computes across a set of related rows without collapsing them - GROUP BY answers "what is the total per group," windows answer "how does this row compare to its group." Your Stack Overflow dump - a quarter-million users, 243k posts - is where most of these earn their keep: rank users by reputation, top answers per tag, month-over-month post counts. chinook stays for the cases where a small, readable result makes the shape obvious. We go past the basics into the parts that actually separate candidates: the frame clause and ROWS vs RANGE, the FIRST_VALUE / LAST_VALUE / NTH_VALUE frame traps, LAG and LEAD in both directions, centered vs trailing moving averages, deduplication with ROW_NUMBER, and percent-of-partition. Give this module twice the reps of anything else.

6.1 Anatomy of a window

function(...) OVER (
    PARTITION BY ...   -- restart the calculation per group (optional)
    ORDER BY ...       -- define row order inside the partition (needed for rank/lag/running)
    ROWS BETWEEN ...   -- the frame: which nearby rows feed the function (optional)
)
-- chinook: every track, its price, and its genre's average - side by side
SELECT name, genre_id, unit_price,
       AVG(unit_price) OVER (PARTITION BY genre_id) AS genre_avg,
       unit_price - AVG(unit_price) OVER (PARTITION BY genre_id) AS vs_genre
FROM track;

Every row survives. GROUP BY would have collapsed each genre to one row and lost the track detail - that contrast is the definition to give when asked "what is a window function?"

Where windows may appear Only in SELECT and ORDER BY. Never in WHERE, GROUP BY, or HAVING - windows are computed after those steps (recall Module 2's execution order). "Filter on a window function" therefore always means: compute it in a CTE, filter in the next step. That sentence answers about a quarter of all window interview questions.

6.2 The ranking trio

-- stackoverflow: rank users by reputation
SELECT displayname, reputation,
       ROW_NUMBER() OVER (ORDER BY reputation DESC) AS row_num,     -- 1,2,3,4 - ties broken arbitrarily
       RANK()       OVER (ORDER BY reputation DESC) AS rnk,         -- 1,2,2,4 - ties share, gaps after
       DENSE_RANK() OVER (ORDER BY reputation DESC) AS dense_rnk    -- 1,2,2,3 - ties share, no gaps
FROM users
LIMIT 20;
Which one? The interview answer "Exactly one row per group even with ties" - ROW_NUMBER (dedup, pagination). "Olympic-style, ties share and the next rank skips" - RANK. "Give me everything in the top 3 values" - DENSE_RANK. If they ask "second highest salary," ask whether ties should both appear; the choice between the three IS the question.

6.3 Top-N per group: the most-asked interview pattern, period

-- chinook: top 3 tracks by revenue within each genre
WITH track_revenue AS (
    SELECT t.genre_id, t.name AS track_name,
           SUM(il.unit_price * il.quantity) AS revenue
    FROM invoice_line il
    JOIN track t ON t.track_id = il.track_id
    GROUP BY t.genre_id, t.name
),
ranked AS (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY genre_id
                              ORDER BY revenue DESC) AS rn
    FROM track_revenue
)
SELECT genre_id, track_name, ROUND(revenue::numeric, 2) AS revenue
FROM ranked
WHERE rn <= 3
ORDER BY genre_id, rn;

Aggregate, rank in a CTE, filter the rank outside. Drill this shape until you can type it half asleep - it will appear in your interview in some costume (top products per category, top answers per tag, top sellers per region). The costume changes; the three-step skeleton does not.

6.4 LAG and LEAD: reaching into other rows

-- stackoverflow: month-over-month question volume
WITH monthly AS (
    SELECT date_trunc('month', creationdate)::date AS month,
           COUNT(*) AS questions
    FROM posts
    WHERE posttypeid = 1
    GROUP BY 1
)
SELECT month, questions,
       LAG(questions) OVER (ORDER BY month) AS prev_month,
       questions - LAG(questions) OVER (ORDER BY month) AS change,
       ROUND(100.0 * (questions - LAG(questions) OVER (ORDER BY month))
             / NULLIF(LAG(questions) OVER (ORDER BY month), 0), 1) AS pct_change
FROM monthly
ORDER BY month;

LAG(x) is the previous row's x; LEAD(x) the next row's; both take an optional offset and default: LAG(questions, 1, 0). The first row's LAG is NULL - decide deliberately whether that becomes 0 (via the default) or stays NULL (no prior period exists, which is usually the honest answer). Note the order of operations: aggregate to one row per month first, then window over those months - windowing the raw posts would compare individual posts, not months.

6.5 Running totals and frames

-- chinook: running revenue through time, per customer lifetime, and a 3-row moving average
SELECT invoice_id, customer_id, invoice_date, total,
       SUM(total) OVER (ORDER BY invoice_date, invoice_id)         AS running_total,
       SUM(total) OVER (PARTITION BY customer_id
                        ORDER BY invoice_date, invoice_id)        AS customer_running,
       AVG(total) OVER (ORDER BY invoice_date, invoice_id
                        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW) AS moving_avg_3
FROM invoice;
The frame gotcha worth points With ORDER BY but no explicit frame, the default is RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW - and RANGE includes all peer rows that tie on the ORDER BY value. Two invoices on the same date show the same running total. If you want strict row-by-row accumulation, either add a tiebreaker to ORDER BY (as above, invoice_id) or write ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW explicitly. Knowing ROWS vs RANGE is genuinely differentiating.

6.6 Percent of total, NTILE, FIRST_VALUE

-- Percent of total without a self-join (the Module 5.6 refactor):
SELECT billing_country,
       SUM(total) AS revenue,
       ROUND(100.0 * SUM(total) / SUM(SUM(total)) OVER (), 1) AS pct_of_total
FROM invoice
GROUP BY billing_country
ORDER BY revenue DESC;
-- SUM(SUM(total)) OVER (): inner SUM is the group aggregate, outer is a window across groups.
-- stackoverflow NTILE: split users into equal buckets (quartiles here)
SELECT displayname, reputation,
       NTILE(4) OVER (ORDER BY reputation) AS quartile
FROM users
WHERE reputation > 1
LIMIT 20;
-- FIRST_VALUE / LAST_VALUE: LAST_VALUE needs the full frame or it lies
SELECT name, genre_id, milliseconds,
       FIRST_VALUE(name) OVER w AS shortest_in_genre,
       LAST_VALUE(name)  OVER (w ROWS BETWEEN UNBOUNDED PRECEDING
                                       AND UNBOUNDED FOLLOWING) AS longest_in_genre
FROM track
WINDOW w AS (PARTITION BY genre_id ORDER BY milliseconds);
-- Without the frame, LAST_VALUE's default frame ends at CURRENT ROW: it returns the row itself.
-- Also note the WINDOW clause: name a window once, reuse it. Clean and impressive.

6.7 ROWS vs RANGE: the frame you did not ask for

-- chinook: the same running total computed two ways, on a date where two invoices tie
SELECT invoice_id, invoice_date::date, total,
       SUM(total) OVER (ORDER BY invoice_date::date
                        RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS range_total,
       SUM(total) OVER (ORDER BY invoice_date::date, invoice_id
                        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS rows_total
FROM invoice
ORDER BY invoice_date::date, invoice_id;

On the two invoices that share 2021-02-01, range_total reads 39.60 on both rows while rows_total reads 37.62 then 39.60. That is the whole distinction in one screenshot: ROWS counts physical rows, so each row extends the frame by exactly one; RANGE counts logical peers, so every row that ties on the ORDER BY value shares one frame and therefore one value. RANGE is the default whenever you write ORDER BY without a frame - which is why "my running total repeats on tied dates" is the single most common window surprise.

Why it matters Reach for ROWS by default for running totals and moving windows - it does the row-by-row thing people picture. Reach for RANGE deliberately when you actually want peers grouped (e.g. "cumulative revenue as of the end of each day" where same-day order is meaningless). The offsets are also different vocabularies: ROWS takes a row count (2 PRECEDING = two rows back), RANGE takes a value offset (INTERVAL '7 days' PRECEDING over a date column). Saying "ROWS is positional, RANGE is value-based" out loud is a clean senior signal.

6.8 NTH_VALUE and the frame that makes LAST_VALUE lie

-- chinook: shortest, second-shortest, and longest track per genre (genre 1 shown)
SELECT name, genre_id, milliseconds,
       FIRST_VALUE(name)   OVER w AS shortest,
       NTH_VALUE(name, 2) OVER w AS second_shortest,
       LAST_VALUE(name)    OVER (w ROWS BETWEEN UNBOUNDED PRECEDING
                                        AND UNBOUNDED FOLLOWING) AS longest
FROM track
WHERE genre_id = 1
WINDOW w AS (PARTITION BY genre_id ORDER BY milliseconds);

Both LAST_VALUE and NTH_VALUE are held hostage by the default frame. With ORDER BY present, the frame runs UNBOUNDED PRECEDING to CURRENT ROW - so on the first row of a genre, NTH_VALUE(name, 2) has no second row in its frame yet and returns NULL, and LAST_VALUE returns the current row rather than the genuine last. FIRST_VALUE looks fine only because the frame always starts at the top. The fix is the same for both: widen the frame to the full partition with ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING.

Why it matters "Get the last / Nth value in a partition" is a trap laid on purpose in interviews. The naive query runs without error and returns plausible-looking rubbish - exactly the failure mode reviewers probe for. If you only remember one thing: FIRST_VALUE is safe under the default frame, LAST_VALUE and NTH_VALUE are not.

6.9 Deduplicating with ROW_NUMBER

-- stackoverflow: keep the single highest-reputation user per display name
WITH ranked AS (
    SELECT id, displayname, reputation,
           ROW_NUMBER() OVER (PARTITION BY displayname
                              ORDER BY reputation DESC, id) AS rn,
           COUNT(*)     OVER (PARTITION BY displayname) AS name_count
    FROM users
)
SELECT displayname, id, reputation, name_count
FROM ranked
WHERE rn = 1 AND name_count BETWEEN 2 AND 5
ORDER BY reputation DESC;

The dedup recipe: PARTITION BY the columns that define "the same thing," ORDER BY a rule that decides which duplicate wins (here highest reputation, id as a deterministic tiebreaker), then keep rn = 1. The parallel COUNT(*) OVER (PARTITION BY displayname) lets you see how many collided without a second pass - handy for confirming there really were duplicates before you delete anything.

Why it matters This is the export-cleanup and idempotency workhorse: collapse a table to one row per key, keeping the "best" copy. It beats SELECT DISTINCT the moment your tiebreak needs a rule ("keep the most recent," "keep the highest score"), and it beats a GROUP-BY-then-join because it never widens or double-counts. The tiebreaker column in ORDER BY is not optional - without it, "which duplicate survives" is nondeterministic and your results shift run to run.

6.10 Bounded frames and centered averages

-- chinook: monthly revenue with a centered 3-month moving average
WITH monthly AS (
    SELECT date_trunc('month', invoice_date)::date AS month,
           SUM(total) AS revenue
    FROM invoice
    GROUP BY 1
)
SELECT month, revenue,
       ROUND(AVG(revenue) OVER (ORDER BY month
                                ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING), 2) AS centered_ma_3,
       COUNT(*) OVER (ORDER BY month
                      ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING) AS months_in_frame
FROM monthly
ORDER BY month;

Frames reach forward too. 1 PRECEDING AND 1 FOLLOWING centers the window on the current row - each smoothed value blends the month before, the month itself, and the month after. Contrast with the trailing 6 PRECEDING AND CURRENT ROW from 6.5: trailing looks only backward (no peeking at the future, which is what you want for a live dashboard), centered is symmetric (better for after-the-fact trend analysis). The endpoints tell on themselves again - the first and last month only have two rows in frame (see months_in_frame), so their smoothed value is built from fewer neighbors.

Why it matters The frame is a tiny sublanguage - UNBOUNDED PRECEDING, n PRECEDING, CURRENT ROW, n FOLLOWING, UNBOUNDED FOLLOWING - and every moving-window question is just a choice of two endpoints. Trailing vs centered is a real interview follow-up ("would you use future rows here?"): on anything forecast-facing or streaming, using FOLLOWING rows leaks the future and is a bug, not a smoothing choice.

P6.1stackoverflowwarmup

Rank the top 15 users by reputation with all three ranking functions in one query. Find (or construct in your head) a tie and explain each column's value at the tie.

solution
SELECT displayname, reputation,
       ROW_NUMBER() OVER w AS row_num,
       RANK()       OVER w AS rnk,
       DENSE_RANK() OVER w AS dense_rnk
FROM users
WINDOW w AS (ORDER BY reputation DESC)
ORDER BY reputation DESC
LIMIT 15;

At a two-way tie for 3rd: row_num gives 3 and 4 (arbitrary order), rnk gives 3 and 3 then jumps to 5, dense_rnk gives 3 and 3 then 4. The top of this list is the real DBA Stack Exchange leaderboard - the famous high-rep names are genuinely in your data.

P6.2chinookcore

The classic: the single longest track in each genre. Genre name, track name, length in minutes. Exactly one row per genre.

hint
"Exactly one per group" = ROW_NUMBER, PARTITION BY genre, ORDER BY milliseconds DESC, filter rn = 1 outside the CTE.
solution
WITH ranked AS (
    SELECT g.name AS genre, t.name AS track,
           ROUND(t.milliseconds / 60000.0, 1) AS minutes,
           ROW_NUMBER() OVER (PARTITION BY g.genre_id
                              ORDER BY t.milliseconds DESC) AS rn
    FROM track t
    JOIN genre g ON g.genre_id = t.genre_id
)
SELECT genre, track, minutes
FROM ranked
WHERE rn = 1
ORDER BY minutes DESC;

Follow-up they will ask: "what if two tracks tie for longest?" ROW_NUMBER keeps one arbitrarily; RANK() with WHERE rnk = 1 keeps both. State the tradeoff, ask which they want.

P6.3chinookcore

Each genre's 3 highest-revenue tracks: genre_id, track name, revenue, and the rank within the genre. Revenue is unit_price * quantity summed from invoice_line. (Top-N per group again, deliberately, one day after 6.2 - this is the spaced repetition.)

solution
WITH track_revenue AS (

           SUM(il.unit_price * il.quantity) AS revenue
    FROM invoice_line il
    JOIN track t ON t.track_id = il.track_id
    GROUP BY t.genre_id, t.name
),
ranked AS (

FROM ranked
WHERE rn <= 3
ORDER BY genre_id, rn;

Identical skeleton to the section 6.3 worked example above: aggregate the grain you want (track within genre), rank inside a CTE, filter rn <= 3 outside. Say "top-N per group" out loud when you recognize it - naming the pattern is half the interview.

P6.4stackoverflowcore

Monthly answer volume with month-over-month change and percent change (1 decimal). Count answers (posttypeid = 2) per calendar month first, then window over the months.

solution
WITH monthly AS (
    SELECT date_trunc('month', creationdate)::date AS month,
           COUNT(*) AS answers
    FROM posts
    WHERE posttypeid = 2
    GROUP BY 1
)
SELECT month, answers,
       answers - LAG(answers) OVER (ORDER BY month) AS mom_change,
       ROUND(100.0 * (answers - LAG(answers) OVER (ORDER BY month))
             / NULLIF(LAG(answers) OVER (ORDER BY month), 0), 1) AS mom_pct
FROM monthly
ORDER BY month;

Aggregate first, window second - windows on the raw post rows would compare posts, not months. That sequencing error is the most common window mistake in interviews. NULLIF guards the first month (LAG is NULL) and any month with a zero predecessor from a divide-by-zero.

P6.5chinookcore

Per customer: each invoice with its running lifetime total and the invoice's sequence number (1st, 2nd, 3rd purchase...). Order deterministically.

solution
SELECT customer_id, invoice_id, invoice_date, total,
       ROW_NUMBER() OVER w AS purchase_seq,
       SUM(total)   OVER (w ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS lifetime_running
FROM invoice
WINDOW w AS (PARTITION BY customer_id ORDER BY invoice_date, invoice_id)
ORDER BY customer_id, purchase_seq;

invoice_id in the window ORDER BY is the tiebreaker for same-day invoices; the explicit ROWS frame makes the running sum strictly row-by-row regardless of ties. Both details are the 6.5 concept paying off.

P6.6stackoverflowcore

7-day moving average of daily post counts. One CTE for the daily counts, then the window with an explicit ROWS frame in the outer query. Why is the first 6 days' "7-day average" misleading, and what would you do about it?

solution
WITH daily AS (
    SELECT creationdate::date AS day, COUNT(*) AS posts
    FROM posts
    GROUP BY 1
)
SELECT day, posts,
       ROUND(AVG(posts) OVER (ORDER BY day
                              ROWS BETWEEN 6 PRECEDING AND CURRENT ROW), 0) AS ma_7d,
       COUNT(*) OVER (ORDER BY day
                      ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) AS days_in_window
FROM daily
ORDER BY day;

The first 6 rows average fewer than 7 days (the frame is truncated at the start), so they look artificially volatile. Options: report days_in_window alongside (as here), filter to rows where it equals 7, or accept it and footnote it. Naming the options is the senior answer; silently shipping the misleading rows is the junior one.

P6.7chinookinterview

Genre revenue with percent of total revenue and a running cumulative percent (a Pareto/ABC analysis - straight out of manufacturing). Which genres make up the first 80%?

hint
Aggregate per genre, then two windows over the aggregate: SUM(SUM(...)) OVER () for the total and a running SUM ordered by revenue DESC.
solution
WITH genre_rev AS (
    SELECT g.name AS genre,
           SUM(il.unit_price * il.quantity) AS revenue
    FROM invoice_line il
    JOIN track t ON t.track_id = il.track_id
    JOIN genre g ON g.genre_id = t.genre_id
    GROUP BY g.name
)
SELECT genre,
       ROUND(revenue::numeric, 2) AS revenue,
       ROUND(100.0 * revenue / SUM(revenue) OVER (), 1) AS pct,
       ROUND(100.0 * SUM(revenue) OVER (ORDER BY revenue DESC)
             / SUM(revenue) OVER (), 1) AS cumulative_pct
FROM genre_rev
ORDER BY revenue DESC;

Read down cumulative_pct to the first row at or past 80 - everything above it is your A class. You already know this analysis from OEE work; now you can produce it in one query and say so in interviews.

P6.8stackoverflowinterview

Gaps between posts: for one prolific user (use owneruserid = 1876, the most prolific answerer here), list their posts in order with the time since their previous post. Then find their single longest gap.

solution
WITH gaps AS (
    SELECT owneruserid, id, creationdate,
           creationdate - LAG(creationdate) OVER (
               PARTITION BY owneruserid ORDER BY creationdate) AS gap
    FROM posts
    WHERE owneruserid = 1876   -- swap in any prolific user
)
SELECT * FROM gaps ORDER BY gap DESC NULLS LAST LIMIT 1;

The first post's gap is NULL (no predecessor) - NULLS LAST keeps it from masquerading as the longest gap. Subtracting two timestamps yields an interval, so the answer reads like "368 days ..." directly. LAG-against-yourself is the core of sessionization and retention, both coming in Module 7.

P6.9stackoverflowinterview

Answering cadence: for every user, the average number of days between consecutive answers. (LAG per user in a CTE, AVG outside.) Return the 10 fastest-answering users with at least 100 answers.

solution
WITH gaps AS (
    SELECT owneruserid,
           EXTRACT(EPOCH FROM creationdate - LAG(creationdate) OVER (
               PARTITION BY owneruserid ORDER BY creationdate, id)) / 86400 AS gap_days
    FROM posts
    WHERE posttypeid = 2 AND owneruserid IS NOT NULL
)
SELECT owneruserid,
       COUNT(gap_days) + 1              AS answers,
       ROUND(AVG(gap_days)::numeric, 2)   AS avg_days_between
FROM gaps
GROUP BY owneruserid
HAVING COUNT(gap_days) + 1 >= 100
ORDER BY avg_days_between
LIMIT 10;

COUNT(gap_days) skips the NULL first-answer row, so answers = gaps + 1. Windows inside the CTE, plain aggregation outside - the two layers never mix, which is the whole architecture of window-based analysis. User 1876 tops this too: thousands of answers, under a day apart on average.

P6.10stackoverflowinterview

Reputation quartiles: assign every user with reputation above 1 to a quartile with NTILE(4) over reputation, then summarize each quartile - user count, min and max reputation, average upvotes. What do you notice about upvotes across quartiles, and about the min/max reputation of the middle buckets?

solution
WITH bucketed AS (
    SELECT reputation, upvotes,
           NTILE(4) OVER (ORDER BY reputation) AS quartile
    FROM users
    WHERE reputation > 1
)
SELECT quartile,
       COUNT(*) AS users,
       MIN(reputation) AS min_rep,
       MAX(reputation) AS max_rep,
       ROUND(AVG(upvotes), 1) AS avg_upvotes
FROM bucketed
GROUP BY quartile
ORDER BY quartile;

NTILE forces near-equal bucket sizes, so when a value dominates (huge numbers of users sit at the default-ish low reputation) the middle quartiles can share the same min and max - the boundary between them is arbitrary. Contrast with CASE-based bucketing on fixed thresholds, where a bucket can be empty. "NTILE for equal populations, CASE for fixed cutoffs" is a tidy line to have ready; the upvote average climbing steeply into the top quartile is the real-world payoff.

P6.11stackoverflowhard

Redo the "each user's very first post" problem with ROW_NUMBER instead of a MIN-and-join, handling the same-day tie deterministically by lowest post id. Return the user, the first post's id, date, and whether it was a question or an answer. Compare the two solutions' length and clarity.

solution
WITH ranked AS (
    SELECT owneruserid, id, creationdate, posttypeid,
           ROW_NUMBER() OVER (PARTITION BY owneruserid
                              ORDER BY creationdate, id) AS rn
    FROM posts
    WHERE owneruserid IS NOT NULL
)
SELECT owneruserid,
       id AS first_post_id,
       creationdate::date AS first_post_date,
       CASE posttypeid WHEN 1 THEN 'question' WHEN 2 THEN 'answer' ELSE 'other' END AS first_post_kind
FROM ranked
WHERE rn = 1
ORDER BY first_post_date
LIMIT 20;

One CTE, ties resolved explicitly (creationdate then id), exactly one row per user guaranteed. The MIN-join alternative is GROUP BY owneruserid to get each user's MIN(creationdate), then join back to posts on user + date - two passes and a same-day ambiguity you have to break with another rule. In an interview, write the ROW_NUMBER version and mention the MIN-join as the alternative; showing you know both is the point.

P6.12stackoverflowhard

Tag league table by year: for each year, rank the four big database tags (postgresql, mysql, sql-server, oracle) by how many answers their questions attracted, and show only each year's top 3 with their answer count and share of that year's tagged answers (1 decimal). A tag is read off the parent question's pipe-delimited tags. Three window concepts in one query.

hint
Join each answer to its parent question (a.parentid = q.id), classify the question's tag with a CASE over tags LIKE '%|postgresql|%' etc., aggregate to (year, tag), then RANK within year and compute share with SUM(...) OVER (PARTITION BY yr).
solution
WITH yearly AS (
    SELECT EXTRACT(YEAR FROM a.creationdate)::int AS yr,
           CASE
               WHEN q.tags LIKE '%|postgresql|%' THEN 'postgresql'
               WHEN q.tags LIKE '%|mysql|%'      THEN 'mysql'
               WHEN q.tags LIKE '%|sql-server|%' THEN 'sql-server'
               WHEN q.tags LIKE '%|oracle|%'     THEN 'oracle'
               ELSE 'other'
           END AS tag,
           COUNT(*) AS answers
    FROM posts a
    JOIN posts q ON q.id = a.parentid
    WHERE a.posttypeid = 2
    GROUP BY 1, 2
),
ranked AS (
    SELECT yr, tag, answers,
           RANK() OVER (PARTITION BY yr ORDER BY answers DESC) AS rnk,
           ROUND(100.0 * answers / SUM(answers) OVER (PARTITION BY yr), 1) AS pct_of_year
    FROM yearly
    WHERE tag <> 'other'
)
SELECT yr, rnk, tag, answers, pct_of_year
FROM ranked
WHERE rnk <= 3
ORDER BY yr, rnk;

RANK (not ROW_NUMBER) so tied tags both make the podium. PARTITION BY yr appears in both windows - rank within year, share within year. The CASE ladder is order-sensitive: a question tagged both mysql and postgresql lands in postgresql because that branch is tested first - a real modeling choice to say out loud. This is a fair replica of a senior screen question.

P6.13stackoverflowhard

Per question, how long until it was answered, and the median response time by year. Combine your P5.5 CTE work with PERCENTILE_CONT (an ordered-set aggregate - close cousin of windows and a frequent interview cameo).

solution
WITH first_answers AS (
    SELECT parentid AS question_id, MIN(creationdate) AS first_answer_at
    FROM posts
    WHERE posttypeid = 2
    GROUP BY parentid
),
response_times AS (
    SELECT EXTRACT(YEAR FROM q.creationdate)::int AS yr,
           EXTRACT(EPOCH FROM fa.first_answer_at - q.creationdate) / 3600 AS hours_to_answer
    FROM posts q
    JOIN first_answers fa ON fa.question_id = q.id
    WHERE q.posttypeid = 1
)
SELECT yr,
       COUNT(*) AS answered_questions,
       ROUND(PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY hours_to_answer)::numeric, 2) AS median_hours,
       ROUND(AVG(hours_to_answer)::numeric, 2) AS mean_hours
FROM response_times
GROUP BY yr
ORDER BY yr;

Mean is dragged far above median by a few questions answered years later - report both and say why. "For skewed durations I default to median via PERCENTILE_CONT" is a sentence that separates senior candidates.

P6.14chinookhard

The comprehension test (predict before you run): explain, cell by cell for the first 6 output rows, what these four columns will contain, then run it and check yourself.

SELECT customer_id, invoice_date::date, total,
  SUM(total) OVER ()                                            AS a,
  SUM(total) OVER (PARTITION BY customer_id)                    AS b,
  SUM(total) OVER (ORDER BY invoice_date)                       AS c,
  SUM(total) OVER (PARTITION BY customer_id ORDER BY invoice_date) AS d
FROM invoice
ORDER BY customer_id, invoice_date
LIMIT 6;
solution

a: grand total of all invoices, identical on every row (empty OVER = one partition, no order, whole-set frame). b: the row's customer's lifetime total, identical within each customer. c: running total across ALL customers by date - and because the default frame is RANGE, same-date invoices (even for different customers) show the same value. d: running total within the customer by date, same RANGE caveat for a customer's same-day invoices.

If you got all four including the RANGE peer behavior, you understand windows well enough for any interview. If c surprised you, reread 6.5 - that gotcha is worth real points.

P6.15chinookcore

For customer 1, show every invoice with two running totals side by side: one ordered by date alone with a RANGE frame, one ordered by (date, invoice_id) with a ROWS frame. Do they ever disagree, and why?

hint
Both are SUM(total) OVER (PARTITION BY customer_id ORDER BY ...). One uses RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, the other ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW. A difference only appears when two invoices tie on the ORDER BY value.
solution
SELECT customer_id, invoice_id, invoice_date::date, total,
       SUM(total) OVER (PARTITION BY customer_id ORDER BY invoice_date::date
                        RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS range_running,
       SUM(total) OVER (PARTITION BY customer_id ORDER BY invoice_date::date, invoice_id
                        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS rows_running
FROM invoice
WHERE customer_id = 1
ORDER BY invoice_date::date, invoice_id;

If this customer never has two invoices on one date, the two columns match exactly - which is the honest answer to give: RANGE and ROWS diverge only at ties on the ORDER BY value. Add a tiebreaker column and a ROWS frame and the ambiguity is gone regardless. That is the 6.7 concept in a single query you can run to convince yourself.

P6.16chinookcore

Rank albums by their track count (most tracks first) with ROW_NUMBER, RANK, and DENSE_RANK in one query, top 15. Point to a spot where albums tie and read off what each of the three columns does there.

hint
Aggregate track count per album in a CTE (COUNT(*) over the album-to-track join), then apply all three ranking functions over a single named WINDOW w AS (ORDER BY tracks DESC).
solution
WITH album_len AS (
    SELECT al.album_id, al.title, COUNT(*) AS tracks
    FROM album al
    JOIN track t ON t.album_id = al.album_id
    GROUP BY al.album_id, al.title
)
SELECT title, tracks,
       ROW_NUMBER() OVER w AS row_num,
       RANK()       OVER w AS rnk,
       DENSE_RANK() OVER w AS dense_rnk
FROM album_len
WINDOW w AS (ORDER BY tracks DESC)
ORDER BY tracks DESC
LIMIT 15;

Many albums have the same track count, so ties are everywhere below the top: at a block of albums with equal tracks, row_num keeps incrementing (arbitrary order among them), rnk repeats the block's start rank then skips ahead, dense_rnk repeats then advances by one. Same data, three different questions - "give me a strict order," "Olympic standings," "distinct tiers" - answered by picking the function, not rewriting the query.

P6.17stackoverflowinterview

For every question with at least 5 answers, find the id and score of its second highest-scoring answer using NTH_VALUE (not ROW_NUMBER). Return the 10 questions whose runner-up answer scored highest. Mind the frame.

hint
NTH_VALUE under the default frame stops at CURRENT ROW and will return NULL on early rows. Give the window an explicit ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING so every row sees the whole partition, then DISTINCT the per-question result.
solution
WITH scored AS (
    SELECT parentid, id, score,
           NTH_VALUE(id, 2)    OVER w AS second_id,
           NTH_VALUE(score, 2) OVER w AS second_score,
           COUNT(*) OVER (PARTITION BY parentid) AS answer_count
    FROM posts
    WHERE posttypeid = 2 AND parentid IS NOT NULL
    WINDOW w AS (PARTITION BY parentid ORDER BY score DESC, id
                    ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING)
)
SELECT DISTINCT parentid AS question_id, second_id AS second_best_answer,
       second_score, answer_count
FROM scored
WHERE answer_count >= 5
ORDER BY second_score DESC
LIMIT 10;

Without the explicit full frame this returns NULL for the runner-up on early rows and quietly wrong answers elsewhere - the 6.8 trap. NTH_VALUE broadcasts the same second-place value to every row in the partition, so DISTINCT collapses back to one row per question. A ROW_NUMBER-then-filter-rn=2 solution is equally valid and often clearer; NTH_VALUE is the point here because interviewers ask about it specifically.

P6.18stackoverflowcore

For user 1876, list their posts in date order and, using LEAD, the date of their next post and the days until it. LAG looks back; this is the same idea looking forward.

hint
LEAD(creationdate) OVER (PARTITION BY owneruserid ORDER BY creationdate, id) is the next row's date. Subtract to get an interval; use EXTRACT(EPOCH FROM ...) / 86400 for total days rather than EXTRACT(DAY FROM ...), which only reads the day component of the interval.
solution
SELECT id, creationdate::date AS posted,
       (LEAD(creationdate) OVER w)::date AS next_post,
       ROUND(EXTRACT(EPOCH FROM LEAD(creationdate) OVER w - creationdate)
             / 86400.0, 1) AS days_to_next
FROM posts
WHERE owneruserid = 1876
WINDOW w AS (PARTITION BY owneruserid ORDER BY creationdate, id)
ORDER BY creationdate;

The last post has no successor, so its LEAD (and days_to_next) is NULL - the mirror image of LAG's NULL first row. Watch the interval trap: EXTRACT(DAY FROM interval) returns only the "days" field of an interval, so a 368-day gap could read as some small number; EPOCH over 86400 gives the true total. Order the same window by (creationdate, id) so ties on a busy day resolve deterministically.

P6.19chinookhard

Monthly revenue with a centered 3-month moving average and a count of how many months actually fell in each frame. Then explain why the very first and last months' smoothed values are the least trustworthy.

hint
Aggregate revenue per month in a CTE (date_trunc('month', invoice_date)), then an AVG with ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING. Add COUNT(*) over the same frame to expose the truncated endpoints.
solution
WITH monthly AS (
    SELECT date_trunc('month', invoice_date)::date AS month,
           SUM(total) AS revenue
    FROM invoice
    GROUP BY 1
)
SELECT month, revenue,
       ROUND(AVG(revenue) OVER w, 2) AS centered_ma_3,
       COUNT(*) OVER w AS months_in_frame
FROM monthly
WINDOW w AS (ORDER BY month ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING)
ORDER BY month;

months_in_frame is 2 on the first and last rows and 3 everywhere in between: an endpoint has only one neighbor, so its "3-month" average is really a 2-month average and jumps around more. Options are the same as any truncated-frame case - report the count, trim the endpoints, or footnote them. A centered frame trades that endpoint weakness for symmetry (no lag), which is the right call for retrospective trend work but wrong for anything that must not see the future.

P6.20stackoverflowhard

Concentration check for user 1876: among their positively scored posts, what share of their total score do their top 5 posts contribute? Return each top-5 post's score and its percent of the user's total.

hint
One pass: SUM(score) OVER (PARTITION BY owneruserid) gives the whole-user total on every row (a partition-level window, no ORDER BY), and ROW_NUMBER() OVER (... ORDER BY score DESC, id) ranks the posts. Filter rn <= 5 outside and divide, guarding with NULLIF.
solution
WITH user_posts AS (
    SELECT owneruserid, id, score,
           SUM(score)   OVER (PARTITION BY owneruserid) AS user_total,
           ROW_NUMBER() OVER (PARTITION BY owneruserid
                              ORDER BY score DESC, id) AS rn
    FROM posts
    WHERE owneruserid = 1876 AND score > 0
)
SELECT id, score, user_total,
       ROUND(100.0 * score / NULLIF(user_total, 0), 2) AS pct_of_user_total
FROM user_posts
WHERE rn <= 5
ORDER BY score DESC;

Two windows over the same partition doing different jobs: a partition-wide SUM (no ORDER BY, so the whole-partition total lands on every row) as the denominator, and a ROW_NUMBER for the top-5 cutoff. For a prolific answerer the top 5 posts are a tiny slice of a large total - the long-tail shape you would summarize as "no single post carries them." Same x / SUM(x) OVER (partition) percent-of-partition idea as 6.6, now scoped per user instead of globally.
