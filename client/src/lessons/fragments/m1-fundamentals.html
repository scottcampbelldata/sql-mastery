-- module 1 / three sessions /

Fundamentals

Retrieval, filtering, sorting, and the three-valued logic of NULL. Nothing here is hard, but senior candidates get dinged on NULL handling and operator precedence more often than on window functions. Get these airtight.

1.1 SELECT, aliases, and expressions

-- chinook: columns, an expression, and aliases
SELECT
    name                              AS track_name,
    milliseconds / 60000.0            AS minutes,   -- 60000.0 forces decimal division
    unit_price,
    unit_price * 1.10                 AS price_after_10pct_increase
FROM track
LIMIT 10;
Integer division trap In Postgres, milliseconds / 60000 with two integers returns an integer (truncated). Divide by 60000.0 or cast one side: milliseconds::numeric / 60000. Interviewers plant this trap constantly in "average" questions.

1.2 WHERE: comparison, IN, BETWEEN, LIKE

-- northwind: several filter styles in one query
SELECT product_name, unit_price, units_in_stock
FROM products
WHERE unit_price BETWEEN 10 AND 50          -- inclusive on both ends
  AND category_id IN (1, 2, 8)
  AND product_name ILIKE '%ch%'            -- ILIKE = case-insensitive (Postgres)
  AND discontinued = 0;
-- Operator precedence: AND binds tighter than OR. These two differ:
WHERE country = 'USA' OR country = 'Canada' AND city = 'Vancouver'
-- means: USA (any city), OR (Canada AND Vancouver)

WHERE (country = 'USA' OR country = 'Canada') AND city = 'Vancouver'
-- means: Vancouver rows in either country (i.e., just Canada in practice)
HabitAny time you mix AND with OR, parenthesize. Zero exceptions. It reads as careful, not as insecure.

1.3 NULL: the part beginners underestimate

NULL means "unknown," and comparisons with unknown are neither true nor false - they are NULL, and WHERE only keeps rows where the condition is true.

-- northwind: orders that have not shipped
SELECT order_id, order_date, shipped_date
FROM orders
WHERE shipped_date IS NULL;      -- correct

-- WHERE shipped_date = NULL      returns ZERO rows, always. = NULL is never true.
-- WHERE shipped_date != '1997-01-01' also silently DROPS the NULL rows.
-- To keep NULLs when using != you must ask for them explicitly:
WHERE shipped_date IS DISTINCT FROM DATE '1997-01-01'
-- IS DISTINCT FROM treats NULL as a comparable value. A great phrase to use in interviews.

1.4 ORDER BY, LIMIT, DISTINCT

-- stackoverflow: top reputation, newest first as tiebreaker
SELECT display_name, reputation, creation_date
FROM users
ORDER BY reputation DESC, creation_date DESC
LIMIT 20;
-- Postgres sorts NULLs LAST on ASC and FIRST on DESC by default. Control it:
ORDER BY shipped_date DESC NULLS LAST
-- DISTINCT applies to the whole selected row, not the first column:
SELECT DISTINCT ship_country, ship_city FROM orders;  -- unique (country, city) pairs
Interview soundbite "LIMIT without ORDER BY returns an arbitrary set of rows - the database gives you whatever is cheapest. If the question says top or first, I need an ORDER BY." Saying this unprompted is an easy credibility win.

Practice - Session A (problems 1-4)

P1.1chinookwarmup

List the 15 longest tracks. Show the track name, its length in whole minutes (rounded to 1 decimal), and its unit price. Longest first.

hint
Length lives in milliseconds. Watch integer division; use ROUND(x, 1).
solution
SELECT name,
       ROUND(milliseconds / 60000.0, 1) AS minutes,
       unit_price
FROM track
ORDER BY milliseconds DESC
LIMIT 15;

Note we ORDER BY the raw column, not the rounded alias - rounding can create ties that reorder rows. Either is accepted, but knowing the difference is the point.

P1.2northwindwarmup

Find all products priced between 20 and 40 (inclusive) that are not discontinued, in categories 1, 2, or 7. Show name, price, and stock, cheapest first, and break price ties by higher stock first.

solution
SELECT product_name, unit_price, units_in_stock
FROM products
WHERE unit_price BETWEEN 20 AND 40
  AND discontinued = 0
  AND category_id IN (1, 2, 7)
ORDER BY unit_price ASC, units_in_stock DESC;
P1.3northwindcore

Show every order that has an order_date but no shipped_date - the unshipped backlog. Return order_id, customer_id, order_date, oldest orders first.

hint
= NULL will silently return nothing. What is the correct predicate?
solution
SELECT order_id, customer_id, order_date
FROM orders
WHERE shipped_date IS NULL
ORDER BY order_date;

If asked "how would you verify this?", answer: compare COUNT(*) with COUNT(shipped_date) on the whole table - the difference should equal this query's row count.

P1.4stackoverflowcore

Find users whose display_name contains "sql" in any casing and whose reputation is at least 1000. Show name, reputation, and location, highest reputation first, and push NULL locations to the end without excluding them.

solution
SELECT display_name, reputation, location
FROM users
WHERE display_name ILIKE '%sql%'
  AND reputation >= 1000
ORDER BY reputation DESC, location NULLS LAST;

Two Postgres-flavored details in one query: ILIKE and NULLS LAST. In a dialect without ILIKE you would write LOWER(display_name) LIKE '%sql%'.

Practice - Session B (problems 5-8)

P1.5nyctaxicore

Pull 25 suspicious trips: fare over 200, distance under 1 mile. Show pickup_datetime, trip_distance, fare_amount, tip_amount, most expensive first. (Data-quality hunting like this is a real senior analyst task.)

solution
SELECT pickup_datetime, trip_distance, fare_amount, tip_amount
FROM trips
WHERE fare_amount > 200
  AND trip_distance < 1
ORDER BY fare_amount DESC
LIMIT 25;

On a huge table this may be slow without an index - fine for now; Module 8 revisits exactly this query with EXPLAIN ANALYZE.

P1.6northwindcore

List orders shipped to Germany or France where freight exceeded 100, OR any order (any country) with freight above 500. Parenthesize correctly. Return order_id, ship_country, freight.

hint
Read it as two groups joined by OR: (country condition AND freight > 100) OR (freight > 500).
solution
SELECT order_id, ship_country, freight
FROM orders
WHERE (ship_country IN ('Germany', 'France') AND freight > 100)
   OR freight > 500
ORDER BY freight DESC;

Without the parentheses, AND would bind first and quietly change the logic. This exact structure appears in interview screeners because it is quick to grade.

P1.7chinookcore

Produce the unique list of (billing_country, billing_city) pairs that appear on invoices, sorted by country then city. Then, without running it, predict: will SELECT DISTINCT billing_country return more, fewer, or the same number of rows?

solution
SELECT DISTINCT billing_country, billing_city
FROM invoice
ORDER BY billing_country, billing_city;

Fewer. DISTINCT on the pair keeps one row per unique combination; collapsing to country alone can only merge rows, never split them.

P1.8adventureworkscore

From production.product, list products with no color recorded and a list price above 0. Show name, productnumber, listprice, priciest first. State out loud (or in a comment) why color != 'Black' would not find these rows.

solution
SELECT name, productnumber, listprice
FROM production.product
WHERE color IS NULL
  AND listprice > 0
ORDER BY listprice DESC;

color != 'Black' evaluates to NULL (not true) when color is NULL, so those rows are dropped by WHERE. NULL rows only match IS NULL / IS NOT NULL / IS DISTINCT FROM.

Practice - Session C (problems 9-12)

P1.9stackoverflowcore

Return the 10 highest-scoring questions (not answers) created in 2018. Show title, score, view_count, creation_date.

hint
post_type_id = 1 means question. For "created in 2018", a date range beats extracting the year (Module 8 explains why).
solution
SELECT title, score, view_count, creation_date
FROM posts
WHERE post_type_id = 1
  AND creation_date >= DATE '2018-01-01'
  AND creation_date <  DATE '2019-01-01'
ORDER BY score DESC
LIMIT 10;

The half-open range [Jan 1, next Jan 1) is the professional idiom: it handles timestamps correctly (Dec 31 23:59:59 is included) and lets an index on creation_date do its job, unlike EXTRACT(YEAR ...) = 2018.

P1.10northwindinterview

From order_details, compute line revenue (unit_price * quantity * (1 - discount)) and return the 20 most valuable lines with order_id, product_id, quantity, discount, and the computed revenue rounded to 2 decimals. Then answer: why can you not write WHERE revenue > 1000 using the alias?

solution
SELECT order_id, product_id, quantity, discount,
       ROUND((unit_price * quantity * (1 - discount))::numeric, 2) AS revenue
FROM order_details
ORDER BY unit_price * quantity * (1 - discount) DESC
LIMIT 20;

WHERE is evaluated before SELECT in the logical order of execution, so the alias does not exist yet at WHERE time. ORDER BY runs after SELECT, which is why the alias would work there. This ordering question is a top-5 interview screener - Module 2 covers the full sequence.

P1.11chinookinterview

Find tracks where the composer is unknown (NULL) or the composer field contains the word "Bach" in any casing. Show track name and composer. How many rows does each branch of the OR contribute? (Answer with two extra COUNT queries.)

solution
SELECT name, composer
FROM track
WHERE composer IS NULL
   OR composer ILIKE '%bach%';

-- branch counts:
SELECT COUNT(*) FROM track WHERE composer IS NULL;
SELECT COUNT(*) FROM track WHERE composer ILIKE '%bach%';

The two branches cannot overlap (a NULL composer cannot also match ILIKE), so the branch counts sum to the total - a sanity check worth narrating.

P1.12nyctaxiinterview

Pull the 50 most recent trips that paid by payment_type 1 with a tip of exactly 0. Then explain: if tip_amount were nullable, what rows might this query miss, and how would you include them?

solution
SELECT trip_id, pickup_datetime, fare_amount, tip_amount
FROM trips
WHERE payment_type = 1
  AND tip_amount = 0
ORDER BY pickup_datetime DESC
LIMIT 50;

If tip_amount could be NULL, tip_amount = 0 misses NULL rows (unknown tip is not the same as zero tip - and saying that distinction out loud is the senior move). To treat unknown as zero: COALESCE(tip_amount, 0) = 0, while noting it may hide data-quality issues.
