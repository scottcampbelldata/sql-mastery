Data Modeling

Querying data and shaping data for a dashboard are different senior skills. A BI tool like Power BI or Tableau does not want your normalized OLTP tables - it wants a star schema: one fact table of measurable events surrounded by dimension tables of descriptive attributes, or a single wide flat table it can slice without joins. This module is that translation, written entirely as SQL you already know: chinook is your normalized store to model into a star, and Stack Overflow is your denormalization-at-scale example. Along the way we walk the normal forms 1NF to 3NF on real tables, weigh surrogate against natural and composite keys, and prove that playlist_track is a genuine many-to-many bridge - the modeling fundamentals every design round tests.

9.1 Normalized vs. star: two shapes, two jobs

chinook is normalized (third normal form): every fact lives in exactly one place, so invoice_line stores a track_id, not the track's name, album, artist, and genre. That is perfect for an app that writes data - no duplication, no update anomalies. It is painful for analysis, where "revenue by artist by month" needs five joins every single time.

A star schema flips the tradeoff for reads:

  • Fact table - one row per business event, holding foreign keys plus numeric measures. Here: one row per invoice line, with quantity and line_revenue.
  • Dimension tables - the descriptive "by what" attributes you filter and group on: customer, track, date. Wide, textual, relatively small.
  • Grain - the single most important decision: exactly what one fact row means. Get this wrong and every measure is wrong.
Star vs. snowflake A star flattens each dimension into one table (dim_track already carries album, artist, and genre names). A snowflake leaves those normalized into sub-tables (track -> album -> artist). Stars win for BI: fewer joins, simpler for the tool and the analyst. chinook as shipped is essentially a snowflake; your job in this module is to collapse it into a star.

9.2 Pick the grain first

Before a single JOIN, name the grain out loud. For sales analytics the natural grain is the invoice line: one row per track sold on an invoice. That grain supports revenue, units, average selling price, and every "by dimension" rollup. Choosing the invoice (order) grain instead would lose per-track detail; choosing the customer grain would lose everything time-based. State it: "one row per invoice line; measures are quantity and line revenue; dimensions are date, customer, and track."

The additivity trap Measures are not all equal. quantity and line_revenue are fully additive - they sum correctly across every dimension. unit_price is non-additive - summing prices is meaningless; you average it. A ratio like margin percent is non-additive too - you recompute it from summed numerator and denominator, never average the percentages. Knowing which measures are additive, semi-additive (balances over time), and non-additive is a senior modeling signal.

9.3 Build the fact table

The fact table is a SELECT that pins the grain, carries the dimension keys, and computes the measures. This is the query a nightly job would run to (re)build fact_sales:

SELECT il.invoice_line_id                         AS sales_key,     -- the fact's own key
       i.invoice_date::date                       AS date_key,      -- FK to dim_date
       i.customer_id                              AS customer_key,  -- FK to dim_customer
       il.track_id                                AS track_key,     -- FK to dim_track
       il.quantity,                                             -- additive measure
       il.unit_price,                                           -- non-additive (average it)
       il.unit_price * il.quantity                AS line_revenue   -- additive measure
FROM invoice_line il
JOIN invoice i ON i.invoice_id = il.invoice_id
ORDER BY il.invoice_line_id
LIMIT 50;

Notice the fact stays narrow: keys and numbers only, no descriptive text. The names live in the dimensions, which is what keeps a fact table small even at a billion rows.

9.4 Build the dimensions

dim_track collapses the track -> album -> artist -> genre snowflake into one flat row per track. LEFT JOINs matter here: some tracks have no album or no genre, and a dimension must never drop a key the fact references.

SELECT t.track_id                AS track_key,
       t.name                    AS track_name,
       al.title                  AS album_title,
       ar.name                   AS artist_name,
       COALESCE(g.name, 'Unknown') AS genre_name
FROM track t
LEFT JOIN album al  ON al.album_id = t.album_id
LEFT JOIN artist ar ON ar.artist_id = al.artist_id
LEFT JOIN genre g   ON g.genre_id = t.genre_id
ORDER BY t.track_id
LIMIT 50;

A date dimension is the one dimension you manufacture rather than harvest, with generate_series. Every calendar attribute you might group by lives here once, so the fact only stores a date and the reports get weekday, month name, and quarter for free.

SELECT d::date                          AS date_key,
       EXTRACT(YEAR  FROM d)::int         AS yr,
       EXTRACT(MONTH FROM d)::int         AS mo,
       TO_CHAR(d, 'Mon')                  AS month_name,
       EXTRACT(QUARTER FROM d)::int       AS quarter,
       (EXTRACT(DOW FROM d) IN (0, 6))   AS is_weekend
FROM generate_series(DATE '2021-01-01', DATE '2025-12-31', INTERVAL '1 day') d
LIMIT 20;
Surrogate keys and slowly changing dimensions Production dimensions use a surrogate key (a meaningless auto-number) instead of the source's natural id, so history survives when a source id is reused. That enables SCD Type 2: when a customer's country changes you insert a new dimension row with valid-from/valid-to dates rather than overwriting, so old facts still join to the country that was true at the time. You will not build this on a read-only teaching DB, but naming "surrogate key" and "SCD Type 2" answers the modeling interview question directly.

9.5 The semantic layer: one wide table for the BI tool

Many BI setups skip the explicit star and hand the tool a single wide flat table: the fact with every dimension already joined in. It is the star, pre-joined. On chinook's small data this is the pragmatic choice - one view, zero joins for the analyst:

-- Run locally you would wrap this in CREATE VIEW vw_sales AS ( ... );
-- the read-only teaching DB rejects DDL, so run the SELECT and picture the view.
SELECT il.invoice_line_id            AS sales_key,
       i.invoice_date::date          AS sale_date,
       c.country                     AS customer_country,
       ar.name                       AS artist,
       COALESCE(g.name, 'Unknown')     AS genre,
       il.quantity,
       il.unit_price * il.quantity   AS line_revenue
FROM invoice_line il
JOIN invoice i    ON i.invoice_id = il.invoice_id
JOIN customer c   ON c.customer_id = i.customer_id
JOIN track t      ON t.track_id = il.track_id
LEFT JOIN album al  ON al.album_id = t.album_id
LEFT JOIN artist ar ON ar.artist_id = al.artist_id
LEFT JOIN genre g   ON g.genre_id = t.genre_id
ORDER BY sale_date, sales_key
LIMIT 50;
Wide table vs. true star "Would you hand the BI tool a wide table or a real star schema?" Wide flat table: simplest, fastest for a single tool on modest data, but it duplicates dimension text on every fact row and there is no shared place to fix an attribute. True star: no duplication, dimensions reusable across many facts, the standard at scale. Say the tradeoff and pick based on data size and how many fact tables share the dimensions - that judgment is the answer they want.

9.6 Denormalization at scale (Stack Overflow)

Stack Overflow is the other direction. posts is already wide and denormalized - it even caches answercount and favoritecount so the site never recomputes them. At a quarter-million posts you pre-join the few things a dashboard always needs (the author's reputation, the primary tag) into one analytics row, so the dashboard reads one table instead of re-joining users on every load:

SELECT p.id                       AS post_key,
       p.creationdate::date        AS created,
       p.score,
       p.answercount,
       u.id                        AS author_key,
       u.reputation                AS author_reputation,
       SPLIT_PART(TRIM(BOTH '|' FROM p.tags), '|', 1) AS primary_tag
FROM posts p
JOIN users u ON u.id = p.owneruserid
WHERE p.posttypeid = 1
LIMIT 50;

The tradeoff is the mirror of normalization: reads get faster and simpler, but the reputation is now a copy that goes stale until the next rebuild. Senior answer: denormalize deliberately, for a known read pattern, and own the refresh.

9.7 The talking points that round out a modeling answer

  • Normal forms in one breath: 1NF = atomic values (no comma-lists in a cell), 2NF/3NF = every non-key column depends on the whole key and nothing but the key. OLTP aims for 3NF; analytics deliberately breaks it.
  • Fact types: transaction (one row per event, what we built), periodic snapshot (one row per entity per period, e.g. daily balances), accumulating snapshot (one row per process, updated as it moves through stages).
  • Conformed dimensions: one dim_customer shared by sales, support, and web facts so "customer" means the same thing everywhere. This is the whole point of a star over a wide table.
  • Grain, grain, grain: if you can only say one sentence about a model, say its grain. Every measure and every join correctness follows from it.

9.8 Normalization walk: 1NF to 2NF to 3NF

Normalization is the discipline chinook already follows. Walk it in three steps on a concrete before/after, imagining one denormalized sales_flat table that a beginner might build and fixing it form by form.

1NF - atomic values, no repeating groups. A cell holds one value; a row holds one thing. A column like tags = 'sql|performance|index' or track_names = 'A, B, C' breaks 1NF. Stack Overflow's posts.tags is exactly this violation, kept on purpose for storage - you can see the packed pipe-list and count how many values each cell crams in:

SELECT id,
       tags,
       (LENGTH(TRIM(BOTH '|' FROM tags))
        - LENGTH(REPLACE(TRIM(BOTH '|' FROM tags), '|', '')) + 1) AS tag_count
FROM posts
WHERE posttypeid = 1 AND tags IS NOT NULL
ORDER BY tag_count DESC, id
LIMIT 10;

The top row packs seven tags into one column. To reach 1NF you split that into a post_tag(post_id, tag) bridge - one row per pairing - which is the same shape as chinook's playlist_track. chinook is already in 1NF: no comma-lists anywhere.

2NF - no partial dependencies on part of a composite key. This bites only when the primary key is composite. Imagine keying a fact on (playlist_id, track_id) and also storing playlist_name in it: the name depends on playlist_id alone, not the whole key, so it repeats on every track in the playlist and drifts. The fix is to move playlist_name out to the playlist table keyed by playlist_id - which is precisely how chinook ships. playlist_track holds only the two keys and nothing else.

3NF - no transitive dependencies; non-key columns depend on the key, the whole key, and nothing but the key. The classic violation is storing artist_name on track: a track's artist is determined by its album, and the album determines the artist, so artist_name depends on the key through album_id, not directly. chinook resolves this into the snowflake chain - the artist name lives once on artist, reached track -> album -> artist:

SELECT t.track_id,
       t.name        AS track_name,
       t.album_id,
       al.title      AS album_title,
       al.artist_id,
       ar.name       AS artist_name
FROM track t
JOIN album al  ON al.album_id = t.album_id
JOIN artist ar ON ar.artist_id = al.artist_id
WHERE t.track_id = 1;
Why it matters This is the exact chain you collapse when you build dim_track. 3NF is the shape that makes writes safe: change AC/DC's name in one row and every track follows. The star schema deliberately undoes 3NF for read speed. Knowing you are trading one for the other on purpose - and being able to name which form a design violates - is the difference between a junior and a senior modeling answer.

9.9 Surrogate, natural, and composite keys

Every dimension needs a key. Three kinds show up in interviews, and chinook has a live example of each.

A natural key is a real-world attribute that is already unique - like genre.name. You can prove a column is a valid natural key by checking that the row count equals the distinct count:

SELECT COUNT(*)                AS rows,
       COUNT(DISTINCT name)   AS distinct_names
FROM genre;

Both come back 25, so genre.name genuinely identifies a genre. Now try the same test on a column that looks unique but is not - playlist.name:

SELECT name, COUNT(*) AS n
FROM playlist
GROUP BY name
HAVING COUNT(*) > 1
ORDER BY n DESC;

"Music", "Movies", "Audiobooks", and "TV Shows" each appear twice. That is why chinook keys playlists on a surrogate key - the meaningless auto-number playlist_id - instead of the name. A surrogate key never collides, never changes, and survives a rename; it is the default choice for a warehouse dimension for exactly these reasons.

A composite key is two or more columns that are unique only together. playlist_track is keyed on (playlist_id, track_id): neither column is unique alone, but the pair is. Prove it by comparing total rows to distinct pairs:

SELECT COUNT(*)                              AS total_rows,
       COUNT(DISTINCT (playlist_id, track_id)) AS distinct_pairs
FROM playlist_track;
Why it matters Both come back 8715 - equal counts mean the pair is unique with no accidental duplicate pairings, so (playlist_id, track_id) is a valid composite key. This exact "rows vs. distinct" pattern is how you audit any candidate key on real data before you trust it as a join column or a dimension's identity. The senior instinct: default to a surrogate key for dimensions, use a composite natural key for a bridge, and always verify uniqueness on the live data rather than assuming it.

9.10 Many-to-many: the bridge table

Not everything is one-to-many. A track can sit on many playlists and a playlist holds many tracks - a many-to-many relationship, which no single foreign key can express. The resolution is a bridge table (also called a junction or associative table): playlist_track holds one row per (playlist, track) pairing and nothing else. You can prove the relationship is truly M:N by checking that both directions exceed one:

SELECT MAX(playlists_per_track) AS max_playlists_per_track,
       (SELECT MAX(tracks_per_playlist)
          FROM (SELECT COUNT(*) AS tracks_per_playlist
                  FROM playlist_track GROUP BY playlist_id) b) AS max_tracks_per_playlist
FROM (SELECT COUNT(*) AS playlists_per_track
         FROM playlist_track GROUP BY track_id) a;

The answer: a single track appears on as many as 5 playlists, and the largest playlist holds 3290 tracks. Both sides greater than one confirms many-to-many. To read the relationship you traverse the bridge with two joins - here, every playlist one specific track belongs to:

SELECT t.name AS track_name,
       p.playlist_id,
       p.name AS playlist_name
FROM playlist_track pt
JOIN track t    ON t.track_id = pt.track_id
JOIN playlist p ON p.playlist_id = pt.playlist_id
WHERE pt.track_id = 3499
ORDER BY p.playlist_id;
Why it matters In a star schema a bridge becomes a factless fact table (or a bridge dimension): it records that a relationship exists without carrying its own measures, and it is how you model many-to-many between a fact and a dimension - a customer with several accounts, a sale spanning several sales reps. Recognizing when you need a bridge, and keeping it to just the two keys, is the pattern the M:N interview question is fishing for.

P9.6chinookcore

Candidate-key audit: prove that track.name is not a valid natural key by listing the track names that occur more than once, most-duplicated first. Return the name and its occurrence count, top 10.

hint
GROUP BY name and keep only groups with COUNT(*) > 1 using HAVING. Order by the count descending.
solution
SELECT name, COUNT(*) AS occurrences
FROM track
GROUP BY name
HAVING COUNT(*) > 1
ORDER BY occurrences DESC, name
LIMIT 10;

Names like "The Trooper" and "Hallowed Be Thy Name" appear five times each (same song across live and studio albums), so name cannot identify a track. That is why chinook uses the surrogate track_id. This "GROUP BY ... HAVING COUNT(*) > 1" pattern is the fastest way to disqualify a natural key on real data.

Prove playlist_track is a genuine composite key: return the total row count alongside the count of distinct (playlist_id, track_id) pairs. Equal counts mean the pair is unique with no duplicate pairings.

hint
COUNT(*) for total rows, and COUNT(DISTINCT (playlist_id, track_id)) for unique pairs. Compare the two.
solution
SELECT COUNT(*)                              AS total_rows,
       COUNT(DISTINCT (playlist_id, track_id)) AS distinct_pairs
FROM playlist_track;

Both return 8715, so (playlist_id, track_id) is unique - a valid composite key with no accidental duplicate pairings. This "rows vs. distinct pairs" check is the standard audit before you trust a composite key as a join column or a bridge's identity.

P9.8chinookinterview

Demonstrate that the playlist-to-track relationship is many-to-many. In one row return the maximum number of playlists any single track belongs to, and the maximum number of tracks any single playlist holds. Both greater than one proves M:N.

hint
Count rows per track_id in a subquery and take the MAX; do the same per playlist_id in a second subquery. You do not need to join to track or playlist - the bridge alone answers it.
solution

       (SELECT MAX(tracks_per_playlist)
          FROM (SELECT COUNT(*) AS tracks_per_playlist

A track appears on up to 5 playlists and the largest playlist holds 3290 tracks. Being able to demonstrate M:N from the data, not just assert it, is the senior move.

P9.9chinookinterview

Traverse the bridge to answer a business question: which genres reach the most distinct playlists? Join playlist_track to track and genre, and return each genre name with its count of distinct playlists, highest first, top 10.

hint
The genre lives on track, so go playlist_track -> track -> genre. Use COUNT(DISTINCT playlist_id) so a genre appearing many times on one playlist still counts once.
solution
SELECT g.name AS genre_name,
       COUNT(DISTINCT pt.playlist_id) AS playlists
FROM playlist_track pt
JOIN track t ON t.track_id = pt.track_id
JOIN genre g ON g.genre_id = t.genre_id
GROUP BY g.name
ORDER BY playlists DESC, genre_name
LIMIT 10;

Classical reaches the most playlists (7). The DISTINCT inside the count is the correctness catch: without it, a genre that owns hundreds of tracks on one playlist would be over-counted. Resolving a many-to-many into a clean per-dimension rollup is the everyday payoff of understanding the bridge.

P9.10stackoverflowhard

Show the 1NF violation on real data: for questions, return the id, the raw tags value, and how many tags are packed into that single column, most-packed first. Top 10. This is the non-atomic cell you would normalize into a post_tag bridge.

hint
Tags are pipe-delimited with leading and trailing pipes. TRIM the outer pipes, then count separators: length before minus length after removing '|', plus one. Filter posttypeid = 1 and tags IS NOT NULL.
solution
SELECT id,
       tags,
       (LENGTH(TRIM(BOTH '|' FROM tags))

The top question crams seven tags into one cell - a textbook 1NF violation. Counting separators by the length-difference trick is the portable way to size a delimited list without splitting it.
