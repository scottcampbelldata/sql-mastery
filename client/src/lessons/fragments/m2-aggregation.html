-- module 2 / five sessions /

Aggregation

GROUP BY collapses rows into groups; aggregate functions summarize each group. This module also contains the single most-asked conceptual interview question in existence: the logical order of query execution. You will go deep on how COUNT and AVG treat NULLs, WHERE vs HAVING, conditional aggregation and rows-to-columns pivots, multi-level subtotals with GROUPING SETS and ROLLUP, rate metrics, and list-building with STRING_AGG. Practice runs on chinook invoices - small enough to eyeball every total - and on stackoverflow posts, votes, and badges, where a group can hold a quarter-million rows.

2.1 The logical order of execution (memorize this)

-- Written order:          Logical execution order:
-- SELECT                  1. FROM / JOIN     (build the working set)
-- FROM / JOIN             2. WHERE           (filter individual rows)
-- WHERE                   3. GROUP BY        (collapse into groups)
-- GROUP BY                4. HAVING          (filter whole groups)
-- HAVING                  5. SELECT          (compute output columns)
-- ORDER BY                6. DISTINCT        (dedup the output)
-- LIMIT                   7. ORDER BY        (sort the output)
--                         8. LIMIT / OFFSET  (trim the output)

Every "why can't I..." question is answered by this list. Why can't WHERE see a SELECT alias? WHERE runs first. Why can't WHERE filter on SUM()? Groups do not exist yet at WHERE time - that is HAVING's job. Why can ORDER BY use an alias? It runs after SELECT.

2.2 The aggregate functions and their NULL behavior

-- chinook: one row summarizing every invoice
SELECT
    COUNT(*)                   AS all_rows,        -- counts rows, NULLs included
    COUNT(billing_state)       AS rows_with_state, -- skips NULLs (only 210 of 412 have a state)
    COUNT(DISTINCT customer_id) AS customers,       -- unique non-NULL values
    SUM(total)                 AS total_revenue,   -- ignores NULLs
    AVG(total)                 AS avg_invoice,     -- ignores NULLs in the denominator too!
    MIN(invoice_date)          AS first_invoice,
    MAX(invoice_date)          AS last_invoice
FROM invoice;
The AVG trap AVG(col) divides by the count of non-NULL values. If you want NULLs to count as zero, you must say so: AVG(COALESCE(col, 0)). "Which average does the business want?" is a genuine senior clarifying question.

2.3 GROUP BY and HAVING

-- chinook: revenue by billing country, big markets only
SELECT billing_country,
       COUNT(*)          AS invoice_count,
       SUM(total)        AS revenue,
       ROUND(AVG(total), 2) AS avg_invoice
FROM invoice
GROUP BY billing_country
HAVING SUM(total) > 100
ORDER BY revenue DESC;

Rule: every column in SELECT must either be inside an aggregate or listed in GROUP BY. Postgres enforces this; interviewers test it.

-- WHERE and HAVING together: filter rows first, then filter groups
-- stackoverflow: among 2013 posts only, which authors posted more than 50 times?
SELECT owneruserid, COUNT(*) AS posts_2013
FROM posts
WHERE creationdate >= DATE '2013-01-01'
  AND creationdate <  DATE '2014-01-01'
  AND owneruserid IS NOT NULL
GROUP BY owneruserid
HAVING COUNT(*) > 50
ORDER BY posts_2013 DESC;

WHERE narrows to 2013 rows before any grouping; HAVING then keeps only the busy authors. Swap the two and you get an error - you cannot filter a group that does not exist yet.

2.4 Conditional aggregation with FILTER

Counting different things in one pass is a constant real-world need. Postgres has the cleanest syntax anywhere:

-- stackoverflow: questions vs answers per user, one scan of posts
SELECT owneruserid,
       COUNT(*) FILTER (WHERE posttypeid = 1) AS questions,
       COUNT(*) FILTER (WHERE posttypeid = 2) AS answers
FROM posts
WHERE owneruserid IS NOT NULL
GROUP BY owneruserid
ORDER BY answers DESC
LIMIT 20;

-- Portable equivalent (know both - many interviews assume this form):
SUM(CASE WHEN posttypeid = 1 THEN 1 ELSE 0 END) AS questions

Recall the codes from Module 0: posttypeid 1 = question, 2 = answer. The top of this list is user 1876, the most prolific answerer in the dump.

2.5 Rows to columns: the conditional-aggregation pivot

Extend FILTER one step further and you have a pivot: turn category values that live in rows into named columns. Here each year of chinook revenue becomes its own column, one row per country.

-- chinook: revenue per country, one column per year (2023-2025) plus an all-years total
SELECT billing_country,
       ROUND(SUM(total) FILTER (WHERE EXTRACT(YEAR FROM invoice_date) = 2023)::numeric, 2) AS y2023,
       ROUND(SUM(total) FILTER (WHERE EXTRACT(YEAR FROM invoice_date) = 2024)::numeric, 2) AS y2024,
       ROUND(SUM(total) FILTER (WHERE EXTRACT(YEAR FROM invoice_date) = 2025)::numeric, 2) AS y2025,
       ROUND(SUM(total)::numeric, 2) AS all_years
FROM invoice
GROUP BY billing_country
ORDER BY all_years DESC
LIMIT 6;

why it matters This is the shape every "put the months across the top" report wants, and it is the manual version of what a BI tool's pivot control does. Note a year with no invoices for a country returns NULL, not 0 - SUM over zero matching rows is NULL. Wrap the whole term in COALESCE(..., 0) if the report needs a hard zero. The portable form swaps each FILTER for SUM(CASE WHEN ... THEN total END).

2.6 Multi-level totals: GROUPING SETS, ROLLUP, CUBE

Often you want detail rows and subtotals in one result set. Running the query twice and stapling the outputs together is the beginner move; GROUPING SETS asks for several grainularities at once in a single pass.

-- chinook: per-country detail plus a grand-total row, from one scan
SELECT billing_country,
       COUNT(*)               AS invoices,
       SUM(total)             AS revenue
FROM invoice
GROUP BY GROUPING SETS ((billing_country), ());

The empty set () is the grand total: it groups every row together, so billing_country comes back NULL on that row. ROLLUP (a, b) is shorthand for the hierarchy (a,b), (a), () - detail, per-a subtotal, grand total - and CUBE (a, b) gives every combination (a,b), (a), (b), ().

-- Label the subtotal row with GROUPING(): 1 means "this column was rolled up"
SELECT CASE WHEN GROUPING(billing_country) = 1
            THEN '(all countries)' ELSE billing_country END AS country,
       COUNT(*)               AS invoices,
       ROUND(SUM(total)::numeric, 2) AS revenue
FROM invoice
WHERE billing_country IN ('USA', 'Canada', 'France')
GROUP BY ROLLUP (billing_country)
ORDER BY GROUPING(billing_country), revenue DESC;

why it matters A NULL in a subtotal row is ambiguous - is it "rolled up" or a real NULL value in the data? GROUPING(col) returns 1 only when the column was collapsed for that row, which is exactly how you both label the total and sort it to the bottom. This is a frequent senior-screen follow-up: "how do you tell a subtotal apart from a genuine NULL?"

2.7 Rate and share metrics: AVG over a boolean cast to int

"What fraction of rows satisfy a condition?" is a rate. The neat trick: cast the boolean to an integer (true becomes 1, false becomes 0) and take AVG - the mean of a 0/1 column is the proportion of 1s.

-- stackoverflow: share of each year's questions that got an accepted answer
SELECT EXTRACT(YEAR FROM creationdate)::int AS yr,
       COUNT(*) AS questions,
       ROUND(AVG((acceptedanswerid IS NOT NULL)::int) * 100, 1) AS pct_accepted
FROM posts
WHERE posttypeid = 1
GROUP BY 1
ORDER BY yr;

why it matters The acceptance rate falls from roughly 68% in the early years to under 30% by 2024 - a real trend you would call out loud. Because false casts to 0 and true to 1, AVG(bool::int), SUM(bool::int)::numeric / COUNT(*), and COUNT(*) FILTER (WHERE bool) / COUNT(*)::numeric are three spellings of the same rate. AVG is the shortest, but watch NULLs: a NULL boolean casts to NULL and drops out of the AVG denominator, so decide whether an unknown should count as a "no" first.

2.8 Collapsing a group into a list: STRING_AGG and ARRAY_AGG

Sometimes the summary you want is not a number but the members themselves, gathered into one cell. STRING_AGG joins them into a delimited string; ARRAY_AGG keeps them as a real array. Both take an ORDER BY so the list is deterministic.

-- chinook: every album an artist released, as one ordered, semicolon-joined list
SELECT ar.name AS artist,
       COUNT(*) AS albums,
       STRING_AGG(al.title, '; ' ORDER BY al.title) AS album_list
FROM artist ar
JOIN album al ON al.artist_id = ar.artist_id
GROUP BY ar.name
HAVING COUNT(*) >= 3
ORDER BY albums DESC
LIMIT 5;

why it matters This is how you produce "top 5 tags per user" or "all SKUs in this order" without a second round-trip. ORDER BY inside the aggregate is what makes the output reproducible - without it the list order is undefined. Add DISTINCT (STRING_AGG(DISTINCT mt.name, ', ')) to dedupe. Reach for ARRAY_AGG when the caller is code that wants a real list to iterate, and STRING_AGG when a human will read it.

Practice - Session A (problems 1-3)

P2.1chinookwarmup

How many customers, how many distinct countries, and how many customers have no company on file? One query, one row.

solution
SELECT COUNT(*)                    AS customers,
       COUNT(DISTINCT country)     AS countries,
       COUNT(*) - COUNT(company)  AS no_company
FROM customer;

The COUNT(*) - COUNT(col) idiom for NULL counting comes up in nearly every data-quality discussion. Here 49 of the 59 customers are individuals with no company recorded - exactly the kind of "how complete is this column?" number an interviewer wants you to reach for without thinking.

P2.2chinookwarmup

Average, minimum, and maximum invoice total per billing_country, only for countries with at least 10 invoices, highest average first. Round the average to 2 decimals.

solution
SELECT billing_country,
       COUNT(*)            AS invoices,
       ROUND(AVG(total), 2) AS avg_total,
       MIN(total)          AS min_total,
       MAX(total)          AS max_total
FROM invoice
GROUP BY billing_country
HAVING COUNT(*) >= 10
ORDER BY avg_total DESC;

MIN/MAX/AVG in one pass over each country's invoices - the standard "distribution per group" shape. Note how tight chinook's invoice totals are: everything lands between roughly one and twenty-six dollars, because a single invoice is just a handful of 99-cent tracks.

P2.3stackoverflowcore

Per posttypeid (1 = question, 2 = answer): post count, average score, average view count, and the average score-per-view as a percentage (viewcount > 0 only, to avoid division problems). Round money-like numbers to 2 decimals and the percentage to 3.

hint
The average of per-post ratios and the ratio of averages are different numbers. Compute the per-post percentage inside AVG. Filter viewcount > 0 in WHERE, and cast to ::numeric so the division is not integer division.
solution
SELECT posttypeid,
       COUNT(*)                              AS posts,
       ROUND(AVG(score)::numeric, 2)          AS avg_score,
       ROUND(AVG(viewcount)::numeric, 1)      AS avg_views,
       ROUND(AVG(score::numeric / viewcount) * 100, 3) AS avg_score_per_view_pct
FROM posts
WHERE viewcount > 0
GROUP BY posttypeid
ORDER BY posts DESC;

Interview follow-up you should expect: "why did you filter viewcount > 0?" Answer: to prevent division by zero and to exclude un-viewed posts that would distort the ratio - and I would confirm with the business whether those posts should be excluded from the counts too. (In this dump answers carry no viewcount, so the surviving rows are essentially the questions - a good reminder to check what a filter silently removes.)

Practice - Session B (problems 4-6)

P2.4chinookcore

Using invoice_line only: total revenue per invoice (revenue = unit_price * quantity - there is no discount column in chinook). Show invoice_id, number of line items, total quantity, and total revenue rounded to 2 decimals. Only invoices with revenue over 20, richest first.

solution
SELECT invoice_id,
       COUNT(*)      AS line_count,
       SUM(quantity) AS total_qty,
       ROUND(SUM(unit_price * quantity)::numeric, 2) AS revenue
FROM invoice_line
GROUP BY invoice_id
HAVING SUM(unit_price * quantity) > 20
ORDER BY revenue DESC;

Note HAVING repeats the expression rather than using the alias - HAVING runs before SELECT, so the alias does not exist yet. (Postgres actually permits GROUP BY aliases as a convenience, but not HAVING aliases.) And note the revenue formula: chinook has no discount column, so it is just unit_price * quantity - always read the schema before assuming a discount term exists.

P2.5stackoverflowcore

Which badge names have been awarded more than 1,000 times, and to how many distinct users? Most-awarded first.

solution
SELECT name,
       COUNT(*)              AS times_awarded,
       COUNT(DISTINCT userid) AS distinct_users
FROM badges
GROUP BY name
HAVING COUNT(*) > 1000
ORDER BY times_awarded DESC;

If times_awarded is much larger than distinct_users for a badge, it is a repeatable badge - "Popular Question" and "Notable Question" can land on the same user many times, whereas "Autobiographer" is once per person. That gap between the two counts is the kind of observation a senior volunteers while the query is still running.

P2.6chinookcore

Per genre_id in track: track count, average length in minutes (1 decimal), and total catalog value (SUM of unit_price). Only genres with 50+ tracks. Why might genre_id be NULL for some tracks, and what happens to those rows here?

solution
SELECT genre_id,
       COUNT(*)                                AS tracks,
       ROUND(AVG(milliseconds) / 60000.0, 1)  AS avg_minutes,
       SUM(unit_price)                         AS catalog_value
FROM track
GROUP BY genre_id
HAVING COUNT(*) >= 50
ORDER BY tracks DESC;

GROUP BY treats all NULLs as one group, so any unclassified tracks would appear as a single genre_id = NULL row (if 50+ of them existed). That NULLs group together in GROUP BY - but never equal each other in joins - is a classic conceptual question.

Practice - Session C (problems 7-9)

P2.7stackoverflowinterview

One row per year of posts.creationdate: total posts, questions, answers, and the answer-to-question ratio (2 decimals). Use FILTER. Chronological order.

hint
Group by EXTRACT(YEAR FROM creationdate) or date_trunc('year', creationdate). Guard the ratio against zero questions with NULLIF.
solution
SELECT EXTRACT(YEAR FROM creationdate)::int AS yr,
       COUNT(*) AS posts,
       COUNT(*) FILTER (WHERE posttypeid = 1) AS questions,
       COUNT(*) FILTER (WHERE posttypeid = 2) AS answers,
       ROUND(
         COUNT(*) FILTER (WHERE posttypeid = 2)::numeric
         / NULLIF(COUNT(*) FILTER (WHERE posttypeid = 1), 0), 2) AS ans_per_q
FROM posts
GROUP BY 1
ORDER BY yr;

Three interview staples in one: FILTER, NULLIF-guarded division, and the ::numeric cast to avoid integer division. The trend is real and worth a sentence out loud: the answer-to-question ratio drifts from roughly 3.9 in the early years toward about 1.0 by 2024 as the site matures. GROUP BY 1 (positional) is fine to use, just be able to name the tradeoff: shorter but fragile if the SELECT list is reordered.

P2.8stackoverflowinterview

From posts (questions and answers only): per posttypeid, count posts and split the total score into a positive bucket (score > 0) and a non-positive bucket (score <= 0) using conditional aggregation. Sort by total score descending.

solution
SELECT posttypeid,
       COUNT(*)   AS posts,
       SUM(score) AS total_score,
       SUM(score) FILTER (WHERE score > 0)  AS positive_score,
       SUM(score) FILTER (WHERE score <= 0) AS nonpositive_score
FROM posts
WHERE posttypeid IN (1, 2)
GROUP BY posttypeid
ORDER BY total_score DESC;

Splitting one measure into two mutually exclusive buckets with SUM(...) FILTER (WHERE ...) is the everyday reporting pattern - the online-vs-offline, new-vs-returning, region-A-vs-B split. The portable form is SUM(CASE WHEN score > 0 THEN score ELSE 0 END); know both, because FILTER is Postgres-only and some interviews assume the CASE version.

P2.9chinookinterview

The classic two-level aggregation: what is the average number of invoices per customer? (Not average invoice value - average invoice count.) Solve it with a subquery in FROM; you will redo it with a CTE in Module 5.

hint
Step 1: invoices per customer (GROUP BY customer_id). Step 2: AVG over those counts.
solution
SELECT ROUND(AVG(invoice_count), 2) AS avg_invoices_per_customer
FROM (
    SELECT customer_id, COUNT(*) AS invoice_count
    FROM invoice
    GROUP BY customer_id
) per_customer;

Clarifying question a senior asks first: "customers with zero invoices - do they count in the average?" This version excludes them (they have no rows in invoice). Including them requires starting from the customer table with a LEFT JOIN - Module 3. In chinook every customer has bought something, so the two answers coincide here, but you should still say the caveat.

Practice - Session D (problems 10-12)

P2.10stackoverflowinterview

Activity curve: for one week of data (pick any 7-day range in the 2008-2024 span), count posts per hour of day (0-23) and show the average score per hour. Which three hours have the highest average score?

solution
SELECT EXTRACT(HOUR FROM creationdate)::int AS hr,
       COUNT(*) AS posts,
       ROUND(AVG(score), 2) AS avg_score
FROM posts
WHERE creationdate >= '2016-06-01'
  AND creationdate <  '2016-06-08'
GROUP BY hr
ORDER BY avg_score DESC
LIMIT 3;

Adjust the date range to whatever window you like - the data runs 2008-09 to 2024-03 (find the exact bounds with SELECT MIN(creationdate), MAX(creationdate) FROM posts, itself a Module 0 move). Rerun without LIMIT and with ORDER BY hr to see the full 24-hour curve. Careful with one week of data: some hours hold only a handful of posts, so a single high-scoring post can dominate the average - a good moment to mention sample size out loud.

P2.11chinookinterview

Debug drill. A teammate wrote this and got an error. Fix it two different ways and explain the difference in results, if any:

/* BROKEN - this errors on purpose; the two fixes are below */
SELECT billing_country, billing_city, SUM(total) AS revenue
FROM invoice
WHERE SUM(total) > 50
GROUP BY billing_country;
solution

Two bugs: aggregates are illegal in WHERE (groups do not exist yet), and billing_city is selected without being grouped or aggregated.

-- Fix 1: group by both columns (city-level revenue)
SELECT billing_country, billing_city, SUM(total) AS revenue
FROM invoice
GROUP BY billing_country, billing_city
HAVING SUM(total) > 50;

-- Fix 2: drop the city (country-level revenue)
SELECT billing_country, SUM(total) AS revenue
FROM invoice
GROUP BY billing_country
HAVING SUM(total) > 50;

They answer different questions: Fix 1's grain is (country, city), Fix 2's grain is country. "What grain do you want?" is the clarifying question that resolves it. Debug-the-query rounds like this are common in senior screens.

P2.12stackoverflowhard

Distribution question: bucket users by reputation - under 100, 100-999, 1000-9999, 10000+ - and show the user count and percentage of all users in each bucket (1 decimal). One query.

hint
CASE builds the bucket (Module 4 previews it). For the percentage inside a grouped query, divide COUNT(*) by SUM(COUNT(*)) OVER () - or keep it simple with a subquery for the total.
solution
SELECT CASE
         WHEN reputation < 100   THEN 'a. under 100'
         WHEN reputation < 1000  THEN 'b. 100-999'
         WHEN reputation < 10000 THEN 'c. 1000-9999'
         ELSE 'd. 10000+'
       END AS bucket,
       COUNT(*) AS users,
       ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER (), 1) AS pct
FROM users
GROUP BY 1
ORDER BY bucket;

The letter prefixes make the buckets sort correctly as text - a small practical trick. SUM(COUNT(*)) OVER () is a window over the grouped result; if that feels like magic now, it will be obvious after Module 6. The subquery alternative: ROUND(100.0 * COUNT(*) / (SELECT COUNT(*) FROM users), 1). The shape is stark - well over half of these users sit under 100 reputation, and only a sliver clear 10,000.

Practice - Session E (problems 13-17)

P2.13chinookcore

Data-quality snapshot on track.composer: total tracks, tracks that name a composer, distinct composers, and tracks with the composer missing. One query, one row.

hint
COUNT(*) counts every row; COUNT(composer) skips NULLs; COUNT(DISTINCT composer) counts unique non-NULL values. The missing count is COUNT(*) - COUNT(composer).
solution
SELECT COUNT(*)                      AS all_tracks,
       COUNT(composer)              AS with_composer,
       COUNT(DISTINCT composer)     AS distinct_composers,
       COUNT(*) - COUNT(composer)   AS missing_composer
FROM track;

The three flavors of COUNT in one line are the whole point: 3503 tracks, 2526 name a composer, only 853 of those are distinct (composers repeat across an album), and 977 are blank. Interviewers love watching whether you reach for COUNT(col) vs COUNT(*) without being reminded that they differ on NULLs.

P2.14chinookinterview

Build a pivot: per genre_id (only genres with 100+ tracks), one column for MPEG tracks (media_type_id = 1), one for protected AAC (media_type_id = 2), one for everything else, and a total. Busiest genre first.

hint
Three COUNT(*) FILTER (WHERE ...) expressions, one per bucket, plus a plain COUNT(*). Put the 100-track floor in HAVING.
solution
SELECT genre_id,
       COUNT(*) FILTER (WHERE media_type_id = 1)          AS mpeg,
       COUNT(*) FILTER (WHERE media_type_id = 2)          AS aac_protected,
       COUNT(*) FILTER (WHERE media_type_id NOT IN (1, 2)) AS other,
       COUNT(*)                                          AS total
FROM track
GROUP BY genre_id
HAVING COUNT(*) >= 100
ORDER BY total DESC;

The buckets are mutually exclusive and cover every row, so the three columns sum to the total - a good self-check that your FILTER conditions did not overlap or leave a gap. Rock (genre_id 1) is the only genre carrying protected AAC files in quantity. The portable spelling is COUNT(*) FILTER (WHERE c) rewritten as SUM(CASE WHEN c THEN 1 ELSE 0 END).

P2.15stackoverflowinterview

Per year of creationdate, over questions only (posttypeid = 1): how many questions, and what share of them received at least one answer (answercount > 0)? Express the share as a percentage to 1 decimal. Chronological order.

hint
A share is the mean of a 0/1 flag. Cast the boolean answercount > 0 to int and take AVG(...) * 100. Group by the extracted year.
solution
SELECT EXTRACT(YEAR FROM creationdate)::int AS yr,
       COUNT(*) AS questions,
       ROUND(AVG((answercount > 0)::int) * 100, 1) AS pct_with_answer
FROM posts
WHERE posttypeid = 1
GROUP BY 1
ORDER BY yr;

The AVG(bool::int) rate idiom in its natural habitat. The share of answered questions slides from essentially 100% in the sparse early years to about 63% in 2024 - the same maturing-site story the acceptance rate told, from a different angle. Follow-up to expect: "what would change if answercount could be NULL?" A NULL flag would drop out of the AVG rather than count as a zero, quietly shrinking the denominator.

P2.16chinookcore

For every genre with 100+ tracks, show the genre name, its track count, and a single comma-joined, alphabetized list of the distinct media-type names that genre uses. Busiest genre first.

hint
Join track to genre and media_type for the names. Use STRING_AGG(DISTINCT mt.name, ', ' ORDER BY mt.name) - DISTINCT collapses repeats, ORDER BY makes the list deterministic.
solution
SELECT g.name AS genre,
       COUNT(*) AS tracks,
       STRING_AGG(DISTINCT mt.name, ', ' ORDER BY mt.name) AS media_types
FROM track t
JOIN genre g       ON g.genre_id = t.genre_id
JOIN media_type mt ON mt.media_type_id = t.media_type_id
GROUP BY g.name
HAVING COUNT(*) >= 100
ORDER BY tracks DESC;

STRING_AGG turns a whole group into one readable cell - here you can see at a glance that Rock spans three formats while Metal is MPEG-only. The DISTINCT ... ORDER BY pair matters: drop DISTINCT and every track repeats its format in the list; drop ORDER BY and the order is undefined between runs. (Joins arrive properly in Module 3 - this is a preview to make the aggregate worth reading.)

P2.17stackoverflowhard

For 2020 and 2021 only, and questions and answers only, produce a table with both the per-(year, posttypeid) counts and a per-year total row, in one query. Add a flag column that is 1 on the year-total rows. Order by year, with the total row last within each year.

hint
GROUPING SETS ((year, posttypeid), (year)) asks for both grains at once. GROUPING(posttypeid) is 1 on the rolled-up total rows; use it as the flag and in ORDER BY. Filter the date range and posttypeid IN (1, 2) in WHERE.
solution
SELECT EXTRACT(YEAR FROM creationdate)::int AS yr,
       posttypeid,
       GROUPING(posttypeid) AS is_year_total,
       COUNT(*) AS posts
FROM posts
WHERE creationdate >= DATE '2020-01-01'
  AND creationdate <  DATE '2022-01-01'
  AND posttypeid IN (1, 2)
GROUP BY GROUPING SETS (
         (EXTRACT(YEAR FROM creationdate), posttypeid),
         (EXTRACT(YEAR FROM creationdate))
       )
ORDER BY yr, posttypeid NULLS LAST;

The total rows come back with posttypeid NULL and is_year_total = 1; each year's total equals its question and answer counts summed. NULLS LAST in the ORDER BY is what parks the subtotal beneath its detail rows. This is the multi-grain report interviewers reach for to see whether you know GROUPING SETS exists or would instead UNION ALL two separately-grouped queries (correct, but more code and two scans).
