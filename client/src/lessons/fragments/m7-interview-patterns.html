Interview Patterns

Senior interviews recycle about a dozen named patterns in different costumes. Learn to hear the costume and name the pattern - "that's Nth-highest by value," "that's a self-join on consecutive periods" - and the rest is typing. Each pattern below has the trigger phrases, the shape, and a worked example on your data: the small clean chinook store where the result fits on one screen, the large stackoverflow dump where the pattern earns its keep at scale.

Pattern 1: Deduplication

Trigger phrases: "remove duplicates," "keep the latest record per," "the table has multiple rows per user."

-- Shape: ROW_NUMBER over the duplicate key, ordered by the keep-rule, filter rn = 1.
-- stackoverflow flavor: one row per (user, badge name) = their MOST RECENT award of it.
-- Scoped to one prolific user here so the demo is readable; drop the WHERE for the whole table.
WITH ranked AS (
    SELECT userid, name, date,
           ROW_NUMBER() OVER (PARTITION BY userid, name
                              ORDER BY date DESC, id DESC) AS rn
    FROM badges
    WHERE userid = 1876
)
SELECT userid, name, date FROM ranked WHERE rn = 1;

Always add a deterministic tiebreaker (id here). If asked to delete duplicates rather than select around them, the same ranking drives a DELETE with a USING or IN clause - describe it rather than run it in an interview.

Pattern 2: Nth highest

Trigger phrases: "second highest salary," "third most expensive," any ordinal.

-- The answer is DENSE_RANK (values), with ROW_NUMBER (rows) as the contrast:
WITH ranked AS (
    SELECT name, unit_price,
           DENSE_RANK() OVER (ORDER BY unit_price DESC) AS dr
    FROM track
)
SELECT * FROM ranked WHERE dr = 2;
-- Also know: OFFSET 1 LIMIT 1 after ORDER BY DESC (fails on ties, breaks with NULLs);
-- and MAX(x) WHERE x < (SELECT MAX(x)) (only reaches 2nd). Explain why windows win.

Pattern 3: Year-over-year / period-over-period

Trigger phrases: "compared to the same month last year," "growth rate," "MoM/YoY."

-- MoM = LAG(1). YoY on monthly data = LAG(12) - but ONLY if every month exists!
-- Missing months silently shift LAG(12) to the wrong row. The robust form is a self-join:
WITH monthly AS (
    SELECT date_trunc('month', invoice_date)::date AS month,
           SUM(total) AS revenue
    FROM invoice
    GROUP BY 1
)
SELECT cur.month,
       cur.revenue,
       prev.revenue AS revenue_last_year,
       ROUND(100.0 * (cur.revenue - prev.revenue) / NULLIF(prev.revenue, 0), 1) AS yoy_pct
FROM monthly cur
LEFT JOIN monthly prev ON prev.month = cur.month - INTERVAL '1 year'
ORDER BY cur.month;

Saying "LAG(12) assumes a gap-free series; the self-join on month minus one year is robust to gaps" is a top-tier senior answer.

Pattern 4: Cohort retention

Trigger phrases: "retention," "of customers who first bought in January, how many bought again in February," "cohort."

-- Shape: (1) each customer's cohort = first activity month, (2) all activity months,
-- (3) join and count distinct customers per (cohort, months_since).
-- chinook flavor: purchase retention by first-purchase month
WITH cohorts AS (
    SELECT customer_id,
           date_trunc('month', MIN(invoice_date))::date AS cohort_month
    FROM invoice
    GROUP BY customer_id
),
activity AS (
    SELECT DISTINCT customer_id,
           date_trunc('month', invoice_date)::date AS active_month
    FROM invoice
)
SELECT c.cohort_month,
       (EXTRACT(YEAR FROM a.active_month)  - EXTRACT(YEAR FROM c.cohort_month)) * 12
     + (EXTRACT(MONTH FROM a.active_month) - EXTRACT(MONTH FROM c.cohort_month)) AS months_since,
       COUNT(DISTINCT a.customer_id) AS active_customers
FROM cohorts c
JOIN activity a ON a.customer_id = c.customer_id
GROUP BY 1, 2
ORDER BY 1, 2;

Pattern 5: Consecutive events / gaps and islands

Trigger phrases: "3 days in a row," "consecutive," "streak," "continuous ranges."

-- The magic trick: for consecutive sequences, (value - ROW_NUMBER) is constant per island.
-- chinook flavor: streaks of consecutive calendar dates on which any invoice was issued
WITH days AS (
    SELECT DISTINCT invoice_date::date AS day FROM invoice
),
grouped AS (
    SELECT day,
           day - (ROW_NUMBER() OVER (ORDER BY day))::int AS island_key
    FROM days
)
SELECT MIN(day) AS streak_start,
       MAX(day) AS streak_end,
       COUNT(*)  AS streak_days
FROM grouped
GROUP BY island_key
ORDER BY streak_days DESC;

Why it works: along a run of consecutive dates, both the date and the row number increase by 1 per row, so their difference is constant - and it changes the moment a day is skipped. Walk the interviewer through that sentence; it is the whole trick.

Pattern 6: Pivoting with conditional aggregation

Trigger phrases: "one column per year," "reshape long to wide," "crosstab."

-- chinook: genre revenue by year, years as columns (line revenue = unit_price * quantity)
SELECT g.name AS genre,
  ROUND(SUM(il.unit_price * il.quantity) FILTER (WHERE i.invoice_date >= '2021-01-01' AND i.invoice_date < '2022-01-01')::numeric, 0) AS y2021,
  ROUND(SUM(il.unit_price * il.quantity) FILTER (WHERE i.invoice_date >= '2022-01-01' AND i.invoice_date < '2023-01-01')::numeric, 0) AS y2022,
  ROUND(SUM(il.unit_price * il.quantity) FILTER (WHERE i.invoice_date >= '2023-01-01' AND i.invoice_date < '2024-01-01')::numeric, 0) AS y2023
FROM invoice_line il
JOIN invoice i ON i.invoice_id = il.invoice_id
JOIN track t   ON t.track_id = il.track_id
JOIN genre g   ON g.genre_id = t.genre_id
GROUP BY g.name
ORDER BY g.name;

SQL cannot make columns dynamically - the column list is fixed at write time. If asked "what if the years are not known in advance," the answer is: generate the SQL programmatically or pivot in the BI layer. Knowing the boundary is the senior part.

Pattern 7: Sessionization

Trigger phrases: "group events into sessions," "a new session starts after 1 hour of inactivity."

-- Shape: LAG to measure the gap, CASE to flag session starts, running SUM of flags = session id.
-- stackoverflow flavor: a user's posting sessions (new session after 1 hour idle)
WITH events AS (
    SELECT owneruserid, creationdate,
           CASE WHEN creationdate - LAG(creationdate) OVER w > INTERVAL '1 hour'
                  OR LAG(creationdate) OVER w IS NULL
                THEN 1 ELSE 0 END AS new_session
    FROM posts
    WHERE owneruserid = 1876
    WINDOW w AS (PARTITION BY owneruserid ORDER BY creationdate)
)
SELECT *,
       SUM(new_session) OVER (PARTITION BY owneruserid
                              ORDER BY creationdate) AS session_id
FROM events;

Flag-then-running-sum is the same engine as gaps and islands; recognizing that they are one family is what makes both easy to reconstruct under pressure. (User 1876 is the most prolific answerer in this dump - a real, busy timeline to sessionize.)

Pattern 8: Dimensional modeling and the semantic layer

Trigger phrases: "how would you model this for a dashboard," "star schema," "fact vs dimension," "build a semantic layer," "one big table."

Recognize the shift from "answer this question" to "shape this data so a dashboard can answer it over and over." Chinook is normalized for clean writes, so the genre scorecard in P7.12 needed four joins just to attach a name to a sale - and a BI tool would re-run those joins on every click. The senior move is a star: a fact at one clear grain (one invoice line: keys plus additive measures) surrounded by flattened dimensions (a dim_track carrying album, artist, and genre), so "revenue by genre by month" becomes a two-table join that reads the way the business talks. Name the vocabulary in one breath - grain, fact, dimension, star, denormalize-for-reads - and note that a single wide "semantic layer" view (what a CREATE VIEW or dbt model wraps) is the pragmatic end state for one dashboard.

Build it in Module 9 This module is about recognizing the pattern. Module 9 - Data Modeling is where you build the fact table, the dimensions, and the wide semantic-layer query step by step with runnable drills. P7.13 below is the payoff: once the model exists, the business question collapses to a short GROUP BY.

Practice - thirteen problems, mixed costume

The prompts below deliberately do not name their pattern. Your first job on each: say which pattern it is before writing a line.

P7.1stackoverflowinterview

Some users appear to have duplicate badge awards (same user, same badge name, possibly different dates). Produce one row per (user, badge name) keeping the earliest award date, for badges named 'Nice Answer'.

hint
Pattern 1. The duplicate key is (userid, name); the keep-rule is earliest date.
solution
WITH ranked AS (
    SELECT userid, name, date,
           ROW_NUMBER() OVER (PARTITION BY userid, name
                              ORDER BY date, id) AS rn
    FROM badges
    WHERE name = 'Nice Answer'
)
SELECT userid, name, date FROM ranked WHERE rn = 1;

Alternative for "earliest date only": plain GROUP BY userid, name with MIN(date). ROW_NUMBER wins when you must keep the whole row, not just the date - say that distinction out loud.

P7.2chinookinterview

What is the third-highest invoice total, and which invoices hit it? All ties must appear.

solution
WITH ranked AS (
    SELECT invoice_id, customer_id, total,
           DENSE_RANK() OVER (ORDER BY total DESC) AS dr
    FROM invoice
)
SELECT * FROM ranked WHERE dr = 3;

"All ties must appear" + "third-highest value" = DENSE_RANK, stated before typing. That is the whole grade.

P7.3stackoverflowinterview

Weekly posting volume for the first quarter of 2016 with percent change from the prior week, and flag weeks that dropped more than 10%.

solution
WITH weekly AS (
    SELECT date_trunc('week', creationdate)::date AS week,
           COUNT(*) AS posts
    FROM posts
    WHERE creationdate >= '2016-01-01' AND creationdate < '2016-04-01'
    GROUP BY 1
),
compared AS (
    SELECT week, posts,
           ROUND(100.0 * (posts - LAG(posts) OVER (ORDER BY week))
                 / NULLIF(LAG(posts) OVER (ORDER BY week), 0), 1) AS wow_pct
    FROM weekly
)
SELECT *,
       CASE WHEN wow_pct < -10 THEN 'ALERT' END AS flag
FROM compared
ORDER BY week;

Note the second CTE exists so the CASE can reference wow_pct by name - you cannot filter or branch on a window in the same SELECT that defines it. Partial first and last weeks (the truncated week straddling the range boundary) will look like fake drops; mention trimming them.

P7.4chinookinterview

Of the customers who placed their first order in 2021, what percentage also placed at least one order in 2022? And in 2023?

hint
Pattern 4, simplified to one cohort. CTE for first-order year = 2021 customers; conditional EXISTS or FILTER for later-year activity.
solution
WITH cohort_2021 AS (
    SELECT customer_id
    FROM invoice
    GROUP BY customer_id
    HAVING EXTRACT(YEAR FROM MIN(invoice_date)) = 2021
)
SELECT COUNT(*) AS cohort_size,
       ROUND(100.0 * COUNT(*) FILTER (WHERE EXISTS (
           SELECT 1 FROM invoice i
           WHERE i.customer_id = c.customer_id
             AND i.invoice_date >= '2022-01-01' AND i.invoice_date < '2023-01-01'))
           / COUNT(*), 1) AS pct_active_2022,
       ROUND(100.0 * COUNT(*) FILTER (WHERE EXISTS (
           SELECT 1 FROM invoice i
           WHERE i.customer_id = c.customer_id
             AND i.invoice_date >= '2023-01-01' AND i.invoice_date < '2024-01-01'))
           / COUNT(*), 1) AS pct_active_2023
FROM cohort_2021 c;

Retention questions always hide a definition question: "active" = placed an order? logged in? Ask. Here the data only knows invoices, so state the assumption and proceed.

P7.5stackoverflowhard

Find users who posted on at least 5 consecutive calendar days at least once. Return the user, their longest streak, and its start and end dates. (Scope the set of users first so the table is manageable, e.g. reputation > 50000.)

hint
Pattern 5 with a partition: DISTINCT (user, day), then day - ROW_NUMBER() OVER (PARTITION BY user ORDER BY day) as the island key.
solution
WITH user_days AS (
    SELECT DISTINCT p.owneruserid, p.creationdate::date AS day
    FROM posts p
    JOIN users u ON u.id = p.owneruserid
    WHERE u.reputation > 50000
),
islands AS (
    SELECT owneruserid, day,
           day - (ROW_NUMBER() OVER (PARTITION BY owneruserid
                                     ORDER BY day))::int AS island_key
    FROM user_days
),
streaks AS (
    SELECT owneruserid,
           MIN(day) AS streak_start,
           MAX(day) AS streak_end,
           COUNT(*) AS streak_days
    FROM islands
    GROUP BY owneruserid, island_key
),
best AS (
    SELECT *,
           ROW_NUMBER() OVER (PARTITION BY owneruserid
                              ORDER BY streak_days DESC, streak_start) AS rn
    FROM streaks
    WHERE streak_days >= 5
)
SELECT owneruserid, streak_days, streak_start, streak_end
FROM best
WHERE rn = 1
ORDER BY streak_days DESC;

Four CTEs, each a named step, and the final one is Pattern 1 (top-1 per user) riding on top of Pattern 5. Compound patterns like this are exactly what a hard senior round looks like.

P7.6chinookinterview

Build a wide report: one row per billing country, with total revenue in columns for 2021, 2022, and 2023 (adjust to the years in your data - find them first).

solution
-- find the years first (Module 0 reflex):
SELECT MIN(invoice_date), MAX(invoice_date) FROM invoice;

SELECT billing_country,
  SUM(total) FILTER (WHERE EXTRACT(YEAR FROM invoice_date) = 2021) AS y2021,
  SUM(total) FILTER (WHERE EXTRACT(YEAR FROM invoice_date) = 2022) AS y2022,
  SUM(total) FILTER (WHERE EXTRACT(YEAR FROM invoice_date) = 2023) AS y2023
FROM invoice
GROUP BY billing_country
ORDER BY billing_country;

Countries with no sales in a year show NULL, not 0 - wrap in COALESCE if the report needs zeros. Small detail, real report bug.

P7.7chinookhard

Track-length distribution summary in one row: min, 25th percentile, median, 75th, 95th, 99th, max, and mean of milliseconds (positive durations only). Then: which single number would you report to a stakeholder as "typical track length," and why?

solution
SELECT
  MIN(milliseconds) AS min,
  PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY milliseconds) AS p25,
  PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY milliseconds) AS median,
  PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY milliseconds) AS p75,
  PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY milliseconds) AS p95,
  PERCENTILE_CONT(0.99) WITHIN GROUP (ORDER BY milliseconds) AS p99,
  MAX(milliseconds) AS max,
  ROUND(AVG(milliseconds)::numeric, 0) AS mean
FROM track
WHERE milliseconds > 0;

Median. Track lengths are right-skewed (a handful of 30-minute classical pieces and hour-long podcast tracks pull the tail), so the mean sits above what most tracks run. p95/p99 vs max also quantifies how extreme the longest tracks are - a data-shape statement in the same breath.

P7.8stackoverflowhard

Sessionize one busy user's comments with a 30-minute idle threshold: session id, session start, session end, comments in session. Then their average session length in minutes.

solution
WITH flagged AS (
    SELECT userid, creationdate,
           CASE WHEN creationdate - LAG(creationdate) OVER w > INTERVAL '30 minutes'
                  OR LAG(creationdate) OVER w IS NULL
                THEN 1 ELSE 0 END AS is_start
    FROM comments
    WHERE userid = 993   -- a very busy commenter in this dump
    WINDOW w AS (ORDER BY creationdate)
),
sessions AS (
    SELECT *,
           SUM(is_start) OVER (ORDER BY creationdate) AS session_id
    FROM flagged
),
summarized AS (
    SELECT session_id,
           MIN(creationdate) AS session_start,
           MAX(creationdate) AS session_end,
           COUNT(*) AS comments
    FROM sessions
    GROUP BY session_id
)
SELECT *,
       ROUND((EXTRACT(EPOCH FROM session_end - session_start) / 60)::numeric, 1) AS minutes
FROM summarized
ORDER BY session_id;
-- average session length: wrap once more, AVG(minutes).

Flag, running-sum, group - once you can produce these three steps from memory, every sessionization variant (30 min, 1 hour, per user, per post) is a parameter change.

P7.9stackoverflowhard

YoY by quarter, robustly: quarterly answer volume with same-quarter-last-year comparison using the self-join form (not LAG(4)). Include quarters where last year is missing, showing NULL.

solution
WITH quarterly AS (
    SELECT date_trunc('quarter', creationdate)::date AS qtr,
           COUNT(*) AS answers
    FROM posts
    WHERE posttypeid = 2
    GROUP BY 1
)
SELECT cur.qtr,
       cur.answers,
       prev.answers AS same_qtr_last_year,
       ROUND(100.0 * (cur.answers - prev.answers) / NULLIF(prev.answers, 0), 1) AS yoy_pct
FROM quarterly cur
LEFT JOIN quarterly prev
       ON prev.qtr = (cur.qtr - INTERVAL '1 year')::date
ORDER BY cur.qtr;

LEFT JOIN keeps year-one quarters with NULL comparison instead of dropping them - which is honest reporting. The cast on the join condition keeps both sides date-typed.

P7.10chinookhard

Share-shift analysis: each genre's share of total revenue in 2022 vs 2021, and the change in percentage points. Which genres gained share? (Combines pivot, percent-of-total, and a derived comparison.)

solution
WITH genre_year AS (
    SELECT g.name AS genre,
           EXTRACT(YEAR FROM i.invoice_date)::int AS yr,
           SUM(il.unit_price * il.quantity) AS revenue
    FROM invoice_line il
    JOIN invoice i ON i.invoice_id = il.invoice_id
    JOIN track t   ON t.track_id = il.track_id
    JOIN genre g   ON g.genre_id = t.genre_id
    WHERE i.invoice_date >= '2021-01-01' AND i.invoice_date < '2023-01-01'
    GROUP BY 1, 2
),
shares AS (
    SELECT genre, yr,
           100.0 * revenue / SUM(revenue) OVER (PARTITION BY yr) AS share_pct
    FROM genre_year
)
SELECT genre,
       ROUND(MAX(share_pct) FILTER (WHERE yr = 2021)::numeric, 1) AS share_2021,
       ROUND(MAX(share_pct) FILTER (WHERE yr = 2022)::numeric, 1) AS share_2022,
       ROUND((MAX(share_pct) FILTER (WHERE yr = 2022)
            - MAX(share_pct) FILTER (WHERE yr = 2021))::numeric, 1) AS pp_change
FROM shares
GROUP BY genre
ORDER BY pp_change DESC NULLS LAST;

MAX(x) FILTER (WHERE yr = ...) is the standard trick for pivoting a value you know is unique per (group, year) - MAX of one value is that value. Percentage points, not percent change of a percentage - vocabulary interviewers absolutely notice.

P7.11chinookhard

Customer lifecycle snapshot: one row per customer with first purchase date, last purchase date, total invoices, lifetime value, and a status of 'active' (purchased in the final 6 months of the dataset), 'lapsing' (6-12 months), or 'churned' (older). Anchor "now" to MAX(invoice_date) in the data, not the real today.

solution
WITH anchor AS (
    SELECT MAX(invoice_date) AS data_now FROM invoice
),
per_customer AS (
    SELECT customer_id,
           MIN(invoice_date) AS first_purchase,
           MAX(invoice_date) AS last_purchase,
           COUNT(*)          AS invoices,
           SUM(total)        AS ltv
    FROM invoice
    GROUP BY customer_id
)
SELECT p.customer_id, p.first_purchase::date, p.last_purchase::date,
       p.invoices, ROUND(p.ltv::numeric, 2) AS ltv,
       CASE
         WHEN p.last_purchase >= a.data_now - INTERVAL '6 months'  THEN 'active'
         WHEN p.last_purchase >= a.data_now - INTERVAL '12 months' THEN 'lapsing'
         ELSE 'churned'
       END AS status
FROM per_customer p
CROSS JOIN anchor a
ORDER BY ltv DESC;

Anchoring "now" to the data's own max date makes results reproducible on a static dataset - a detail that reads as real analyst experience, because it is.

P7.12chinookhard

Capstone: build a one-query "genre scorecard" - per genre: lines sold, revenue, revenue share of total, median line price, average lines sold per active month, and rank by revenue. Present it as if handing a table to a director.

solution
WITH per_genre AS (
    SELECT g.name AS genre,
           COUNT(*)                      AS lines,
           SUM(il.unit_price * il.quantity) AS revenue,
           PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY il.unit_price) AS median_price,
           COUNT(DISTINCT date_trunc('month', i.invoice_date)) AS active_months
    FROM invoice_line il
    JOIN invoice i ON i.invoice_id = il.invoice_id
    JOIN track t   ON t.track_id = il.track_id
    JOIN genre g   ON g.genre_id = t.genre_id
    GROUP BY g.name
)
SELECT genre,
       lines,
       ROUND(revenue::numeric, 0) AS revenue,
       ROUND(100.0 * revenue / SUM(revenue) OVER (), 1) AS revenue_share_pct,
       ROUND(median_price::numeric, 2) AS median_price,
       ROUND(lines::numeric / active_months, 1) AS avg_lines_per_month,
       RANK() OVER (ORDER BY revenue DESC) AS revenue_rank
FROM per_genre
ORDER BY revenue_rank;

Everything in one place: aggregation, ordered-set aggregate, window over the grouped result, a derived per-month rate. If you can write this cold and narrate it, you are interview-ready on patterns - the remaining work is speed and performance talk.

P7.13chinookinterview

Pattern: dimensional / semantic layer. This is the payoff of modeling - a business question that becomes a short GROUP BY over a clean join. Report the top 10 (genre, billing_country) pairs by total line revenue for 2023. Line revenue is unit_price * quantity.

hint
Join invoice_line -> invoice (for date and country) and invoice_line -> track -> genre (for the genre label), filter 2023 with a half-open range, then GROUP BY genre and country.
solution
SELECT g.name AS genre,
       i.billing_country,
       ROUND(SUM(il.unit_price * il.quantity)::numeric, 2) AS revenue
FROM invoice_line il
JOIN invoice i ON i.invoice_id = il.invoice_id
JOIN track t   ON t.track_id = il.track_id
JOIN genre g   ON g.genre_id = t.genre_id
WHERE i.invoice_date >= '2023-01-01' AND i.invoice_date < '2024-01-01'
GROUP BY g.name, i.billing_country
ORDER BY revenue DESC
LIMIT 10;

Once the model exists this is trivial - and that is the entire point of modeling. The genre join is INNER because a null-genre track carries no label to group by (a modeling choice to state out loud; LEFT JOIN with COALESCE(g.name, 'Unknown') is the alternative). The half-open >= '2023-01-01' AND < '2024-01-01' keeps the filter sargable, exactly as Module 8 preaches.
