Know Your Schemas

Interviewers hand you unfamiliar tables and watch what you do first. The right first move is always the same: inspect the schema, sample the data, check for NULLs and duplicates. This module makes that reflex automatic and builds your reference map for the other eight modules. You have two databases: chinook (a small, clean digital-music store) and stackoverflow (a real, large Q&A dump from the DBA Stack Exchange).

The exploration toolkit

These work in every PostgreSQL database. Memorize the first three - you will type them hundreds of times.

-- 1. What tables exist? (psql shortcut: \dt or \dt *.* for all schemas)
SELECT table_schema, table_name
FROM information_schema.tables
WHERE table_type = 'BASE TABLE'
  AND table_schema NOT IN ('pg_catalog', 'information_schema')
ORDER BY table_schema, table_name;
-- 2. What columns does a table have? (psql shortcut: \d tablename)
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_name = 'customer'
ORDER BY ordinal_position;
-- 3. What does the data actually look like?
SELECT * FROM customer LIMIT 10;
-- 4. How big is it, and is the key really unique?
SELECT COUNT(*) AS total_rows,
       COUNT(DISTINCT customer_id) AS distinct_keys
FROM customer;
-- If these two numbers differ, customer_id is not unique. Say that out loud in an interview.
-- 5. Where are the NULLs hiding?
SELECT COUNT(*) - COUNT(company) AS customers_without_company
FROM customer;
-- COUNT(column) skips NULLs; COUNT(*) does not. The difference is the NULL count.
Interview habit When handed a new table, narrate: "First I want to confirm the grain of this table - what does one row represent? Then I will check whether the join key is unique and whether the date column has NULLs." That one sentence signals senior before you have written any SQL.

Reference map: the tables you will use most

Every query in Modules 1-8 uses the column names listed here. Both databases live in the public schema, so no schema-qualifying is needed - SELECT * FROM posts, not public.posts.

chinook (public schema, snake_case)

A digital-music store: artists make albums, albums hold tracks, customers buy tracks on invoices, and a small staff supports them. Roughly 3,500 tracks and 400 invoices - small enough to read every result by eye.

TableKey columns
artistartist_id, name
albumalbum_id, title, artist_id
tracktrack_id, name, album_id, genre_id, media_type_id, composer, milliseconds, bytes, unit_price
genregenre_id, name (25 genres)
media_typemedia_type_id, name
playlistplaylist_id, name
playlist_trackplaylist_id, track_id
invoiceinvoice_id, customer_id, invoice_date, billing_country, billing_city, total
invoice_lineinvoice_line_id, invoice_id, track_id, unit_price, quantity
customercustomer_id, first_name, last_name, company, city, country, support_rep_id, email
employeeemployee_id, first_name, last_name, title, reports_to, hire_date

The money math: line revenue = unit_price * quantity from invoice_line (there is no discount column). The hierarchy: employee.reports_to points at another employee_id - three clean levels, from the General Manager down to support staff, which makes it the recursive-CTE example in Module 5. The join spine: album.artist_id → artist, track.album_id → album, track.genre_id → genre, invoice.customer_id → customer, invoice_line.invoice_id → invoice, invoice_line.track_id → track, customer.support_rep_id → employee.

stackoverflow (public schema, lowercase - no underscores)

A real dump from the DBA Stack Exchange: ~248k users, ~243k posts, ~912k votes. The recognizable top-reputation users (Aaron Bertrand, Erwin Brandstetter) are really in here. This is where every performance, window, and interview-pattern drill earns its keep.

Column naming - read this once Every column is lowercase with no underscores: it is owneruserid, not owner_user_id; creationdate, not creation_date; displayname, not display_name; upvotes/downvotes, not up_votes. The user primary key is id (a post's author is posts.owneruserid = users.id).
TableKey columns
usersid, displayname, reputation, creationdate, location, views, upvotes, downvotes
postsid, posttypeid (1=question, 2=answer), owneruserid, parentid (an answer points at its question), acceptedanswerid, creationdate, score, viewcount, title, tags, answercount, commentcount
commentsid, postid, userid, score, text, creationdate
votesid, postid, votetypeid (2=up, 3=down), userid, creationdate, bountyamount
badgesid, userid, name, class (1=gold, 2=silver, 3=bronze), date, tagbased
tagsid, tagname, count

The relationships worth memorizing: an answer's parentid is its question's id, and a question's acceptedanswerid points at one answer - both enable self-joins on posts, a classic interview move. Join posts.owneruserid to users.id for authorship, votes.postid/comments.postid/badges.userid for engagement.

The tags gotcha posts.tags is a single pipe-delimited string like |mysql|innodb|myisam|, not a list. To find posts on one tag, match the delimited form: WHERE tags LIKE '%|postgresql|%'. The busiest tags here are sql-server, mysql, and postgresql.
